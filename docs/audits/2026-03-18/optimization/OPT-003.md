# OPT-003: Async Cache Deletes Block the Request Path

## Impact
MEDIUM

## Category
I/O / Latency Optimization

## Location
- File: packages/core/src/services/APIService.ts
- Line(s): 432-454
- Function: getCachedPrice()

## Current Performance
When a cache entry is stale (expired TTL) or corrupted (version mismatch, checksum failure), the entry is deleted with `await this.cache.delete(cacheKey)` on the request path. The caller must wait for the delete to complete before proceeding to fetch fresh data from the API. For fast backends (in-memory Map), the delete is negligible. For slower backends (KV, Redis), it adds latency to every stale cache hit.

## Bottleneck Analysis
```typescript
// APIService.ts:432-454
// Check cache version
if (cached.version && cached.version !== API_CACHE_VERSION) {
  this.metrics.evictions++;
  await this.cache.delete(cacheKey);  // ← Blocks until delete completes
  return null;
}

// Check if cache has expired
if (Date.now() - cached.timestamp > cached.ttl) {
  this.metrics.evictions++;
  await this.cache.delete(cacheKey);  // ← Blocks until delete completes
  return null;
}

// Validate checksum
if (cached.checksum) {
  const computedChecksum = generateChecksum(cached.data);
  if (computedChecksum !== cached.checksum) {
    this.metrics.evictions++;
    this.logger.warn(`Cache corruption detected for key: ${cacheKey}`);
    await this.cache.delete(cacheKey);  // ← Blocks until delete completes
    return null;
  }
}
```

Three separate `await` points, each blocking the response. The delete is a "best effort" cleanup — the caller doesn't need to wait for it.

## Proposed Optimization
```typescript
// Fire-and-forget the delete — caller gets null immediately
if (cached.version && cached.version !== API_CACHE_VERSION) {
  this.metrics.evictions++;
  void this.cache.delete(cacheKey).catch(() => {});  // Non-blocking
  return null;
}
```

## Expected Improvement
- Eliminates 1-50ms latency per stale cache hit (depending on backend)
- No impact on correctness — the next `getCachedPrice()` call will re-check staleness

## Trade-offs
- If the delete fails silently, the stale entry persists until the next cleanup cycle
- `void` expressions are less debuggable than awaited operations
- The `eslint` rule `no-floating-promises` may flag `void` usage — use `catch(() => {})` to satisfy it

## Benchmark Approach
- Measure p95 latency of `getPriceData()` before/after with KV cache backend
- Count cache evictions via `this.metrics.evictions` to understand frequency
