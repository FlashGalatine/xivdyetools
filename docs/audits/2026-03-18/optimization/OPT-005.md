# OPT-005: Rate Limiter Cleanup Iterates All Entries with Array Recreation

## Impact
LOW

## Category
Algorithm / Memory

## Location
- File: packages/rate-limiter/src/backends/memory.ts
- Line(s): 164-175
- Function: cleanupOldEntries()

## Current Performance
The cleanup function iterates ALL entries in the request log and creates a new filtered array for each one. For a large number of tracked keys (10,000+), this is O(n×m) where n is the number of keys and m is the average timestamps per key.

## Bottleneck Analysis
```typescript
// memory.ts:164-175
private cleanupOldEntries(maxAge: number): void {
  const cutoff = Date.now() - maxAge;

  this.requestLog.forEach((timestamps, key) => {
    const filtered = timestamps.filter((ts) => ts > cutoff);  // New array per key
    if (filtered.length === 0) {
      this.requestLog.delete(key);
    } else {
      this.requestLog.set(key, filtered);  // Replace with new array
    }
  });
}
```

For 10,000 keys with an average of 50 timestamps each:
- 10,000 `filter()` calls, each creating a new array
- 500,000 timestamp comparisons
- Up to 10,000 `Map.set()` calls for replacement

## Impact Assessment
**Low in practice.** The cleanup runs every 100 requests (`DEFAULT_CLEANUP_INTERVAL = 100`). With a 10-entry default max (`DEFAULT_MAX_ENTRIES = 10,000`), the iteration is fast. Workers also restart frequently, resetting the in-memory state.

## Proposed Optimization
```typescript
// In-place modification avoids array allocation
private cleanupOldEntries(maxAge: number): void {
  const cutoff = Date.now() - maxAge;
  const toDelete: string[] = [];

  for (const [key, timestamps] of this.requestLog) {
    // Filter in-place using splice from the front
    let i = 0;
    while (i < timestamps.length && timestamps[i] <= cutoff) i++;
    if (i === timestamps.length) {
      toDelete.push(key);
    } else if (i > 0) {
      timestamps.splice(0, i); // In-place removal of expired entries
    }
  }

  for (const key of toDelete) {
    this.requestLog.delete(key);
  }
}
```

Since timestamps are always pushed in chronological order, we can exploit the sorted property for O(n) cleanup instead of O(n×m).

## Expected Improvement
- Eliminates array allocation during cleanup
- Exploits sorted timestamps for faster scanning
- Reduces GC pressure under high key counts

## Trade-offs
- `splice()` has its own overhead (array shifting), but only from the front
- In-place modification is slightly less readable than `filter()`

## Benchmark Approach
- Measure cleanup duration with 10,000 keys and 100 timestamps each
- Compare GC pause times before/after
