# OPT-002: No Pagination Bounds on Universalis Proxy Response Size

## Impact
MEDIUM

## Category
I/O / Resource Protection

## Location
- File: apps/universalis-proxy/src/index.ts
- Line(s): 150-158
- Function: GET /api/v2/aggregated/:datacenter/:itemIds handler

## Current Performance
The proxy validates item count (max 100 per PROXY-CRITICAL-003) and datacenter name, but does not bound the response size. A request for 100 items across a large datacenter like "Crystal" (containing multiple worlds) can return several megabytes of listing data from Universalis.

## Bottleneck Analysis
```typescript
// universalis-proxy/index.ts:150-158
const ids = itemIds.split(',').map(Number);

// Reject empty or excessive item counts
if (ids.length === 0 || ids.length > 100) {
  return c.json({
    error: 'Item count must be between 1 and 100',
    provided: ids.length,
  }, 400);
}
// No response size limit after this point
```

Universalis returns detailed listing data including individual listings, world prices, and historical data. For 100 items, the response can be 2-5 MB, which:
1. Consumes Worker CPU time for JSON parsing/forwarding
2. Increases egress bandwidth costs
3. Can cause client-side memory issues on mobile browsers

## Proposed Optimization
```typescript
// Option 1: Limit response fields via query parameter
const fields = 'currentAveragePrice,minPrice,nqSaleVelocity,hqSaleVelocity';
const upstreamUrl = `${base}/${datacenter}/${normalizedIds}?fields=${fields}`;

// Option 2: Stream response with size limit
const response = await fetch(upstreamUrl);
const contentLength = parseInt(response.headers.get('content-length') || '0');
if (contentLength > MAX_RESPONSE_SIZE) {
  return c.json({ error: 'Response too large' }, 413);
}
```

## Expected Improvement
- Option 1: Response size reduced by ~80% (only summary data, no individual listings)
- Option 2: Protects against unexpectedly large responses

## Trade-offs
- Option 1 changes the API surface — clients relying on listing data would break
- Option 2 requires content-length header from Universalis (may not always be present)

## Benchmark Approach
- Measure average response sizes per item count via Worker analytics
- Compare CPU time per request before/after field filtering
