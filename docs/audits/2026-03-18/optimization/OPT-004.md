# OPT-004: Memory Leak Risk from Event Handler Accumulation in Image Upload

## Impact
MEDIUM

## Category
Memory / Resource Management

## Location
- File: apps/web-app/src/components/image-upload-display.ts
- Line(s): 323-356
- Function: Image/FileReader handling

## Current Performance
The image upload component creates new `FileReader` and `Image` objects on each upload attempt. Comments reference BUG-016 fixes that clear `onload`/`onerror` handlers after use. However, if a user rapidly uploads multiple images without waiting for each to complete, multiple FileReader/Image objects accumulate in memory with active event handlers.

## Bottleneck Analysis
```typescript
// image-upload-display.ts:323-356 (approximate)
// BUG-016: Clear handler to release memory
// BUG-016: Clear handlers after successful load
// BUG-016: Clear handlers on error
```

Each upload cycle creates:
1. A `FileReader` with `onload` and `onerror` handlers
2. An `Image` with `onload` and `onerror` handlers
3. A `Canvas` context for pixel extraction

If the previous upload is still processing when a new one starts, the old objects aren't cleaned up until their handlers fire or GC runs.

## Proposed Optimization
```typescript
// Track and abort previous upload
private currentReader: FileReader | null = null;

handleImageUpload(file: File) {
  // Abort previous if still reading
  if (this.currentReader && this.currentReader.readyState === FileReader.LOADING) {
    this.currentReader.abort();
  }

  this.currentReader = new FileReader();
  this.currentReader.onload = (e) => {
    // ... process image
    this.currentReader = null; // Release reference
  };
  this.currentReader.onerror = () => {
    this.currentReader = null;
  };
  this.currentReader.readAsDataURL(file);
}
```

Also consider cleanup in the Lit component's `disconnectedCallback()`:
```typescript
disconnectedCallback() {
  super.disconnectedCallback();
  if (this.currentReader?.readyState === FileReader.LOADING) {
    this.currentReader.abort();
  }
  this.currentReader = null;
}
```

## Expected Improvement
- Prevents accumulation of orphaned FileReader/Image objects
- Faster GC under rapid uploads
- Eliminates potential for race conditions between old/new upload results

## Trade-offs
- Aborting a FileReader mid-read discards the partially-read data (expected behavior)
- Slightly more complex component lifecycle management

## Benchmark Approach
- Use Chrome DevTools Memory profiler during rapid image uploads
- Compare heap snapshots before/after optimization
