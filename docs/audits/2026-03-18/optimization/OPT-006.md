# OPT-006: Missing AbortController Cleanup in Universalis Client

## Impact
LOW

## Category
Resource Management / Memory

## Location
- File: apps/discord-worker/src/services/budget/universalis-client.ts
- Line(s): 115-132
- Function: Fetch with timeout pattern

## Current Performance
The universalis client uses an `AbortController` with `setTimeout` for request timeouts. When the request completes (success or failure), the timeout is cleared with `clearTimeout()`. However, the `AbortController` itself is not explicitly aborted, leaving its signal listeners in memory until garbage collection.

## Bottleneck Analysis
```typescript
// universalis-client.ts (approximate lines 115-132)
const controller = new AbortController();
const timeoutId = setTimeout(() => controller.abort(), TIMEOUT_MS);

try {
  const response = await fetch(url, { signal: controller.signal });
  // ... process response
  return data;
} finally {
  clearTimeout(timeoutId);
  // Missing: controller.abort() to clean up signal listeners
}
```

When a request succeeds before the timeout:
1. `clearTimeout` prevents the abort from firing ✓
2. The `AbortController`'s signal remains in an "unaborted" state
3. Any listeners attached to `controller.signal` persist until GC

## Impact Assessment
**Minimal.** The `AbortController` and its signal are local variables that become eligible for GC after the function returns. Modern V8 handles this efficiently. The concern is theoretical for long-lived isolates processing many sequential requests.

## Proposed Optimization
```typescript
try {
  const response = await fetch(url, { signal: controller.signal });
  return data;
} finally {
  clearTimeout(timeoutId);
  controller.abort('Cleanup');  // Release signal listeners immediately
}
```

Note: Calling `abort()` after a successful fetch is a no-op for the fetch itself but signals any remaining listeners to clean up.

## Expected Improvement
- Marginal memory improvement in long-lived Worker isolates
- Follows best practice for AbortController lifecycle management

## Trade-offs
- Calling `abort()` after success may cause confusing log messages if the abort reason is logged
- Requires passing a reason string to distinguish cleanup from actual timeouts

## Benchmark Approach
- Compare memory usage with Chrome DevTools over 1000 sequential requests
- Measure with and without explicit abort() cleanup
