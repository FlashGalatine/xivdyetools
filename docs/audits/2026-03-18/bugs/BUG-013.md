# BUG-013: OAuth State Signature Verification Accepts Unsigned States

## Severity
MEDIUM

## Type
Security / CSRF Protection

## Location
- File: apps/oauth/src/utils/state-signing.ts
- Line(s): 53-96
- Function: verifyState()

## Description
The `verifyState()` function has a backward-compatibility code path that accepts unsigned state parameters when `allowUnsigned = true`. This was intended as a transitional mechanism during the migration to signed states. If this flag is still enabled in any caller past the transition period, an attacker could forge OAuth state parameters without a valid HMAC signature, potentially enabling CSRF attacks.

## Evidence
```typescript
// state-signing.ts (approx lines 53-96)
export function verifyState(
  signedState: string,
  secret: string,
  allowUnsigned: boolean = false  // Default is safe (false)
): StateData | null {
  // ...
  if (allowUnsigned && parts.length === 1) {
    const json = atob(signedState);
    const state = JSON.parse(json) as StateData;
    console.warn('Accepted unsigned state...');
    return state;
  }
  // ... HMAC verification for signed states
}
```

## Mitigating Factors
- `allowUnsigned` defaults to `false` — the secure path is the default
- The `console.warn()` provides visibility when unsigned states are accepted
- OAuth state is a session-local value; exploitation requires the attacker to intercept the OAuth flow

## Recommended Action
1. Verify no callers pass `allowUnsigned: true` in production code
2. If the transition period is over, remove the `allowUnsigned` parameter entirely
3. If it must remain, add a deprecation warning and set a removal date

## Suggested Fix
```typescript
// Remove the unsigned path entirely
export function verifyState(
  signedState: string,
  secret: string
): StateData | null {
  const parts = signedState.split('.');
  if (parts.length !== 2) return null;
  // ... HMAC verification only
}
```

## Why It's Hidden
- The default behavior is secure (`allowUnsigned = false`)
- The flag is only relevant during deployment transitions
- The concern depends on how callers invoke the function, not the function itself
