# BUG-003: Non-Null Assertion on Map.get() After Conditional Set in DyeDatabase

## Severity
LOW

## Type
Code Quality / Type Safety

## Location
- File: packages/core/src/services/dye/DyeDatabase.ts
- Line(s): 340-344
- Function: initialize()

## Description
During dye database initialization, hue bucket indexing uses `has()` + `set()` followed by `get()!` with a non-null assertion. While safe in single-threaded JavaScript (the `set` and `get` are synchronous), the `!` operator bypasses TypeScript's null-safety guarantees and is fragile if the code is ever refactored.

## Evidence
```typescript
// DyeDatabase.ts:340-344
const hueBucket = this.getHueBucket(dye.hsv.h);
if (!this.dyesByHueBucket.has(hueBucket)) {
  this.dyesByHueBucket.set(hueBucket, []);
}
this.dyesByHueBucket.get(hueBucket)!.push(dye);  // Non-null assertion
```

## Reproduction Scenario
Not reproducible in current code — the pattern is safe in synchronous, single-threaded execution. The concern is maintenance fragility: if someone adds caching, async initialization, or a Map subclass, the `!` assertion masks a potential null dereference.

## Suggested Fix
```typescript
// Option 1: Use local variable
const bucket = this.dyesByHueBucket.get(hueBucket) ?? [];
bucket.push(dye);
this.dyesByHueBucket.set(hueBucket, bucket);

// Option 2: Use Map's guaranteed get-after-set (idiomatic)
if (!this.dyesByHueBucket.has(hueBucket)) {
  this.dyesByHueBucket.set(hueBucket, []);
}
const bucket = this.dyesByHueBucket.get(hueBucket);
if (bucket) bucket.push(dye);
```

## Why It's Hidden
TypeScript's `!` operator silences the compiler warning but doesn't add runtime safety. The pattern is correct today but could become a latent bug during refactoring.
