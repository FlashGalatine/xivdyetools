# BUG-010: JWT Verification Missing Required `sub` Claim Validation

## Severity
MEDIUM

## Type
Security / Authentication

## Location
- File: packages/auth/src/jwt.ts
- Line(s): 153-171
- Function: verifyJWT()

## Description
The `verifyJWT()` function validates the `exp` (expiration) claim but does not validate the `sub` (subject) claim. A JWT with a valid signature and non-expired timestamp but no `sub` claim passes verification. In downstream consumers like presets-api, `auth.userDiscordId` is derived from the JWT's `sub` claim — if missing, it could be `undefined`, potentially bypassing user-specific logic.

## Evidence
```typescript
// jwt.ts:153-171
export async function verifyJWT(
  token: string,
  secret: string
): Promise<JWTPayload | null> {
  try {
    const payload = await verifyJWTSignature(token, secret);
    if (!payload) return null;

    // FINDING-003: Require exp claim
    const now = Math.floor(Date.now() / 1000);
    if (!payload.exp || payload.exp < now) {
      return null;
    }

    return payload;  // No check for payload.sub
  } catch {
    return null;
  }
}
```

## Why It's Currently Safe
- Tokens are only created by the OAuth worker, which always includes `sub`
- The `JWTPayload` interface likely defines `sub` as required
- A crafted token without `sub` would need a valid HMAC signature (requires the secret)

## Risk Scenario
If the JWT secret is compromised (or in a future scenario where tokens come from multiple issuers), an attacker could craft a token with valid signature and expiration but no `sub`, potentially bypassing user-specific authorization checks.

## Suggested Fix
```typescript
// After expiration check:
if (!payload.sub) {
  return null;  // Reject tokens without a subject
}
```

## Why It's Hidden
- All tokens in the ecosystem are created by a single trusted issuer (OAuth worker)
- TypeScript types may enforce `sub` at compile time
- The vulnerability requires secret compromise to exploit
