# BUG-009: Missing HSV Validation for External API Callers

## Severity
LOW

## Type
Input Validation / Boundary Check

## Location
- File: packages/core/src/services/dye/DyeDatabase.ts
- Line(s): 164-197 (validation during init), public methods accepting HSV
- Function: getHueBucket(), getHarmonyDyes()

## Description
The `DyeDatabase.initialize()` method thoroughly validates HSV values (h: 0-360, s: 0-100, v: 0-100) for loaded dye data at lines 164-197. However, public methods that accept HSV values from external callers (e.g., `getHueBucket()`, harmony operations) don't re-validate. If a caller provides out-of-range HSV values (e.g., h=400 from a broken color picker), the hue bucket calculation produces invalid indices silently.

## Evidence
```typescript
// DyeDatabase.ts:164-197 — Validation EXISTS for loaded data
if (
  typeof hsv.h !== 'number' ||
  hsv.h < 0 ||
  hsv.h > 360 ||
  typeof hsv.s !== 'number' ||
  hsv.s < 0 ||
  hsv.s > 100 ||
  typeof hsv.v !== 'number' ||
  hsv.v < 0 ||
  hsv.v > 100
) {
  this.logger.warn(`Dye ${idForLog} has invalid HSV values`);
  return false;
}

// But getHueBucket() (line 340) accepts raw h values without clamping
const hueBucket = this.getHueBucket(dye.hsv.h);
```

## Impact
Low — the `getHueBucket()` method likely normalizes via modulo (hue wraps at 360°), and all internal dye data has been validated. Only external callers passing raw HSV could trigger unexpected behavior, and those callers are TypeScript-typed.

## Suggested Fix
```typescript
private getHueBucket(hue: number): number {
  // Normalize hue to 0-360 range (handles overflow from color pickers)
  const normalizedHue = ((hue % 360) + 360) % 360;
  return Math.floor(normalizedHue / 10);
}
```

## Why It's Hidden
- All internal dye data is validated during `initialize()`
- TypeScript types prevent most invalid inputs
- Only affects public API usage with out-of-range HSV from unvalidated sources
