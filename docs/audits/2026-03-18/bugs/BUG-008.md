# BUG-008: LocalizationService Singleton Race Condition Under Concurrent Requests

## Severity
MEDIUM (confirmed, still open)

## Type
Race Condition / Concurrency

## Location
- File: packages/core/src/services/LocalizationService.ts
- Related: apps/discord-worker/src/services/i18n.ts

## Cross-Reference
Same as: `2026-02-06/bugs/BUG-001.md` — LocalizationService Singleton Race Condition

## Description
`LocalizationService` is a singleton with mutable state (`currentLocale`). In Cloudflare Workers, multiple concurrent requests share the same isolate. When requests from users with different language preferences run concurrently, one request's `setLocale()` call can overwrite another request's locale mid-processing.

This was originally identified in the 2026-02-06 discord-worker audit (BUG-001). The issue remains open because the singleton pattern in the core library hasn't been changed.

## Evidence
From the 2026-02-06 audit, the race manifests through `initializeLocale()` in `i18n.ts`:
```
1. Request A (Japanese user) calls initializeLocale('ja')
2. Request A awaits SVG rendering (yields execution)
3. Request B (English user) calls initializeLocale('en')
4. Request A resumes and calls getLocalizedDyeName() — gets English names
```

Multiple command handlers call `initializeLocale()`:
- `harmony.ts` → `await initializeLocale(locale)`
- `budget.ts` → `await initializeLocale(t.getLocale(), logger)`
- `extractor.ts` → `await initializeLocale(locale)`
- `favorites.ts` → `await initializeLocale(t.getLocale())`
- `collection.ts` → `await initializeLocale(locale)`

## Impact
Non-English users may occasionally see dye names in the wrong language. The race window is narrow (between `setLocale` and subsequent reads), and English is the fallback, so English users are unaffected.

## Suggested Fix (from 2026-02-06 audit)
Pass locale as a parameter to `getLocalizedDyeName()` instead of relying on singleton state, or use request-scoped localization instances.

## Why It's Still Open
The fix requires changing the `@xivdyetools/core` library API (either making `LocalizationService` instantiable or adding locale parameters to every method). This is a breaking change affecting all consumers.

## Why It's Hidden
- Only manifests under concurrent load with users of different languages
- English-only users never notice
- The race window is very short
- In development/testing, typically one request at a time
