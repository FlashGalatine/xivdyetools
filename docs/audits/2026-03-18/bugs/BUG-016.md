# BUG-016: Rate Limiter Has No Fail-Closed Behavior

## Severity
MEDIUM

## Type
Security / Availability

## Location
- File: apps/presets-api/src/services/rate-limit-service.ts
- Related: apps/universalis-proxy/src/index.ts

## Description
The rate limiting services use `MemoryRateLimiter` which should not fail under normal conditions. However, if `check()` throws an unexpected exception (memory pressure, prototype pollution, internal error), the calling middleware crashes the request with a 500 error rather than failing closed (denying the request). This means a rate limiter failure manifests as a service outage rather than a rate limit.

In contrast, a "fail-closed" approach would deny the request when the rate limiter is unavailable, preventing abuse during outages.

## Evidence
Rate limiting is typically called from middleware without defensive error handling:
```typescript
// Typical pattern in handlers:
const rateLimitResult = await checkRateLimit(clientIP, rateLimitConfig);
if (!rateLimitResult.allowed) {
  return c.json({ error: 'Rate limit exceeded' }, 429);
}
// No catch block — if checkRateLimit throws, the request 500s
```

## Impact Assessment
**Lower than initially assessed.** The `MemoryRateLimiter` is a simple in-memory implementation with no external dependencies (no network calls, no disk I/O). The only realistic failure scenario is memory exhaustion on the Worker isolate, which would affect all processing regardless.

## Suggested Fix
```typescript
// Wrap rate limit check with fail-closed behavior
async function safeCheckRateLimit(key: string, config: RateLimitConfig): Promise<RateLimitResult> {
  try {
    return await checkRateLimit(key, config);
  } catch (error) {
    console.error('[RATE-LIMIT] Rate limiter error, failing closed', error);
    return {
      allowed: false,
      remaining: 0,
      resetAt: new Date(Date.now() + 60000),
      limit: config.maxRequests,
      retryAfter: 60,
    };
  }
}
```

## Why It's Hidden
- `MemoryRateLimiter` is extremely unlikely to throw
- The failure mode (500 error) is noticeable but misattributed to "server error" rather than rate limiting
- No tests cover the "rate limiter throws" scenario
