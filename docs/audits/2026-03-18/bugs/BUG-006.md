# BUG-006: LRU Cache Delete+Set Not Atomic in Async Contexts

## Severity
MEDIUM

## Type
Race Condition / Concurrency

## Location
- File: packages/core/src/utils/index.ts
- Line(s): 73-83
- Function: LRUCache.get()

## Cross-Reference
Expands on: `2026-01-25/optimization/OPT-001-lru-cache-concurrency.md`

## Description
The LRU cache's `get()` method reorders entries by deleting and re-inserting the key. In single-threaded synchronous code, this is atomic. However, in Cloudflare Workers where multiple requests share an isolate and yield at `await` points, interleaved cache operations can corrupt LRU ordering, leading to premature eviction of hot items.

## Evidence
```typescript
// utils/index.ts:73-83
get(key: K): V | undefined {
  if (!this.cache.has(key)) return undefined;
  const value = this.cache.get(key)!;
  // Move to end (most recently used)
  // Note: Atomic in synchronous contexts, but see concurrency warning above
  this.cache.delete(key);
  this.cache.set(key, value);
  return value;
}
```

The comment on line 79 acknowledges this concern: *"Atomic in synchronous contexts, but see concurrency warning above."*

## Reproduction Scenario
1. Request A calls `cache.get('dye-42')` — key is deleted from Map
2. Request A yields at an `await` (e.g., awaiting API response)
3. Request B calls `cache.set('new-key', value)` — triggers eviction, may evict other items
4. Request A resumes, `cache.set('dye-42', value)` re-inserts — but LRU ordering is now inconsistent

## Impact
- Cache eviction order becomes unpredictable under concurrent load
- Hot items may be evicted prematurely, increasing cache miss rate
- No data corruption — just suboptimal cache behavior

## Suggested Fix
The current implementation is acceptable for the use case (price data cache where occasional extra misses are harmless). For a stricter fix:
```typescript
get(key: K): V | undefined {
  // Don't reorder on read — accept FIFO-like behavior under concurrency
  return this.cache.get(key);
}
```

## Why It's Hidden
- The delete+set is synchronous within a single microtask — no yields between them
- The race requires two requests hitting the same cache key at the exact same async yield point
- Cache misses just trigger a re-fetch, not an error
