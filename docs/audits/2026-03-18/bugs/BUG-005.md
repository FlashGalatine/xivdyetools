# BUG-005: HMAC CryptoKey Cache Uses FIFO Eviction Instead of LRU

## Severity
MEDIUM

## Type
Caching / Performance Degradation

## Location
- File: packages/auth/src/hmac.ts
- Line(s): 62-67
- Function: getOrCreateHmacKey()

## Description
The `cryptoKeyCache` uses simple FIFO eviction: when the cache is full (10 entries), it deletes the *oldest inserted* entry regardless of access frequency. During key rotation scenarios where two secrets alternate rapidly, the most-used key may be repeatedly evicted and re-imported via expensive `crypto.subtle.importKey()` calls.

## Evidence
```typescript
// hmac.ts:62-67
// Evict oldest entries if cache is full (simple FIFO)
if (cryptoKeyCache.size >= CRYPTO_KEY_CACHE_MAX) {
  const firstKey = cryptoKeyCache.keys().next().value;
  if (firstKey !== undefined) {
    cryptoKeyCache.delete(firstKey);
  }
}

cryptoKeyCache.set(cacheKey, key);
```

The cache key format is `${secret}:${usage}`, so one secret generates up to 3 entries (`sign`, `verify`, `both`). During key rotation with 2 secrets, that's 6 entries — well within the 10-entry limit, so eviction wouldn't trigger.

## Actual Impact
**Lower than initially assessed.** With `CRYPTO_KEY_CACHE_MAX = 10` and typical usage patterns (1-2 secrets with 2-3 usages each = 3-6 entries), the cache rarely fills. The FIFO concern only applies if more than ~3 different secrets are in simultaneous use, which is unlikely outside an attack scenario.

## Suggested Fix
```typescript
// Use LRU pattern: move accessed entries to end
export async function getOrCreateHmacKey(...): Promise<CryptoKey> {
  const cacheKey = `${secret}:${usage}`;
  const cached = cryptoKeyCache.get(cacheKey);
  if (cached) {
    // LRU: Move to end by re-inserting
    cryptoKeyCache.delete(cacheKey);
    cryptoKeyCache.set(cacheKey, cached);
    return cached;
  }
  // ... existing import logic
}
```

## Why It's Hidden
- Default cache max (10) is generous for typical 1-2 secret usage
- `crypto.subtle.importKey()` is fast (~0.1ms) so cache misses aren't catastrophic
- The "simple FIFO" comment signals intentional simplicity over optimality
- Only becomes problematic under unusual multi-secret scenarios
