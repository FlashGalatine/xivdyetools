# BUG-011: DyeSearch.searchByName() Null Safety from Untyped Callers

## Severity
LOW

## Type
Input Validation / Null Safety

## Location
- File: packages/core/src/services/dye/DyeSearch.ts
- Line(s): 89-100
- Function: searchByName()

## Description
`searchByName(query: string)` calls `query.toLowerCase().trim()` without null/undefined checking. TypeScript callers are protected by the type signature, but untyped JavaScript callers, `any`-typed inputs, or values from JSON parsing (`JSON.parse` returns `any`) could pass `null` or `undefined`, causing a runtime `TypeError: Cannot read properties of null (reading 'toLowerCase')`.

## Evidence
```typescript
// DyeSearch.ts:89-100
searchByName(query: string): Dye[] {
  this.database.ensureLoaded();
  const lowerQuery = query.toLowerCase().trim();  // Crashes if query is null/undefined

  if (lowerQuery.length === 0) {
    return [];
  }

  const dyes = this.database.getDyesInternal();
  return dyes.filter((dye) => dye.nameLower.includes(lowerQuery));
}
```

## Impact
Low — all known callers are TypeScript and pass validated strings. The function is part of `@xivdyetools/core`'s public API, so third-party consumers *could* call it with invalid input.

## Suggested Fix
```typescript
searchByName(query: string): Dye[] {
  this.database.ensureLoaded();
  if (!query || typeof query !== 'string') return [];
  const lowerQuery = query.toLowerCase().trim();
  // ...
}
```

## Why It's Hidden
- TypeScript types prevent this in all current callers
- Only exploitable through untyped JavaScript or `any`-typed code
- Would manifest as an uncaught TypeError, not a silent failure
