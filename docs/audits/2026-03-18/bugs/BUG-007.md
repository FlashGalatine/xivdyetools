# BUG-007: Missing Locale Fallback Chain for Incomplete Translations

## Severity
MEDIUM

## Type
Localization / Silent Failure

## Location
- File: packages/core/src/services/LocalizationService.ts
- Function: getDyeName(), getLabel()
- Related: packages/core/src/services/localization/TranslationProvider.ts

## Description
When the current locale is set to a non-English language (e.g., `ja`, `ko`) and a translation key doesn't exist in that locale's data, the system returns either the raw key or `undefined` instead of falling back to English. This means if locale data is incomplete or a new dye is added without immediate translation, users see broken text or raw identifiers.

## Evidence
The `LocalizationService` facade delegates to `TranslationProvider` and `LocaleRegistry`. The locale data is loaded lazily per language. If a dye name exists in English but not yet in Korean (common for new dyes), the Korean user sees the raw fallback instead of the English name.

From the locale pipeline: XIVAPI serves en/ja/de/fr natively, but Korean and Chinese names are manually sourced from market board HTML (per `CLAUDE.md`). Manual sourcing means these locales are more likely to have gaps.

## Reproduction Scenario
1. New dye is added to `colors_xiv.json` via a game patch
2. English name is extracted from XIVAPI automatically
3. Korean name hasn't been sourced yet (manual process)
4. Korean user runs `/harmony` command
5. Dye name shows as empty string or English fallback key instead of graceful English fallback

## Suggested Fix
```typescript
// In TranslationProvider or getDyeName():
getDyeName(id: number, locale: LocaleCode): string {
  const localeData = this.registry.getLocaleData(locale);
  const name = localeData?.dyeNames?.[id];
  if (name) return name;

  // Fallback to English if current locale is incomplete
  if (locale !== 'en') {
    const enData = this.registry.getLocaleData('en');
    const enName = enData?.dyeNames?.[id];
    if (enName) return enName;
  }

  return `Dye #${id}`; // Final fallback
}
```

## Why It's Hidden
- English users (majority) never encounter missing translations
- ja/de/fr have complete translations from XIVAPI
- Only ko/zh are vulnerable, and only for newly added dyes
- The issue is intermittent and depends on locale data completeness
