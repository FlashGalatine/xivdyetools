# BUG-001: APIService Deferred Promise — resolvePromise! Non-Null Assertion Fragility

## Severity
LOW (Previously CRITICAL — original race condition was fixed via CORE-BUG-001)

## Type
Code Quality / Fragile Pattern

## Location
- File: packages/core/src/services/APIService.ts
- Line(s): 538-579
- Function: getPriceData()

## Description
The `getPriceData()` method uses a deferred promise pattern where `resolvePromise` is assigned inside the Promise constructor callback, then called later via `resolvePromise!(value)` with a TypeScript non-null assertion. While the original race condition (CORE-BUG-001) was correctly fixed in a prior audit, the `resolvePromise!` pattern remains fragile — any future refactoring that introduces an `await` between the Promise constructor and the first `resolvePromise!` call could silently break the guarantee.

## Evidence
```typescript
// APIService.ts:538-579
let resolvePromise: (value: PriceData | null) => void;

const promise = new Promise<PriceData | null>((resolve) => {
  resolvePromise = resolve;
  // Note: We don't use reject because getPriceData returns null on error
});

this.pendingRequests.set(cacheKey, promise);

try {
  const cached = await this.getCachedPrice(cacheKey);
  if (cached) {
    this.pendingRequests.delete(cacheKey);
    resolvePromise!(cached);   // Non-null assertion
    return cached;
  }
  // ...
  resolvePromise!(data);       // Non-null assertion
  return data;
} catch (error) {
  this.pendingRequests.delete(cacheKey);
  resolvePromise!(null);       // Non-null assertion
  return null;
}
```

## Status
**Previously fixed** — The CORE-BUG-001/002 fix correctly addresses the original race condition. The deferred promise is stored synchronously before any `await`, and cleanup happens in both success and error paths.

## Residual Concern
The `resolvePromise!` assertion is safe because the Promise constructor executor runs synchronously, guaranteeing assignment before use. However:
1. The `!` assertion bypasses TypeScript's definite assignment checking
2. A `Promise.withResolvers()` (ES2024) or a typed helper would make the pattern self-documenting
3. If someone refactors the code to use `async` in the constructor or moves the assignment, the guarantee breaks silently

## Suggested Fix
```typescript
// Use Promise.withResolvers() (available in CF Workers runtime)
const { promise, resolve } = Promise.withResolvers<PriceData | null>();
this.pendingRequests.set(cacheKey, promise);

try {
  // ... resolve(cached) / resolve(data) instead of resolvePromise!(...)
} catch {
  resolve(null);
}
```

## Why It's Hidden
The original race condition was correctly identified and fixed. The remaining fragility is a code quality concern that would only manifest if someone refactors the promise construction pattern.
