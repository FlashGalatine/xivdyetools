# BUG-014: Duplicate Preset Race Condition — Mitigated but Fragile String Matching

## Severity
LOW (mitigated)

## Type
Race Condition / Error Handling

## Location
- File: apps/presets-api/src/handlers/presets.ts
- Line(s): 443-474
- Function: POST /presets handler

## Description
The preset creation handler checks for duplicates before inserting, but two concurrent requests can both pass the duplicate check. The mitigation catches `UNIQUE constraint failed` errors and falls back to voting on the existing preset. This is correct, but the error detection relies on string matching against the D1 error message, which could break if Cloudflare changes the error format.

## Evidence
```typescript
// presets.ts:443-474
let preset;
try {
  preset = await createPreset(c.env.DB, body, auth.userDiscordId!, auth.userName || 'Unknown User', status);
} catch (error) {
  const errorMessage = error instanceof Error ? error.message : String(error);
  // Fragile: depends on exact error message format from D1
  if (errorMessage.includes('UNIQUE constraint failed') && errorMessage.includes('dye_signature')) {
    const existingPreset = await findDuplicatePreset(c.env.DB, body.dyes);
    if (existingPreset) {
      const voteResult = await addVote(c.env.DB, existingPreset.id, auth.userDiscordId!);
      return c.json({
        success: true,
        duplicate: existingPreset,
        vote_added: voteResult.success && !voteResult.already_voted,
      });
    }
  }
  throw error;
}
```

## Mitigation Assessment
The current approach is functional and handles the race condition correctly. The fragility is limited to:
1. D1 error message format changes (unlikely but possible in major D1 updates)
2. `findDuplicatePreset()` returning null after constraint violation (would re-throw the error)

## Suggested Improvement
```sql
-- Use INSERT ... ON CONFLICT instead of catch-based detection
INSERT INTO presets (name, description, dyes, tags, dye_signature, ...)
VALUES (?, ?, ?, ?, ?, ...)
ON CONFLICT (dye_signature) DO UPDATE SET vote_count = vote_count
RETURNING *;
```

## Why It's Hidden
- The mitigation works correctly in practice
- UNIQUE constraint errors from D1/SQLite have been stable
- The race window is narrow (milliseconds between duplicate check and insert)
