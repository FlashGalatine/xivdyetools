# BUG-018: IP Header Spoofing Could Bypass Rate Limiting

## Severity
LOW

## Type
Security / Rate Limit Bypass

## Location
- File: apps/universalis-proxy/src/index.ts
- Line(s): 117
- Function: GET /api/v2/aggregated/:datacenter/:itemIds handler

## Description
The rate limiter uses the client IP address for per-IP limiting. The IP is extracted from request headers with a fallback chain: `CF-Connecting-IP` → `X-Forwarded-For` → `'unknown'`. In production behind Cloudflare, `CF-Connecting-IP` is set by Cloudflare and cannot be spoofed. However, in development or non-Cloudflare deployments, the `X-Forwarded-For` header is client-controlled and can be spoofed.

## Evidence
```typescript
// universalis-proxy/index.ts:117
const clientIP = c.req.header('CF-Connecting-IP')
  || c.req.header('X-Forwarded-For')?.split(',')[0]?.trim()
  || 'unknown';
```

## Impact Assessment
**Low in production.** Cloudflare Workers always receive `CF-Connecting-IP` from Cloudflare's edge, which is the true client IP and cannot be forged. The `X-Forwarded-For` fallback only activates in local development (`wrangler dev`) or if the Worker is somehow accessed without Cloudflare's proxy.

The `'unknown'` fallback is also concerning: if both headers are missing, all requests share the same rate limit key, creating a shared rate limit that could be exhausted by any single user, affecting all users.

## Suggested Fix
```typescript
// Only trust CF-Connecting-IP in production; reject if missing
const clientIP = c.req.header('CF-Connecting-IP');
if (!clientIP && c.env.ENVIRONMENT === 'production') {
  return c.json({ error: 'Unable to identify client' }, 400);
}
const rateLimitKey = clientIP || c.req.header('X-Forwarded-For')?.split(',')[0]?.trim() || 'dev-unknown';
```

## Why It's Hidden
- Production deployment is always behind Cloudflare
- `CF-Connecting-IP` is always present in production Workers
- The `X-Forwarded-For` path is only used in local development
- Rate limit bypass in dev has no security impact
