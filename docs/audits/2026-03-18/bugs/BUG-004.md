# BUG-004: Base64URL Encoding Uses btoa/atob String Building Pattern

## Severity
LOW

## Type
Encoding / Edge Case Fragility

## Location
- File: packages/crypto/src/base64.ts
- Line(s): 39-44 (encode), 58-78 (decode)
- Functions: base64UrlEncodeBytes(), base64UrlDecodeBytes()

## Description
The encoding function builds a binary string character-by-character from a `Uint8Array`, then passes it to `btoa()`. While safe for byte arrays (values 0-255), the `btoa()` function throws a `DOMException` if any character exceeds the Latin1 range (0-255). The current usage is correct, but the proximity of `base64UrlEncode(str)` (which accepts arbitrary strings) to `base64UrlEncodeBytes(bytes)` could lead to confusion.

## Evidence
```typescript
// base64.ts:39-44
export function base64UrlEncodeBytes(bytes: Uint8Array): string {
  let binary = '';
  for (let i = 0; i < bytes.length; i++) {
    binary += String.fromCharCode(bytes[i]);
  }
  return btoa(binary).replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}

// base64.ts:72-75 (decode)
const bytes = new Uint8Array(binary.length);
for (let i = 0; i < binary.length; i++) {
  bytes[i] = binary.charCodeAt(i);
}
```

## Why It's Currently Safe
- `Uint8Array` values are always 0-255
- `btoa()` accepts Latin1 (0-255) characters
- The string-based `base64UrlEncode()` on line 22 correctly uses `TextEncoder` first, ensuring UTF-8 → bytes → base64

## Residual Concern
- String concatenation in a loop (`binary += String.fromCharCode(...)`) is O(n²) for large inputs due to string immutability
- For large byte arrays (>100KB), this could be noticeably slower than a typed array approach
- The `atob()` function in some runtimes may have size limits

## Suggested Fix
```typescript
// Modern approach — avoids string building entirely
export function base64UrlEncodeBytes(bytes: Uint8Array): string {
  // Available in Cloudflare Workers, Node.js 18+, modern browsers
  const base64 = btoa(String.fromCharCode(...bytes));
  return base64.replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}
```

Note: The spread approach has a stack size limit (~100K args). For large arrays, the current loop is actually more robust. Consider keeping the current implementation with a comment explaining why.

## Why It's Hidden
The function signature (`Uint8Array`) prevents misuse from TypeScript callers. Only untyped JavaScript callers could pass invalid data.
