# BUG-002: APIService Error Path Returns Null Indistinguishable from No Data

## Severity
LOW

## Type
Error Handling / API Design

## Location
- File: packages/core/src/services/APIService.ts
- Line(s): 572-581
- Function: getPriceData()

## Description
When `getPriceData()` encounters an error (network failure, timeout, rate limit), it resolves with `null` — the same value returned when a dye genuinely has no market data. Callers cannot distinguish between "the API call failed" and "this item has no listings." This prevents intelligent retry logic and makes debugging harder.

The comment on line 542 explicitly acknowledges this design: *"We don't use reject because getPriceData returns null on error to maintain existing API behavior (non-throwing)."*

## Evidence
```typescript
// APIService.ts:572-581
} catch (error) {
  // CORE-BUG-002 FIX: Clean up on error path as well
  this.pendingRequests.delete(cacheKey);
  this.logger.error('Failed to fetch price data', error);

  // Resolve with null instead of rejecting to maintain existing behavior
  // where errors return null rather than throwing
  resolvePromise!(null);
  return null;
}
```

All callers waiting on the shared promise (via `pendingRequests`) also receive `null` with no way to differentiate failure from empty data.

## Reproduction Scenario
1. User runs `/budget` command requesting price for a valid dye
2. Universalis API is temporarily down (503)
3. `getPriceData()` catches the error and returns `null`
4. Budget command treats the dye as having no market data (price = 0)
5. User sees incorrect "0 gil" pricing instead of an error message

## Suggested Fix
```typescript
// Option 1: Return a result type
type PriceResult = { data: PriceData } | { error: string } | null;

// Option 2: Add an optional error callback
async getPriceData(
  itemID: number,
  worldID?: number,
  dataCenterID?: string,
  onError?: (error: Error) => void
): Promise<PriceData | null>
```

## Why It's Hidden
- The non-throwing API is convenient for most callers who just want "data or nothing"
- Budget command already handles null as "no price available" which is partially correct
- Network errors are transient and self-heal on next request
- The logger captures the error for debugging, just not for the caller
