# OPT-001: api-worker calls `LocalizationService.setLocale()` per-request without memoization

- **Impact:** LOW
- **Category:** Redundant Work
- **Files:**
  - [`apps/api-worker/src/routes/dyes.ts:57`](../../../../apps/api-worker/src/routes/dyes.ts#L57), [`:99`](../../../../apps/api-worker/src/routes/dyes.ts#L99), [`:187`](../../../../apps/api-worker/src/routes/dyes.ts#L187), [`:224`](../../../../apps/api-worker/src/routes/dyes.ts#L224), [`:303`](../../../../apps/api-worker/src/routes/dyes.ts#L303)
  - [`apps/api-worker/src/routes/match.ts:81`](../../../../apps/api-worker/src/routes/match.ts#L81), [`:135`](../../../../apps/api-worker/src/routes/match.ts#L135)

## Description

Every non-default-locale request in the api-worker calls `await LocalizationService.setLocale(locale)` near the top of its handler, then proceeds to call `LocalizationService.getDyeName(dye.itemID)`:

```typescript
// apps/api-worker/src/routes/dyes.ts:55-65 (representative)
const locale = c.req.query('lang') ?? 'en';
await LocalizationService.setLocale(locale);
// ... business logic ...
const localizedName = LocalizationService.getDyeName(dye.itemID) || undefined;
```

Per the 2026-04-07 audit (BUG-008), `LocalizationService` is eagerly initialized at module load and `setLocale` is a synchronous-ish state setter (no I/O). So the per-request cost is small. But:

- It's still an `await` and a state mutation on a **module-scoped singleton**.
- The 7 call sites set the same locale repeatedly within a single request's lifetime.
- The pattern is fragile: if a future refactor makes `setLocale` actually do work (load a different bundle, hit KV, etc.), the inefficiency would silently grow.

## Current Safety

`setLocale` is observed to be a cheap state setter today. No measurable hot-path impact on latency.

## Recommendation

Two complementary cleanups:

**1. Set the locale once per request via middleware.**

Instead of every handler calling `setLocale`, register a middleware on `/v1/*` that reads `?lang=` and calls `setLocale` once:

```typescript
// apps/api-worker/src/middleware/locale.ts (new)
import { LocalizationService } from '@xivdyetools/core';

export const localeMiddleware = async (c: Context, next: () => Promise<void>) => {
  const lang = c.req.query('lang') ?? 'en';
  await LocalizationService.setLocale(lang);
  c.set('locale', lang);
  await next();
};
```

Then handlers can skip the per-call `setLocale` and just read `c.get('locale')` if they need the resolved code.

**2. Long-term: switch from singleton state to per-call locale parameter.**

The cleaner architecture is `LocalizationService.getDyeName(itemID, locale)` — passing the locale explicitly so there's no global state to mutate. The [`TranslationProvider`](../../../../packages/core/src/services/localization/TranslationProvider.ts) already accepts locale as a parameter; the singleton wrapping in `LocalizationService` is what creates the per-request `setLocale` dance. Moving locale to a parameter eliminates an entire category of "wrong locale shown" bugs in concurrent isolates.

## Effort

LOW for option 1 (~15 lines); MEDIUM for option 2 (touches all callers monorepo-wide). Option 1 is the immediate win.
