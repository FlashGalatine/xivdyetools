# XIV Dye Tools — i18n Audit

**Date:** 2026-04-28
**Scope:** Full xivdyetools monorepo (packages + apps)
**Languages:** en, ja, de, fr, ko, zh (6 locales)
**Auditor:** i18n-manager skill

---

## Executive Summary

| Area | Status | Detail |
|------|--------|--------|
| Locale key parity | OK | 3 stores × 6 locales — **100% structural match** |
| Untranslated bleed-through | NEEDS REVIEW | ~150 string-equality matches; mostly false positives (loanwords/acronyms), ~10–15 real |
| CJK font subset script | BROKEN | Path bug post-monorepo restructure — silently skips core dye names |
| CJK font subset coverage | DEGRADED | Subsets 481 KiB / 814 KiB; KR oversized vs. Hangul codepoint count |
| og-worker localization | MISSING | Hardcoded English `TOOL_NAMES`/`HARMONY_NAMES`/etc.; no CJK fonts bundled |
| Hardcoded UI strings | LOW–MED | Worker handlers mostly i18n; og-worker the main offender |

The good news: every locale store has perfect key parity, the build pipeline correctly fans CSV → JSON, and the discord-worker correctly bundles CJK fallback fonts. The bad news: **the subsetting script is broken** and may have been silently producing fonts missing dye-name glyphs since the monorepo restructure. og-worker is essentially monolingual.

## 4. Build Pipeline Observations

The locale pipeline is `fetch_dye_names.py` → `dyenames.csv` → `build-locales.ts` → `src/data/locales/*.json`. Reading [build-locales.ts](../../../packages/core/scripts/build-locales.ts) surfaced several patterns worth documenting:

### 4.1 ko/zh labels rely on hardcoded fallback (not YAML)

[build-locales.ts:140-163](../../../packages/core/scripts/build-locales.ts#L140-L163) defines a `fallbackLabels` object with hardcoded ko/zh translations because the source `localize.yaml` apparently lacks ko/zh entries. This works but is **fragile**: if a new label is added to YAML for en/ja/de/fr but not to the ko/zh fallback, those locales silently lose the key.

**Mitigation already in place:** the parity audit (§2) confirms current symmetry — all 6 locales have the same 7 labels.

**Risk:** future YAML additions could break parity without warning.

### 4.2 English fallback for missing CSV columns

[build-locales.ts:200](../../../packages/core/scripts/build-locales.ts#L200): `const name = row[nameColumn]?.trim() || row[fallbackColumn]?.trim();`

If a row's locale column is empty, the English name is used. This is the right behavior for partial translations, but it does mean **untranslated dye names appear as English in ko/zh JSON**. Spot-checking via grep on `core/zh.json` showed 0 such cases (zh.json had no English dye names) — `dyenames.csv` is fully populated for all 125 dyes.

### 4.3 `meta.generated` timestamp

Each generated JSON includes `meta.generated`: a timestamp like `"2026-04-28T20:51:07.659Z"`. This means **every rebuild produces a diff** in the dist artifacts even when content hasn't changed. The audit ignored this field for parity comparison; suggest considering whether to drop it or pin it (e.g., to git commit hash) to reduce diff noise.

### 4.4 Patch 7.5 dye consolidation

Per project memory, Patch 7.5 consolidated dye items A=52254 / B=52255 / C=52256 with ko/zh names still pending. Confirmed in `core/ko.json` and `core/zh.json` that these IDs do appear and have non-English names — consolidation framework is live as expected.

## 5. CJK Font Subsetting

**This section contains the most critical finding in the audit.**

### 5.1 The subset script has a stale path

[apps/discord-worker/scripts/subset-cjk-fonts.py:46](../../../apps/discord-worker/scripts/subset-cjk-fonts.py#L46):

```python
CORE_LOCALES_DIR = os.path.join(PROJECT_ROOT, "xivdyetools-core", "src", "data", "locales")
```

But `PROJECT_ROOT` resolves to `apps/` (parent of `discord-worker/`). The path `apps/xivdyetools-core/src/data/locales` **does not exist** — core was moved to `packages/core/` in the monorepo restructure.

The script "handles" this by silently skipping with `if os.path.exists(path)` ([line 81](../../../apps/discord-worker/scripts/subset-cjk-fonts.py#L81)) — printing `"Core ja.json: not found (skipped)"` for all 5 languages. **Only bot UI strings get included in the subset.**

#### Impact

If a maintainer runs `python scripts/subset-cjk-fonts.py` after a dye name change:
1. Script appears to succeed (exits 0)
2. Outputs are written to `NotoSansSC-Subset.ttf` / `NotoSansKR-Subset.ttf`
3. New subsets contain only bot UI characters — **dye name characters are missing**
4. Discord bot renders dye names with tofu (□) for missing CJK glyphs
5. No build error, no test failure

The current subsets in the repo (481 KiB SC, 814 KiB KR) are likely from a pre-restructure run when the path was correct. They probably still cover most dye names but **will not be updated correctly going forward** until the path is fixed.

#### Fix

```python
# apps/discord-worker/scripts/subset-cjk-fonts.py
- PROJECT_ROOT = os.path.dirname(WORKER_ROOT)
- CORE_LOCALES_DIR = os.path.join(PROJECT_ROOT, "xivdyetools-core", "src", "data", "locales")
+ MONOREPO_ROOT = os.path.dirname(os.path.dirname(WORKER_ROOT))  # apps/discord-worker → apps → root
+ CORE_LOCALES_DIR = os.path.join(MONOREPO_ROOT, "packages", "core", "src", "data", "locales")
```

Also recommend changing the silent skip to a hard fail when locale dirs are missing — this is exactly the kind of bug a script should refuse to run with.

### 5.2 Required codepoints (all 3 stores combined)

| Script block | Codepoints needed | ja | ko | zh |
|--------------|------------------:|---:|---:|---:|
| CJK Unified Ideographs (U+4E00–9FFF) | **1,124** | 482 | 0 | 911 |
| Hangul Syllables (U+AC00–D7AF) | **561** | 0 | 561 | 0 |
| Katakana (U+30A0–30FF) | **77** | 77 | 0 | 0 |
| Hiragana (U+3040–309F) | **57** | 57 | 0 | 0 |
| CJK Symbols (U+3000–303F) | **6** | — | — | — |
| Fullwidth Forms (U+FF00–FFEF) | **7** | — | — | — |
| **Total non-Latin** | **~1,832** | | | |

### 5.3 Subset sizes vs. content

| File | Actual size | Comment in [fonts.ts:34-39](../../../apps/discord-worker/src/services/fonts.ts#L34-L39) | Verdict |
|------|------------:|----|---------|
| `NotoSansSC-Subset.ttf` | **481 KiB** | "~222 KiB" | Doc says 222 KiB, actual is 2.2× larger |
| `NotoSansKR-Subset.ttf` | **814 KiB** | "~155 KiB" | Doc says 155 KiB, actual is 5.3× larger |

`✶ Insight ─────────────────────────────────────`
The subset for KR at 814 KiB for ~561 Hangul syllables is suspicious. As a back-of-envelope check, Hangul glyphs typically average ~200–400 bytes each in TTF; 561 syllables × ~350 bytes ≈ 200 KiB. The 814 KiB size suggests **either**: (a) the subsetter retained the full layout/feature/cmap tables ([line 127](../../../apps/discord-worker/scripts/subset-cjk-fonts.py#L127): `options.layout_features = ['*']`), which preserves contextual shaping data not strictly needed for these characters, **or** (b) more glyphs are present than required. Re-running the subsetter with the fixed path may shrink it.
`─────────────────────────────────────────────────`

The script keeps `layout_features = ['*']` and `name_IDs = ['*']`, both of which add weight. For a typeface used only for atomic syllables (no shaping), these can usually be reduced to a smaller set safely.

### 5.4 Coverage verification

`fontTools` is not available in the audit environment, so direct glyph-cmap diffing was not performed. Recommend running the verification script in §9 **after** the path fix to confirm the new subsets cover all 1,832 codepoints.

### 5.5 Worker bundle impact

Per project memory, the discord-worker bundle is ~8 MiB (Cloudflare paid plan limit: 10 MiB). The two subset fonts together are 1.27 MiB — about 16% of the bundle. If the KR subset can be reduced from 814 → ~250 KiB, that's ~7% bundle savings, useful headroom against the limit.

## 6. Font Stack Validation

Audited every place where fonts are referenced and bundled.

### 6.1 discord-worker — OK

- Bundle: SpaceGrotesk + Onest + Habibi + NotoSansSC-Subset + NotoSansKR-Subset
- Font config: [services/fonts.ts](../../../apps/discord-worker/src/services/fonts.ts) loads all 5
- @xivdyetools/svg uses `FONTS.headerCjk` / `FONTS.primaryCjk` ([base.ts:262-264](../../../packages/svg/src/base.ts#L262-L264)): `'Onest, Noto Sans SC, Noto Sans KR'`
- Verified usage across [random-dyes-grid.ts](../../../packages/svg/src/random-dyes-grid.ts), [palette-grid.ts](../../../packages/svg/src/palette-grid.ts), [comparison-grid.ts](../../../packages/svg/src/comparison-grid.ts), [dye-info-card.ts](../../../packages/svg/src/dye-info-card.ts), etc.

CJK characters in dye names will render correctly through the fallback chain.

### 6.2 og-worker — DEGRADED

- Bundle: SpaceGrotesk + Onest + Habibi only ([apps/og-worker/src/services/fonts.ts](../../../apps/og-worker/src/services/fonts.ts))
- **No NotoSansSC, no NotoSansKR**
- og-worker has its own `./services/svg` (not the shared @xivdyetools/svg package), so its renderers don't use `FONTS.headerCjk`. Currently safe because content is hardcoded English (§7).
- **Risk if og-worker ever localizes:** any CJK character in a dye name passed to its SVG renderer will become tofu (□). This is a structural blocker for OG-image localization.

The HTML fallback CSS at [og-data-generator.ts:376](../../../apps/og-worker/src/og-data-generator.ts#L376) uses `-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif`. For HTML (not SVG-to-PNG), the user's browser supplies CJK fallback from system fonts, so this is fine for crawler-rendered HTML. The risk is only for the PNG path.

### 6.3 web-app — OK (browser handles fallback)

The web-app's font stack is [Cinzel + Lexend + Habibi + Lexend Giga], all Latin-only woff2 files. CJK rendering happens via the user's system fonts when the browser falls through the CSS stack. This is the standard approach for browser apps and is fine.

## 9. Verification Script (run after fix #1)

```python
# Verify subset font coverage matches all locale codepoints across all stores.
# Run from monorepo root after pip install fonttools.
import json, os
from fontTools.ttLib import TTFont

LOCALE_DIRS = [
    'packages/core/src/data/locales',
    'packages/bot-i18n/src/locales',
    'apps/web-app/src/locales',  # optional — discord-worker doesn't render web-app strings
]
LANGS = ['ja', 'ko', 'zh', 'de', 'fr']

def collect(path, chars):
    def walk(o):
        if isinstance(o, str):
            for c in o: chars.add(ord(c))
        elif isinstance(o, dict):
            for v in o.values(): walk(v)
        elif isinstance(o, list):
            for v in o: walk(v)
    with open(path, encoding='utf-8') as f:
        walk(json.load(f))

needed = set()
for d in LOCALE_DIRS:
    for lang in LANGS:
        p = os.path.join(d, f'{lang}.json')
        if os.path.exists(p): collect(p, needed)

for label, fp, ranges in [
    ('SC subset', 'apps/discord-worker/src/fonts/NotoSansSC-Subset.ttf',
     [(0x4E00,0x9FFF),(0x3400,0x4DBF),(0x30A0,0x30FF),(0x3040,0x309F),(0x3000,0x303F),(0xFF00,0xFFEF)]),
    ('KR subset', 'apps/discord-worker/src/fonts/NotoSansKR-Subset.ttf',
     [(0xAC00,0xD7AF),(0x1100,0x11FF)]),
]:
    font = TTFont(fp); covered = set(font.getBestCmap().keys()); font.close()
    in_range = lambda cp: any(s<=cp<=e for s,e in ranges)
    locale_in_range = {cp for cp in needed if in_range(cp)}
    missing = locale_in_range - covered
    print(f'{label}: needs {len(locale_in_range)}, covers {len(locale_in_range - missing)}, missing {len(missing)}')
    if missing:
        print(f'  Missing: {"".join(chr(c) for c in sorted(missing) if c < 0x10000)[:200]}')
```

A clean run prints `missing 0` for both subsets.
