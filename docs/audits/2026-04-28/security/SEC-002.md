# SEC-002 — `worker-middleware` rate-limit `keyExtractor` is a quiet footgun

- **Severity:** INFO (documentation hardening)
- **Status:** OPEN
- **File:** [`packages/worker-middleware/src/rate-limit.ts:46`](../../../../packages/worker-middleware/src/rate-limit.ts#L46)
- **Discovered:** 2026-04-28

## Summary

The shared `rateLimitMiddleware` factory accepts a fully arbitrary `keyExtractor: (c: Context) => string` callback for deriving the rate-limit key. The package itself is correct — it doesn't trust any client-controlled header — but a future caller who writes a naive extractor like `(c) => c.req.header('X-Forwarded-For')` would silently re-introduce the spoofing class fixed by `BUG-018` / `2026-04-07/FINDING-006`. Today every documented usage is correct; the risk is for future callers who don't read the prior audit history.

## Evidence

```typescript
// packages/worker-middleware/src/rate-limit.ts:30-46
export interface RateLimitMiddlewareOptions {
  // ...
  /**
   * Extract the rate limit key from the Hono context.
   *
   * Common patterns:
   * - IP-based: `(c) => getClientIp(c.req.raw)`
   * - User-based: `(c) => c.get('userId')`
   * - Compound: `(c) => \`${getClientIp(c.req.raw)}:${c.req.path}\``
   */
  keyExtractor: (c: Context) => string;
  // ...
}
```

The JSDoc shows the right examples but does not warn against the wrong ones. `getClientIp` itself is properly hardened ([`packages/rate-limiter/src/ip.ts:53-79`](../../../../packages/rate-limiter/src/ip.ts#L53-L79)) — `trustXForwardedFor` defaults to `false`, and `CF-Connecting-IP` is preferred. But that hardening is bypassed if a worker author writes their own extractor.

## Risk Assessment

A worker that writes `keyExtractor: (c) => c.req.header('cf-connecting-ip') ?? c.req.header('x-forwarded-for') ?? 'anon'` would:

1. Be allowed by the middleware factory (no validation).
2. Pass code review by anyone who doesn't know the prior audit history.
3. Produce a per-IP rate limiter that is trivially bypassable: an attacker sets `X-Forwarded-For: <random UUID>` on each request and gets unlimited quota.

This is exactly the bug class that `BUG-018` previously closed at the **library** layer. Re-introducing it at the **caller** layer is the same vulnerability with a different file path.

| Dimension | Value |
|-----------|-------|
| Likelihood | **Low** (every existing caller is correct, and the library examples steer toward `getClientIp`) |
| Impact if triggered | **High** (rate-limit bypass on a public endpoint → cost amplification, scraping, brute-force) |
| Effort to fix | **Low** (JSDoc warning + lint rule, or a sentinel default) |

**Severity:** INFO — process/documentation hardening, no current vulnerability.

## Recommended Fixes (any of these is sufficient)

### Option A — Strengthen the JSDoc (smallest)

Add an explicit warning above the `keyExtractor` field:

```typescript
/**
 * Extract the rate limit key from the Hono context.
 *
 * SECURITY: Do NOT derive keys from client-controlled headers like
 * `X-Forwarded-For`. They can be trivially spoofed and will defeat
 * the rate limiter. Use `getClientIp(c.req.raw)` from
 * `@xivdyetools/rate-limiter` — it prefers Cloudflare's
 * `CF-Connecting-IP` and ignores `X-Forwarded-For` by default.
 * See `BUG-018` / 2026-04-07/FINDING-006 for prior context.
 *
 * Common patterns:
 * - IP-based:  (c) => getClientIp(c.req.raw)
 * - User-based: (c) => c.get('userId')
 * - Compound:  (c) => `${getClientIp(c.req.raw)}:${c.req.path}`
 */
keyExtractor: (c: Context) => string;
```

### Option B — Provide a default

Make `keyExtractor` optional and default to `(c) => getClientIp(c.req.raw)`. Callers opting out of IP-based limiting must explicitly override. This eliminates the footgun by default.

### Option C — Type-level steering

Export a small set of named extractors (`ipKeyExtractor`, `userIdKeyExtractor`, `compoundKeyExtractor(prefix)`) and document them as the recommended path. Custom callbacks remain possible for advanced cases.

Option A is the cheapest and recommended first step; Option B is the most foolproof and worth doing if a new worker is added in the next sprint.
