# SEC-003 — Dev-only dependency vulnerabilities (esbuild, vite via vitepress) — STANDING

- **Severity:** LOW (development-only, no production exposure)
- **Status:** STANDING — partial fix in place, full fix blocked by upstream
- **File:** [`apps/api-docs/package.json:12`](../../../../apps/api-docs/package.json#L12), [`package.json:23-27`](../../../../package.json#L23-L27)
- **Discovered:** 2026-04-07 (continued tracking)

## Summary

`pnpm audit --audit-level=low` reports **2 moderate** vulnerabilities in the development dependency tree. Both flow through `apps/api-docs` → `vitepress ^1.6.4` → `vite` and `esbuild`. There is no production impact: api-docs is a static VitePress site that renders to HTML at build time, and neither esbuild's dev-server bug nor vite's `.map`-handling path traversal has any runtime presence in any deployed Cloudflare Worker. The previously-tracked `rollup` vulnerability is **closed** — the root `package.json` carries an explicit `pnpm.overrides.rollup: ">=4.59.0"` that knocks the bad versions out of the resolution graph for everything downstream, including `apps/stoat-worker`'s `tsup` build chain.

This finding is identical in posture to `2026-04-07/SEC-006`. It is repeated here only so the 2026-04-28 audit cycle has a complete picture.

## Production Impact: None

| Vuln | What it does | Reaches production? |
|------|--------------|---------------------|
| esbuild ≤ 0.24.2 (GHSA-67mh-4wv8-2f99) | Allows any origin to send requests to esbuild's local dev server and read responses | **No.** esbuild only runs as part of `vitepress dev` on a developer's laptop. No worker bundles esbuild. |
| vite ≤ 6.4.1 (GHSA-4w7w-66w2-5vf9) | Path traversal via `.map` handling under `vite preview` | **No.** Same containment — vite only runs locally for api-docs, not at deploy time on Cloudflare. |

The web-app and maintainer also use vite, but they pin **vite ^8.0.x** directly in their own `package.json` — those versions are above `6.4.1` and unaffected. The vulnerable version is reached *only* through `vitepress`'s pinned older `vite`.

## Why the fix is blocked

`vitepress ^1.6.4` (the latest 1.x release) constrains its peer/transitive `vite` to a 5.x/6.x range that contains the vulnerable versions. VitePress 2.x (currently in alpha at the time of this audit) bumps to vite 7+, which clears the chain. Until VitePress publishes a stable 2.0, the pragmatic options are:

1. **Wait** (current posture). Acceptable because the vuln is dev-only and the api-docs maintainer audience is internal.
2. **Add narrow pnpm overrides for vite/esbuild.** Risky: VitePress 1.x may not actually run on vite 7+ or esbuild 0.25+ without bug fixes that don't exist. We tried this in early 2026-Q1 (per `2026-04-07/SEC-006`) and reverted because builds broke.
3. **Pin VitePress to a 2.x-alpha.** Trades a dev-server CVE for unstable docs builds. Net negative for now.

Option 1 is correct. Re-evaluate when VitePress 2.0 stable ships.
