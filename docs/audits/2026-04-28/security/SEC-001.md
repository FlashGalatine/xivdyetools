# SEC-001 — `innerHTML` interpolation of XIVAuth character data in auth-button

- **Severity:** LOW
- **Status:** OPEN
- **File:** [`apps/web-app/src/components/auth-button.ts:222`](../../../../apps/web-app/src/components/auth-button.ts#L222)
- **Discovered:** 2026-04-28

## Summary

The auth dropdown renders the logged-in XIVAuth user's character name and server by **interpolating them into an `innerHTML` template literal**, rather than constructing DOM nodes with `textContent`. Although the data originates from the server-validated XIVAuth OAuth response (not direct user input), this pattern violates the project's own documented XSS-prevention rule and creates an upstream-trust dependency where there shouldn't be one.

## Evidence

```typescript
// apps/web-app/src/components/auth-button.ts:215-224
if (user.auth_provider === 'xivauth' && user.primary_character) {
  const characterInfo = this.createElement('div', {
    className: 'text-xs mt-1 flex items-center gap-1',
    attributes: {
      style: 'color: var(--theme-text); opacity: 0.7;',
    },
  });
  characterInfo.innerHTML = `<span>${user.primary_character.name}</span><span class="opacity-50">@</span><span>${user.primary_character.server}</span>${user.primary_character.verified ? `<span class="text-green-500 ml-1" title="${LanguageService.t('auth.verified')}">✓</span>` : ''}`;
  header.appendChild(characterInfo);
}
```

`user.primary_character.name` and `user.primary_character.server` are interpolated **without escaping**. If either field ever contained `<`, `>`, `"`, or other HTML-significant characters, those bytes would be parsed as markup.

## Why this is a finding (despite "trusted" data)

The project's own [`apps/web-app/CLAUDE.md`](../../../../apps/web-app/CLAUDE.md) — under **Security Patterns → XSS Prevention** — documents the rule:

> **User Content**: Always use `textContent` for user-controlled data:
>
> ```typescript
> // ✅ SAFE: User content via textContent
> element.textContent = userProvidedString;
>
> // ⚠️ ONLY for static icons from *-icons.ts
> element.innerHTML = ICON_SAVE;  // from ui-icons.ts
> ```

A character name from an OAuth provider is *user-controlled data*. It enters the app from outside, transits a server we don't operate (XIVAuth at the time of issuance), and is shaped by whatever validation rules XIVAuth chose to enforce — which may or may not match this app's threat model. The other `innerHTML` uses on the same component (lines 97, 109, 239) are sanctioned by the rule because they only embed compile-time icon constants (`ICON_DISCORD`, `ICON_XIVAUTH`) and i18n strings from project-controlled locale files. Line 222 is the exception, and it's worth fixing for three reasons:

1. **Convention compliance.** The component should not violate its own project's security rule, even if the current upstream is benign. Future readers will assume the rule holds and may copy this pattern into riskier contexts.
2. **Defense in depth.** A compromise of XIVAuth, a misconfigured XIVAuth deployment, a future XIVAuth feature that loosens character-name validation (e.g., to support extended Unicode), or a malicious account that bypasses XIVAuth's validation could all inject markup that this component would dutifully render. None of those scenarios is exotic — character-name validation across game APIs is historically inconsistent.
3. **No upside.** The `innerHTML` form is not faster or simpler than DOM construction here. Three `<span>` nodes built with `createElement` + `textContent` is the same line count.

## Reproduction (hypothetical)

If XIVAuth's `/me` endpoint were modified or compromised to return:

```json
{
  "primary_character": {
    "name": "Abe<script>alert(1)</script>",
    "server": "Aether",
    "verified": true
  }
}
```

…the dropdown would execute `alert(1)` on every render in a logged-in browser. The CSP (`script-src 'self'`) blocks inline `<script>` tags, so the immediate impact in production today is limited to **HTML structural injection** (e.g., closing the surrounding span, breaking layout, or rendering attacker-chosen attributes via `<img onerror>`). Without CSP, this is full XSS. Either way, the surface is unnecessary.

## Recommended fix

Replace the `innerHTML` template with DOM construction using the existing `createElement` helper plus `textContent`:

```typescript
const charSpan = this.createElement('span', { textContent: user.primary_character.name });
const sep = this.createElement('span', { className: 'opacity-50', textContent: '@' });
const serverSpan = this.createElement('span', { textContent: user.primary_character.server });
characterInfo.appendChild(charSpan);
characterInfo.appendChild(sep);
characterInfo.appendChild(serverSpan);

if (user.primary_character.verified) {
  const check = this.createElement('span', {
    className: 'text-green-500 ml-1',
    textContent: '✓',
    attributes: { title: LanguageService.t('auth.verified') },
  });
  characterInfo.appendChild(check);
}
```

`textContent` setters serialize their argument as a text node, so `<` becomes `&lt;` automatically — no escaping helper required.

## Risk Assessment

| Dimension | Value |
|-----------|-------|
| Likelihood today | **Very Low** — XIVAuth is operated by a known partner and constrains character names to FFXIV-valid strings |
| Likelihood under upstream compromise | **Medium** — would require XIVAuth-side breach or trust-relationship change |
| Impact under current CSP | **Low** — `script-src 'self'` blocks inline scripts; structural injection only |
| Impact without CSP | **High** — full DOM XSS in an authenticated session |
| Effort to fix | **Trivial** (~10 lines) |

**Severity:** LOW — Defense-in-depth fix that aligns the code with documented project policy.
