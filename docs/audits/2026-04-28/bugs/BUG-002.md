# BUG-002: Latent gap — `TranslationProvider.getDyeName()` returns `null` for Patch 7.5 consolidated itemIDs

- **Severity:** LOW (downgraded from MEDIUM during verification — see "Caller analysis" below)
- **Category:** Defensive Hardening / API Contract
- **Files:**
  - [`packages/core/src/services/localization/TranslationProvider.ts:83-101`](../../../../packages/core/src/services/localization/TranslationProvider.ts#L83-L101)
  - [`packages/core/src/config/consolidated-ids.ts:124-127`](../../../../packages/core/src/config/consolidated-ids.ts#L124-L127)
  - [`packages/core/scripts/dyenames.csv`](../../../../packages/core/scripts/dyenames.csv) (does not contain rows for itemIDs 52254 / 52255 / 52256)

## Description

`TranslationProvider.getDyeName(itemID, locale)` looks up dye names by itemID in the locale registry that was generated from `dyenames.csv`. Patch 7.5 introduced three consolidated dye items — itemIDs 52254 (Standard Spectrum Dye), 52255 (Wide Spectrum #1 Dye), 52256 (Wide Spectrum #2 Dye) — but the consolidated metadata lives in [`consolidated-ids.ts`](../../../../packages/core/src/config/consolidated-ids.ts), **not** in the CSV-driven locale JSONs.

```typescript
// TranslationProvider.ts:83
getDyeName(itemID: number, locale: LocaleCode): string | null {
  const localeData = this.registry.getLocale(locale);
  const idStr = String(itemID);
  if (localeData?.dyeNames[idStr]) return localeData.dyeNames[idStr];
  // English fallback
  if (locale !== 'en') {
    const englishData = this.registry.getLocale('en');
    if (englishData?.dyeNames[idStr]) return englishData.dyeNames[idStr];
  }
  return null;  // ← returns null for 52254 / 52255 / 52256
}
```

The dedicated lookup `getConsolidatedDyeName(type, locale)` works correctly because it sources from `CONSOLIDATED_DYES` constants directly. The two lookups are fully separate code paths.

## Caller analysis (verification — 2026-04-28)

A monorepo-wide grep for `getDyeName(` (excluding tests) was run. Findings:

- **30+ web-app components** call `LanguageService.getDyeName(dye.itemID) || dye.name` where `dye` comes from `dyeDatabase.allDyes()` iteration.
- **api-worker** (`routes/dyes.ts`, `routes/match.ts`) calls `LocalizationService.getDyeName(dye.itemID)` similarly.
- `colors_xiv.json` does **not** contain entries for 52254 / 52255 / 52256, and `DyeDatabase.ts` does **not** merge `CONSOLIDATED_DYES` into the iterable database. `dyeDatabase.allDyes()` returns only the 125 original dyes (each carrying a `consolidationType: 'A' | 'B' | 'C' | null` flag pointing to its bucket).

**Therefore:** no caller currently passes 52254 / 52255 / 52256 to `getDyeName()`. The gap is **latent** — exposed only if a future change either (a) adds consolidated dyes to the iterable database, or (b) introduces a code path that looks up a consolidated ID's display name via the general lookup instead of the dedicated one.

## Impact

- **Today:** None observed. The two lookups are correctly separated by intent.
- **Latent risk:** A future contributor wiring up "show consolidated dye name in budget summary" might intuitively reach for `getDyeName(getMarketItemID(dye), locale)` and get `null` for non-English locales, falling back to whatever default surrounds the call (or rendering nothing).

## Recommendation

Two viable hardening approaches; one or the other is enough:

**Option A (preferred) — teach `getDyeName` to consult consolidated metadata:**

```typescript
import { CONSOLIDATED_DYES, CONSOLIDATED_IDS } from '../../config/consolidated-ids.js';

getDyeName(itemID: number, locale: LocaleCode): string | null {
  // ... existing locale-registry lookup ...

  // Fallback: consolidated dyes live outside the CSV-driven registry
  for (const type of ['A', 'B', 'C'] as const) {
    if (CONSOLIDATED_IDS[type] === itemID) {
      return CONSOLIDATED_DYES[type].names[locale] ?? CONSOLIDATED_DYES[type].names.en;
    }
  }
  return null;
}
```

**Option B — document the contract:**

Add a JSDoc note to `getDyeName` clarifying that callers looking up consolidated dye names must use [`getConsolidatedDyeName()`](../../../../packages/core/src/config/consolidated-ids.ts#L124) instead. Less defensive than Option A but cheaper.

## Effort

LOW (Option A: ~10 lines + 1 test) or TRIVIAL (Option B: a JSDoc block).

## Resolution

**Status:** OPEN — non-functional today; defensive hardening recommended.
