# BUG-004: api-worker `kvLimiter` cached at module scope without binding validation

- **Severity:** LOW (downgraded from scout's MED — Cloudflare Workers do not swap KV bindings within an isolate)
- **Category:** Code Hygiene / Pattern Consistency
- **File:** [`apps/api-worker/src/middleware/rate-limit.ts:18-25`](../../../../apps/api-worker/src/middleware/rate-limit.ts#L18-L25)

## Description

The api-worker rate-limit middleware caches a `KVRateLimiter` instance in module scope using a singleton-on-first-call pattern:

```typescript
// apps/api-worker/src/middleware/rate-limit.ts:18-25
let kvLimiter: KVRateLimiter | null = null;

function getKVLimiter(kv: KVNamespace): KVRateLimiter {
  if (!kvLimiter) {
    kvLimiter = new KVRateLimiter({ kv, keyPrefix: 'api:ip:' });
  }
  return kvLimiter;
}
```

The `kv` parameter is captured on first invocation and never re-validated. If a future change causes the worker to handle multiple KV bindings (e.g., environment-tagged rate-limit namespaces, multi-tenant rate limiting), the cached instance would silently use the first binding observed — a behavior that's invisible until something subtle breaks.

## Comparison with sibling workers

[`apps/presets-api/src/middleware/rate-limit.ts`](../../../../apps/presets-api/src/middleware/rate-limit.ts) uses a different pattern: the backend factory is called per-request via the `rateLimitMiddleware` factory's `backend: (c) => ...` hook, which receives the per-request context. There's no module-scope cache to go stale.

## Impact

- **Today:** None — Cloudflare Workers don't swap KV bindings mid-isolate, and api-worker has only one `RATE_LIMIT` KV binding. The cached singleton is functionally identical to a per-request lookup.
- **Future risk:** If api-worker ever adds a second rate-limit namespace (e.g., authenticated vs anonymous tiers, separate test/prod namespaces selected at runtime), this pattern will silently bind to whichever was used first.
- Inconsistent with the per-request-factory pattern used elsewhere; obscures whether the singleton is intentional or accidental.

## Recommendation

Drop the module-scope cache and let `KVRateLimiter` be constructed per-request via the `backend` factory:

```typescript
export const rateLimitMiddleware = createRateLimitMiddleware({
  backend: (c: Context<{ Bindings: Env }>) =>
    new KVRateLimiter({ kv: c.env.RATE_LIMIT, keyPrefix: 'api:ip:' }),
  // ... rest unchanged
});
```

`KVRateLimiter` construction is cheap (no I/O; just stores the binding reference). The micro-optimization of caching it doesn't measurably help, and the consistency win is worth more.

## Effort

TRIVIAL — delete the singleton, inline the construction. ~5 lines removed.

## Resolution

**Status:** OPEN — not breaking, but recommended for pattern consistency.
