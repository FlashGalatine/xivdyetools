# [BUG-001]: `APIService` batch methods don't chunk and throw uncaught above 100 items

## Severity
MEDIUM (latent — published library code; reachable by any consumer on a cold cache, but no in-repo caller currently triggers it)

## Type
Boundary condition / Error-handling gap

## Location
- File: `packages/core/src/services/APIService.ts`
- Functions: `getPricesForItems` (943), `getPricesForDataCenter` (997), `fetchBatchPriceData` (638), `buildBatchApiUrl` (865)

## Description
`getPricesForItems` / `getPricesForDataCenter` are cache-aware "batch" helpers: they check the
cache for each item, collect the uncached ones, and fetch them. They pass the **entire**
uncached list to `fetchBatchPriceData`, which builds a single URL via `buildBatchApiUrl`.
`buildBatchApiUrl` **throws** an `AppError` when given more than 100 IDs:

```typescript
// buildBatchApiUrl (line 872)
if (itemIDs.length > 100) {
  throw new AppError(ErrorCode.INVALID_INPUT, `Cannot fetch more than 100 items in one batch (got ${itemIDs.length})`, 'warning');
}
```

Two problems compound:

1. **No chunking.** Neither the public methods nor `fetchBatchPriceData` splits the list into
   ≤100 windows. The Universalis 100-item limit is enforced by *throwing*, not by batching.
2. **The throw bypasses the method's own resilience.** `fetchBatchPriceData` wraps the fetch in
   a try/catch that deliberately returns an empty `Map` on failure (so a transient API error is
   non-fatal). But `buildBatchApiUrl` is called on **line 651, *before* the `try` on line 653**,
   so its validation throw escapes the catch and propagates out through `getPricesForItems`
   (which has no try/catch) to the caller.

## Reproduction Scenario
On a **cold cache**, call `getPricesForDataCenter(ids, 'Crystal')` (or `getPricesForItems`) with
the full market-tradeable dye set. The dye database has 136 entries (125 standard tradeable +
11 Facewear); the standard tradeable set alone is 125 > 100. All are uncached on first call →
`uncachedItemIDs.length > 100` → `AppError` thrown and **unhandled**.

```typescript
const api = new APIService();           // empty MemoryCacheBackend
const ids = dyeService.getAllDyes().filter(d => d.itemID > 0).map(d => d.itemID); // 125
await api.getPricesForDataCenter(ids, 'Crystal'); // throws AppError, not a partial Map
```

## Why It's Hidden
- The in-repo `discord-worker` `/budget` command does **not** use these methods — it has its own
  `fetchPricesBatched` that chunks into 100s (`apps/discord-worker/src/services/budget/universalis-client.ts:255`)
  and dedups to the 3 consolidated IDs, so it never hits the limit.
- The `web-app` doesn't call these methods directly.
- Unit tests likely exercise small ID arrays (≤100) or pre-warmed caches, so the >100 cold-cache
  path is never hit.
- It only triggers for an external/future consumer of the published `@xivdyetools/core` calling
  the documented batch API with a realistic full-database query — exactly the use the method name
  invites.

## Evidence
```typescript
// fetchBatchPriceData (638) — buildBatchApiUrl is OUTSIDE the try
const url = this.buildBatchApiUrl(itemIDs, dataCenterID);  // line 651  ← can throw, unguarded
try {                                                       // line 653
  const data = await retry(() => this.fetchWithTimeout(url, ...), ...);
  ...
} catch (error) {
  this.logger.error(...);
  return new Map();                                         // intended resilience — bypassed
}
```

## Suggested Fix
Chunk inside `fetchBatchPriceData` and rate-limit per chunk:

```typescript
private async fetchBatchPriceData(itemIDs: number[], dataCenterID?: string): Promise<Map<number, PriceData>> {
  if (itemIDs.length === 0) return new Map();
  const BATCH = 100;
  const merged = new Map<number, PriceData>();
  for (let i = 0; i < itemIDs.length; i += BATCH) {
    const slice = itemIDs.slice(i, i + BATCH);
    await this.rateLimiter.waitIfNeeded();
    this.rateLimiter.recordRequest();
    try {
      const url = this.buildBatchApiUrl(slice, dataCenterID); // now inside try
      const data = await retry(() => this.fetchWithTimeout(url, UNIVERSALIS_API_TIMEOUT),
        UNIVERSALIS_API_RETRY_COUNT, UNIVERSALIS_API_RETRY_DELAY, this.logger);
      if (data) for (const [id, p] of this.parseBatchApiResponse(data)) merged.set(id, p);
    } catch (error) {
      this.logger.error(`Batch chunk failed (${slice.length} items): ${error instanceof Error ? error.message : 'Unknown'}`);
      // continue with remaining chunks instead of aborting the whole request
    }
  }
  return merged;
}
```

Add a regression test: cold cache, 125-item request → resolves to a `Map` (possibly partial),
never throws.
