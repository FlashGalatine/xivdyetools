# SEC-003: JSON Depth Limiting Inconsistent Across Workers

- **Severity:** LOW
- **Category:** Denial of Service (Potential)
- **Files:**
  - `apps/moderation-worker/src/index.ts:132-145` — HAS depth limiting ✅
  - `apps/presets-api/src/index.ts` — MISSING ❌
  - `apps/discord-worker/src/index.ts` — MISSING ❌
  - `apps/oauth/src/index.ts` — MISSING ❌

## Description

The moderation-worker uses `safeParseJSON()` with `maxDepth: 10`, `validateStructure: true`, and `freezeResult: true` to prevent deeply nested JSON payloads from causing stack overflow or excessive memory consumption. Other workers parse JSON using the default Hono body parser without depth limits.

## Risk

LOW — Cloudflare Workers have a 128MB memory limit and 30-second CPU time limit, which naturally bounds the impact of deeply nested JSON. However, a crafted payload with extreme nesting (e.g., 10,000 levels) could consume disproportionate CPU time during parsing.

## Recommendation

Apply `safeParseJSON()` or equivalent depth limiting to all workers that accept JSON input, particularly:
1. `presets-api` — accepts user-submitted preset data
2. `oauth` — accepts OAuth callback data

The discord-worker is lower priority since its JSON comes from Discord's verified webhook payloads.

## Effort

LOW — The `safeParseJSON()` utility already exists in the moderation-worker; extract to a shared utility or middleware.
