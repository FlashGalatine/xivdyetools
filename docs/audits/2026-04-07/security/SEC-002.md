# SEC-002: `innerHTML` Usage in modal-container.ts

- **Severity:** LOW
- **Category:** Cross-Site Scripting (Potential)
- **File:** `apps/web-app/src/components/modal-container.ts:244`
- **Status:** NEW

## Description

The `modal-container.ts` component sets `content.innerHTML = modal.content` where `modal.content` is a string. While currently only set programmatically (not from user input), this pattern creates a potential XSS vector if future changes introduce user-controlled content.

## Current Mitigations

1. **Strict CSP:** The web app's Content Security Policy includes `script-src 'self'` with no `unsafe-inline`, preventing inline script execution even if HTML is injected
2. **Controlled input:** Modal content is only set from application code, not from untrusted sources
3. **No user input path:** No current code path allows user-supplied strings to reach `modal.content`

## Risk

LOW — Current mitigations are strong. This is a defense-in-depth recommendation to prevent future regression.

## Recommendation

Consider using safer DOM methods for future-proofing:

```typescript
// Instead of: content.innerHTML = modal.content;
// Use: content.textContent = modal.content;  // for text-only modals
// Or: content.replaceChildren(...parsedNodes); // for structured content
```

If HTML rendering is required, use a sanitization library or template literals with Lit's `html` tagged template (which handles escaping).

## Effort

LOW — Single line change, but requires verifying all modal content is text-only or properly structured.
