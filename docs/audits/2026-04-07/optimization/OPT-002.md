# OPT-002: Font Loading May Block on Fetch in discord-worker

- **Impact:** LOW
- **Category:** Latency
- **File:** `apps/discord-worker/src/services/fonts.ts`

## Description

The discord-worker loads CJK subset fonts (Noto Sans SC, Noto Sans KR) for SVG text rendering. If fonts are loaded via fetch on each request (rather than bundled or cached), this adds latency to every SVG generation command.

## Current State

Fonts are likely bundled in the Worker (included as binary assets in the build output) given the ~8 MiB bundle size. However, if the font loading path involves async operations per request, it could introduce unnecessary latency.

## Recommendation

Verify that font data is:
1. Bundled at build time (not fetched at runtime)
2. Cached in a module-level variable (not re-parsed per request)

If fonts are fetched at runtime, consider:
- Caching in module scope (persists across requests within same isolate)
- Using `Uint8Array` rather than re-encoding on each use

## Effort

LOW — Verify current implementation; optimize only if runtime fetching is confirmed.

## Resolution

**Status:** Closed — Already Optimal

Verified both `discord-worker` and `og-worker` font loading implementations. Both follow all three best practices:

1. **Bundled at build time** — Fonts are statically imported via wrangler binary asset imports (`import spaceGroteskData from '../fonts/SpaceGrotesk-VariableFont_wght.ttf'`), not fetched at runtime
2. **Module-level cache** — A `let fontBuffersCache: Uint8Array[] | null = null` persists across requests within the same isolate
3. **Lazy initialization** — `Uint8Array` wrapping only occurs on first call, then returns cached buffers

No runtime fetch, no per-request overhead. No changes needed.
