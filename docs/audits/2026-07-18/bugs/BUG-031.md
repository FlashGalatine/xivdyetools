# [BUG-031]: og-worker OG image generators ignore validated `algo` (and 3-dye `ratio`) params — images misrepresent web-app results with a false algorithm label

## Severity

MEDIUM

## Type

Logic error / product correctness drift between OG images and the tools they advertise

## Location

- `apps/og-worker/src/services/svg/harmony.ts:58-151` — `getHarmonyMatches(dye, harmonyType, _algorithm)` never reads `_algorithm`
- `apps/og-worker/src/services/svg/gradient.ts:47-92` — plain RGB lerp regardless of `algo`
- `apps/og-worker/src/services/svg/mixer.ts:69-94` — RGB mixing; `mixThreeColors` ignores the validated `ratio` entirely (equal thirds)
- False label emitted by `apps/og-worker/src/services/svg/og-card.ts:132-142` (`Algorithm: ${algorithm.toUpperCase()}`)

## Description

The routes carefully validate `?algo=oklab|ciede2000|euclidean` (BUG-002 fix, `index.ts:231-236` et al.) and the mixer `ratio` (1-99), then pass them into the generators — where they are discarded:

- **Harmony:** matches come from a hand-rolled LAB-hue proximity scan, not core's `HarmonyGenerator` (which the web-app uses, with algorithm + color-space options). The footer still prints whatever algorithm the URL requested.
- **Gradient:** steps are interpolated in raw RGB regardless of `algo`, while the web-app interpolates per the selected algorithm.
- **Mixer (3-dye):** blends equal thirds, ignoring the `ratio` path parameter that was range-validated and baked into the shared URL.

Net effect: the social-media preview can show different dyes/blends than the shared page actually displays, with an explicit — and false — "Algorithm: CIEDE2000" style label. Beyond user-facing inconsistency, every future web-app algorithm change silently widens the drift.

## Reproduction Scenario

1. Share `https://xivdyetools.app/harmony/?dye=5771&harmony=tetradic&algo=ciede2000` in Discord.
2. The embed image (`/og/harmony/5771/tetradic.png?algo=ciede2000`) footer reads "Algorithm: CIEDE2000", but its four swatches were chosen by LAB-hue proximity with OKLAB deltas.
3. Open the link — the harmony tool computes matches with core's `HarmonyGenerator` under CIEDE2000 and shows a different set of dyes than the preview promised.

## Evidence

```ts
// harmony.ts:58-62 — parameter received and discarded
function getHarmonyMatches(
  dye: Dye,
  harmonyType: HarmonyType,
  _algorithm: MatchingAlgorithm = 'oklab'   // never read below
): Array<{ dye: Dye; delta: number }> {

// gradient.ts:59-61 — RGB lerp regardless of algo
const r = Math.round(r1 + (r2 - r1) * ratio);

// mixer.ts:89-91 — 3-dye mix ignores ratio
const r = Math.round((rgb1.r + rgb2.r + rgb3.r) / 3);
```

## Suggested Fix

- **Harmony:** replace the local hue scan with core's `DyeService.findTriadicDyes / findComplementaryPair / findTetradicDyes / ...` (`HarmonyOptions` supports matching algorithm and color space) so OG images share the web-app's engine.
- **Gradient:** thread `algo` into interpolation via `ColorService.mixColorsOklab / mixColorsLab / mixColorsRgb` per step.
- **Mixer:** implement weighted 3-dye mixing, or remove `ratio` from the 3-dye route + generated URL so the contract matches behavior.
- Minimum viable fix if the above is deferred: stop printing an algorithm name that was not used.

## Why It's Hidden

SVG snapshot tests assert layout and element presence, not that the pictured dyes equal the web-app's algorithm output; nothing diff-tests og-worker results against `HarmonyGenerator`/`ColorService`. Users rarely compare a link preview's swatches against the opened page closely enough to file a report.

## Status

**FIXED 2026-07-19** — the validated params are now actually used: harmony ranks/labels match deltas with the requested metric (oklab/ciede2000-via-CIELAB ΔE2000/euclidean) via the new shared `deltaForAlgorithm`, computed once for each winning candidate; gradient interpolates in the algorithm's space (OKLAB / CIELAB / RGB via core's mixColors*) and matches steps with the requested metric; the 3-dye mixer applies the validated `ratio` (A gets ratio%, B/C split the remainder) in the algorithm's space. The 'Algorithm:' footer now describes what actually ran. (Note: harmony target selection remains hue-scan-based like the web-app's own generator; full engine parity with core's HarmonyGenerator remains a possible follow-up.)
