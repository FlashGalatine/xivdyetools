# [BUG-066]: universalis-proxy rate limiting is per-isolate memory (effectively advisory) with a spoofable X-Forwarded-For fallback

## Severity

LOW

## Type

Security / rate-limit efficacy (documented design tradeoff, restated with concrete numbers)

## Location

- `apps/universalis-proxy/src/services/rate-limiter.ts:40` (module-scope `new MemoryRateLimiter()`)
- `apps/universalis-proxy/src/index.ts:138` (client-IP extraction with `X-Forwarded-For` fallback)

## Description

The production limit (30 req / 60 s per IP, wrangler.toml) is tracked in a per-isolate in-memory `Map`. Cloudflare spins up isolates per PoP and under load, so a single client routed across N isolates gets N × 30 req/min, and isolate recycling resets all counters. The limiter is therefore a soft brake, not an enforcement mechanism — the Cache API + request coalescer are what actually protect upstream.

Separately, the client-IP extraction falls back to `X-Forwarded-For` (attacker-controllable) when `CF-Connecting-IP` is absent. The workspace's own `@xivdyetools/rate-limiter` exports `getClientIp()` which deliberately ignores XFF for this reason (its SEC-002), so the proxy's hand-rolled extraction is strictly weaker than the shared util it already depends on. In practice Cloudflare always sets `CF-Connecting-IP` at the edge, so the XFF path is dev-only — but it is dead-weight risk that invites copy-paste into a context where it matters.

## Reproduction Scenario

1. Run sustained traffic from one IP against `proxy.xivdyetools.app` at ~100 req/min across multiple connections; observe that far more than 30 requests/min succeed (requests land on different isolates/PoPs).
2. (Dev/self-hosted only) Send requests with rotating `X-Forwarded-For: <random-ip>` headers and no CF edge in front → each request gets a fresh limit bucket.

## Evidence

```ts
// services/rate-limiter.ts:36-40
/**
 * Shared rate limiter instance
 * Persists within a Worker isolate            ← per-isolate only
 */
const limiter = new MemoryRateLimiter();
```

```ts
// index.ts:138
const clientIP = c.req.header('CF-Connecting-IP')
  || c.req.header('X-Forwarded-For')?.split(',')[0]?.trim()   // ← spoofable fallback
  || 'unknown';
```

## Suggested Fix

- Either explicitly document the per-isolate limiter as best-effort (comment + CLAUDE.md) and rely on cache/coalescing for upstream protection, or switch to `KVRateLimiter` from `@xivdyetools/rate-limiter` (the api-worker pattern) for a genuinely distributed limit.
- Replace the manual header logic with `getClientIp(c.req.raw)` from `@xivdyetools/rate-limiter` to drop the XFF fallback.

## Why It's Hidden

Unit tests exercise the limiter against a single in-process instance, where it behaves perfectly. The multi-isolate dilution only shows under real distributed load, and the XFF weakness only matters in topologies without the Cloudflare edge — neither appears in CI.
