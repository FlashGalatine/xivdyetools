# [BUG-011]: APIService — cache-write failure discards successfully fetched price data

## Severity
MEDIUM

## Type
Error-handling gap / swallowed success path

## Location
- `packages/core/src/services/APIService.ts:570` (`getPriceData` single-item path)
- `packages/core/src/services/APIService.ts:992-998` (`getPricesForItems`)
- `packages/core/src/services/APIService.ts:1051-1057` (`getPricesForDataCenter`)
- Contrast with read-side handling at `packages/core/src/services/APIService.ts:413-425` (ERROR-001)

## Description
ERROR-001 made cache **reads** failure-tolerant (`getCachedPrice` catches, logs, returns null so the request falls through to the API). Cache **writes** got no such treatment:

1. `getPriceData` awaits `setCachedPrice` inside the main try block. If the injected backend's `set()` throws (localStorage quota exceeded, KV write error — precisely the environments `ICacheBackend` exists for), the catch at `:575-584` logs "Failed to fetch price data" and resolves **null**, even though `data` was fetched successfully. The user sees "no price" because a *cache* failed.

2. Worse in the batch paths: cache writes are collected and awaited with `Promise.all`. One failed write rejects the whole call, throwing away the *entire fetched batch* — and neither `getPricesForItems` nor `getPricesForDataCenter` has any try/catch, so the error propagates uncaught to the command handler.

## Reproduction Scenario
Inject a cache backend whose `set` throws (or fill localStorage to quota in the web app), then call `getPriceData` / `getPricesForItems` for uncached items. The Universalis fetch succeeds; the method returns `null` / throws anyway.

## Evidence
```ts
// APIService.ts:569-574
if (data) {
  await this.setCachedPrice(cacheKey, data);   // ← throws → catch at :575 resolves null
}
resolvePromise!(data);
return data;
```
```ts
// APIService.ts:992-998
const cacheWrites: Promise<void>[] = [];
for (const [itemID, priceData] of batchResults) {
  ...
  cacheWrites.push(this.setCachedPrice(cacheKey, priceData));
  results.set(itemID, priceData);
}
await Promise.all(cacheWrites);   // ← one rejection loses all results + throws to caller
```

## Suggested Fix
Make writes best-effort, symmetric with ERROR-001:
```ts
private async trySetCachedPrice(cacheKey: string, data: PriceData): Promise<void> {
  try { await this.setCachedPrice(cacheKey, data); }
  catch (e) {
    this.metrics.errors++;
    this.logger.error(`Cache write failed for ${cacheKey}: ${e instanceof Error ? e.message : e}`);
  }
}
```
Use it at `:570`, and in the batch paths either push `trySetCachedPrice` calls or replace `Promise.all(cacheWrites)` with `Promise.allSettled`.

## Why It's Hidden
Test cache backends (`MemoryCacheBackend`, mocks) never fail on `set`. Quota/KV-write failures happen only in production under storage pressure, and the resulting log line misattributes the failure to fetching.

## Status

**FIXED 2026-07-19** — new best-effort `trySetCachedPrice` used on the single-item path and both batch paths; a failed cache write logs + counts an error instead of discarding fetched data or rejecting the whole batch.
