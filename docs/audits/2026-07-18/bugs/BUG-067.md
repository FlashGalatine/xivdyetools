# [BUG-067]: api-worker Facewear dyes silently never localize — `getDyeName` keyed by synthetic negative itemID

## Severity

LOW

## Type

Silent data gap / edge case with synthetic IDs

## Location

- Readers: `apps/api-worker/src/routes/dyes.ts:62, 111-113, 180-182, 216-218, 296-298`; `apps/api-worker/src/routes/match.ts:80-82, 138-140`
- Lookup: `packages/core/src/services/LocalizationService.ts:329-338` (`getDyeName(itemID)` — docstring: "Dye item ID (5729-48227)")

## Description

All api-worker handlers resolve localized names via `LocalizationService.getDyeName(dye.itemID)`. The 11 Facewear entries carry synthetic hash-based **negative** itemIDs (domain fact — assigned by `DyeDatabase.initialize()`), which are not keys in the locale dye-name maps. Every lookup returns `null`, so `localizedName` is omitted for every Facewear dye at every `?locale≠en` — no error, no fallback marker, just silently English-only entries mixed into otherwise-localized responses. If ko/zh/ja Facewear names exist in the locale data under some other key (e.g. name-based), they are unreachable through this code path.

This is not the (already-documented) domain rule about market filtering — it is a localization consequence of the same synthetic IDs that no caller accounts for.

## Reproduction Scenario

1. `GET /v1/dyes?locale=ja&category=Facewear` (or any listing that includes Facewear entries).
2. Standard dyes carry `localizedName`; all 11 Facewear entries lack the field entirely — indistinguishable from "translation missing" to an API consumer.

## Suggested Fix

- If localized Facewear names exist in the locale data: add a lookup path in core's `TranslationProvider` keyed by something stable for Facewear (internal `dye.id` or name), and route negative itemIDs through it.
- If they do not exist: document in `apps/api-docs` that `localizedName` is intentionally absent for Facewear entries (itemID < 0), so consumers don't report it as a data bug — and consider emitting `localizedName: dye.name` explicitly for Facewear to make responses uniform.

## Why It's Hidden

The field is optional (`localizedName?`), so serialization never fails; tests assert localization on standard dyes and never on Facewear entries; and consumers can't distinguish "no translation exists" from "lookup silently failed by key type".
