# [BUG-022]: KVRateLimiter "optimistic concurrency" is not implemented — version metadata never compared; verification loop can double-count

## Severity
MEDIUM

## Type
Race condition / async anti-pattern (read-modify-write without compare-and-swap)

## Location
`packages/rate-limiter/src/backends/kv.ts:194-270` (`increment()`; version read at 202-206, unconditional put at 222-225, verification read-back at 227-235)

## Description
`increment()` reads the current entry with `getWithMetadata` and captures `currentVersion`, then **unconditionally** `put`s a new entry with `metadata: { version: currentVersion + 1 }`. The version is written but never checked against anything — there is no compare-and-swap, so the "optimistic concurrency with retries (MOD-BUG-001 fix)" claimed in the class documentation does not exist. Two concurrent increments that both read `count: 5` both write `count: 6` (classic lost update → the limiter under-counts and the effective limit is higher than configured).

Worse, the post-put "verification" (`kv.get`, retry unless `verified.count >= entry.count`) can cause **over-counting**: if a concurrent writer resets the window (`count: 1`) between our put and our verification read, or KV serves a stale read, verification fails and the loop re-runs the entire read-modify-write — incrementing a *second* time for the same request.

## Reproduction Scenario
- **Under-count (lost update):** two isolates call `check(key, cfg)` simultaneously with existing count 5 → both read 5, both put 6, both verify `6 >= 6` → final count 6 instead of 7. Sustained concurrency lets clients exceed `maxRequests` by the concurrency factor.
- **Over-count (double increment):** isolate A writes `count: 6`; isolate B's fresh-window write `count: 1` lands between A's put and A's verification read → A sees `1 < 6`, retries the full loop, and writes `count: 2` — A's single request counted twice.

## Evidence
```ts
// packages/rate-limiter/src/backends/kv.ts:202-235
const result = await this.kv.getWithMetadata<KVMetadata>(kvKey);
...
const currentVersion = result.metadata?.version ?? 0;   // read…
...
await this.kv.put(kvKey, JSON.stringify(entry), {
  expirationTtl: ttl,
  metadata: { version: currentVersion + 1 },            // …written, never compared
});

// Verify write succeeded (simple optimistic check)
const verification = await this.kv.get(kvKey);
if (verification) {
  const verified: KVEntry = JSON.parse(verification) as KVEntry;
  if (verified.count >= entry.count) {                  // lost updates pass this check
    return;
  }
}
// falls through → retry → full re-read + re-put → double increment
```

## Suggested Fix
KV fundamentally cannot do atomic read-modify-write, so pick one honestly:

1. **Preferred (simple):** accept the fixed-window KV limiter as best-effort. Delete the version metadata and the verification read (they add cost and a double-count path, not correctness — see OPT-002), and document the accuracy bound (limit may be exceeded by ~concurrency factor).
2. **Strict:** move strict limits to `UpstashRateLimiter` (atomic `INCR`) or a Durable Object counter — the class docblock (kv.ts:112-113) already recommends exactly this.

Do not keep code shaped like OCC that isn't OCC; it misleads future maintainers into trusting the limit.

## Why It's Hidden
The mock KV in `packages/test-utils/src/cloudflare/kv.ts` is a synchronous in-memory Map — no cross-isolate concurrency, no eventual consistency, no stale reads. Every test sees perfectly serialized operations, so both race directions are unreproducible in the test suite and only manifest under real distributed load, where an occasional off-by-a-few counter is indistinguishable from normal traffic noise.
