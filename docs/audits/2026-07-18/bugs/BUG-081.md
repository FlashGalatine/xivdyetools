# [BUG-081]: maintainer: `stripDyePrefix` full-width-colon handling is a no-op

## Severity
LOW

## Type
Dead logic / character-encoding bug (maintainer dev tool)

## Location
- `apps/maintainer/src/services/xivapiService.ts:79-104` (dead ternary `:81-84`, bogus variant list `:91-95`)

## Description
Byte-level inspection confirms lines 81 and 94 contain **only ASCII characters** — there is no U+FF1A (full-width colon `：`) anywhere in the function, despite the comment claiming otherwise:

1. The `prefixWithColon` ternary tests `prefix.endsWith(':')` twice (same ASCII colon) and returns `prefix` in **both** branches — entirely dead logic.
2. The `prefixVariants` array contains `` `${prefix}:` `` twice; the entry commented `// full-width colon` is the same ASCII colon again.

Net effect: the intended full-width-colon variant (`カララント：`) is never tried. If the `dyePrefixes.ja` config value omits the colon, Japanese names fetched from XIVAPI keep their `カララント：` prefix and are written into the locale data un-stripped. The function currently only works when the configured prefix already contains the exact colon character used in the data.

## Reproduction Scenario
Configure `dyePrefixes.ja = 'カララント'` (no colon) and fetch an item whose Japanese name is `カララント：スートブラック`. `stripDyePrefix` returns the name unchanged (prefix with ASCII `:` appended never matches the full-width `：` in the data), and the un-stripped name propagates into `dyenames`/locale output.

## Evidence
```ts
// xivapiService.ts:81-84 — both branches return `prefix`; condition tests ASCII ':' twice
const prefixWithColon = prefix.endsWith(':') || prefix.endsWith(':')
  ? prefix
  : prefix

// xivapiService.ts:91-95 — the "full-width colon" variant is the same ASCII colon
const prefixVariants = [
  prefix,
  `${prefix}:`,
  `${prefix}:`, // full-width colon   ← comment is wrong; ASCII ':' again
]
```
(Verified with a Python codepoint scan: lines 81 and 94 contain no characters above U+007F.)

## Suggested Fix
```ts
const prefixVariants = [
  prefix,
  prefix + ':',        // ASCII colon
  prefix + '：',   // full-width colon ：
];
for (const variant of prefixVariants) {
  if (name.startsWith(variant)) {
    return name.slice(variant.length).trim();
  }
}
return name;
```
Delete the dead `prefixWithColon` ternary and its `startsWith` branch entirely.

## Why It's Hidden
ASCII `:` and U+FF1A `：` are visually near-identical in most editors, so the duplicated literal reads as intended; the maintainer app has no unit tests (type-check only), and current prefix configs happen to include the colon, masking the gap.
