# [BUG-069]: og-worker pass-through `fetch(request)` can self-fetch on the `og.` custom domain (CF error instead of 404)

## Severity

LOW (needs infra confirmation of the og domain's routing mode)

## Type

Edge case / worker self-fetch on custom domain

## Location

- `apps/og-worker/src/index.ts:171-175` (`createToolHandler` non-crawler pass-through)
- `apps/og-worker/src/index.ts:584-605` (`app.all('*')` catch-all pass-through)
- Route config: `apps/og-worker/wrangler.toml:6-13` (zone routes only for `xivdyetools.app/{tool}/*`)

## Description

For non-crawler requests, both the tool handlers and the catch-all return `fetch(request)` / `fetch(c.req.raw)`. On the zone-routed patterns (`xivdyetools.app/harmony/*`, etc.) a same-zone subrequest bypasses the worker and reaches the origin (the SPA) — correct. But the `/og/*` image endpoints are served from `og.xivdyetools.app`, which — matching the api-worker/universalis-proxy convention — is a **Workers custom domain**, meaning the worker *is* the origin there. A request with a non-crawler User-Agent to any path on `og.xivdyetools.app` that falls through to the catch-all triggers the worker fetching its own hostname; Cloudflare blocks a worker from fetching its own custom domain (error 1042), so the visitor receives a 5xx instead of a sensible 404/redirect. Every stray human or unknown-UA bot hit on the og domain becomes an error.

## Reproduction Scenario

1. In a normal browser (non-crawler UA), visit `https://og.xivdyetools.app/anything-not-an-og-route`.
2. Request reaches `app.all('*')` → `crawlerInfo.isCrawler === false` → `fetch(c.req.raw)` to `og.xivdyetools.app/...` → Cloudflare error 1042 surfaces as a 5xx page.

## Evidence

```ts
// index.ts:584-605
app.all('*', (c) => {
  const crawlerInfo = detectCrawlerFromRequest(c.req.raw);
  if (crawlerInfo.isCrawler) { ...return OG HTML... }
  // Pass through to origin for regular users
  return fetch(c.req.raw);          // ← on a custom domain, "origin" is this worker itself
});
```

```toml
# wrangler.toml — only tool paths are zone-routed; /og/* lives on the og. custom domain
routes = [
  { pattern = "xivdyetools.app/harmony/*", zone_name = "xivdyetools.app" },
  ...
]
```

## Suggested Fix

Guard the pass-through against the worker's own image hostname:

```ts
app.all('*', (c) => {
  const url = new URL(c.req.url);
  const ogHost = new URL(c.env.OG_IMAGE_BASE_URL).hostname;
  if (url.hostname === ogHost) {
    return c.json({ error: 'Not found' }, 404);          // or redirect to APP_BASE_URL
  }
  ...existing crawler / fetch(request) logic...
});
```

Apply the same guard to `createToolHandler`'s pass-through for safety (tool routes shouldn't be reachable on the og host, but the guard is free).

## Why It's Hidden

Crawler-focused tests always send crawler UAs; humans virtually never type the og subdomain by hand; and monitoring shows a trickle of 5xx that is easy to dismiss as scanner noise. The error only occurs on a path/UA combination that no test or normal user flow produces.

## Status

**FIXED 2026-07-19** — `isOgImageHost()` guards both the tool-handler pass-through (302 → APP_BASE_URL) and the catch-all (404 JSON) so the worker never `fetch()`es its own og. custom domain (CF error 1042 → 5xx for stray human/unknown-UA hits).
