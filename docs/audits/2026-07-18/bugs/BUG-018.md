# [BUG-018]: OAuth redirect-URI allowlists disagree between authorize and callback — transition-domain Discord login is broken

## Severity
MEDIUM

## Type
Logic Error / Inconsistent Validation (auth flow)

## Location
- File: `apps/oauth/src/handlers/authorize.ts:69-88` (authorize allowlist)
- File: `apps/oauth/src/handlers/callback.ts:83-109` (Discord GET callback allowlist)
- File: `apps/oauth/src/handlers/xivauth.ts:206-226` (XIVAuth GET callback allowlist)
- File: `apps/oauth/src/constants/oauth.ts:10-17` (`ALLOWED_REDIRECT_ORIGINS`)
- File: `apps/oauth/wrangler.toml:8, 19` (live transition route + `FRONTEND_URL`)

## Description
Three different allowlists validate the same `redirect_uri` value at different steps of the flow:
- `/auth/discord` (authorize.ts:69-73) accepts `ALLOWED_REDIRECT_ORIGINS` (includes `https://xivdyetools.projectgalatine.com` and localhost entries) plus `FRONTEND_URL`.
- Discord `GET /auth/callback` (callback.ts:83-104) accepts **only** `new URL(FRONTEND_URL).origin` (= `https://xivdyetools.app` in prod), plus localhost in dev.
- XIVAuth `GET /auth/xivauth/callback` (xivauth.ts:209-219) accepts `ALLOWED_REDIRECT_ORIGINS` (minus localhost in prod) but **not** `FRONTEND_URL` (works today only because prod FRONTEND_URL happens to be in the constant).

Net effect: a user on the still-routed transition domain (`auth.xivdyetools.projectgalatine.com` is a live route) who starts Discord login with `redirect_uri=https://xivdyetools.projectgalatine.com/auth/callback` passes validation at `/auth/discord`, authenticates at Discord, then is bounced at `GET /auth/callback` with "Untrusted redirect origin" — every time. Additionally, authorize accepts `http://localhost:5173` redirect URIs **in production** (the constant is not environment-filtered at authorize time); it is only caught later at the callback — harmless for security but inconsistent.

## Reproduction Scenario
1. From `https://xivdyetools.projectgalatine.com`, initiate Discord login (redirect_uri = own origin's `/auth/callback`).
2. `/auth/discord` validates it against `ALLOWED_REDIRECT_ORIGINS` → accepted, state signed, user sent to Discord.
3. Complete Discord consent → `GET /auth/callback` checks only the `FRONTEND_URL` origin → redirected to `https://xivdyetools.app/auth/callback?error=Untrusted+redirect+origin`.

## Evidence
```ts
// callback.ts:83-85 — Discord callback allowlist
const allowedOrigins = [ new URL(c.env.FRONTEND_URL).origin ];
```
```ts
// authorize.ts:69-73 — authorize allowlist
const allowedOrigins = [
  ...ALLOWED_REDIRECT_ORIGINS,
  c.env.FRONTEND_URL,
  `${c.env.FRONTEND_URL}/auth/callback`,
];
```

## Suggested Fix
Extract one helper and use it in all four places (both authorize handlers and both GET callbacks):
```ts
export function getAllowedRedirectOrigins(env: Env): string[] {
  const origins = [...ALLOWED_REDIRECT_ORIGINS, env.FRONTEND_URL];
  return env.ENVIRONMENT === 'development'
    ? origins
    : origins.filter((o) => !o.includes('localhost') && !o.includes('127.0.0.1'));
}
```

## Why It's Hidden
Callback tests use `FRONTEND_URL`-origin redirect URIs, so all suites pass. The transition domain exists only in constants and wrangler routes and is exercised by no test; failures surface as user-facing error redirects on a secondary domain, easily misattributed to Discord.

## Status

**FIXED 2026-07-19** — one `getAllowedRedirectOrigins(env)` helper (constants/oauth.ts; localhost filtered outside development) is used by both authorize handlers and both GET callbacks via the shared oauth-flow pipeline; the transition-domain flow now validates consistently at every step.
