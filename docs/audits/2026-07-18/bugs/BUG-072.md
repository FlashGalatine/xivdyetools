# [BUG-072]: /preset submit moderation-channel embeds skip the sanitization that the webhook path applies

## Severity
LOW

## Type
Security consistency gap / content injection into moderation embeds

## Location
`apps/discord-worker/src/handlers/commands/preset.ts:963-977` (`notifyModerationChannel`) and `:1024-1036` (`notifyEditModerationChannel` diff) — vs. `apps/discord-worker/src/index.ts:186-190` (webhook path, which sanitizes)

## Description
The webhook path deliberately sanitizes user content before embedding it in the moderation channel (`index.ts:187-189`, with a SECURITY comment): `sanitizePresetName` / `sanitizePresetDescription`. The Discord-submission path builds the same moderation embed from **raw** values — `grep sanitize` over `preset.ts` returns nothing, and presets-api does not sanitize on ingest either.

Control characters, zalgo, and markdown injection (e.g. closing bold and inserting fake `**Author:**` labels to spoof the attribution line) flow straight into the moderation channel — exactly the audience you least want spoofed. (Embeds never ping, so `@everyone` in a name is not an escalation.)

## Reproduction Scenario
1. `/preset submit` with `name` containing markdown / zalgo / a fake `**Author:** someone-else` line and a description that closes formatting early.
2. The preset lands as pending; `notifyModerationChannel` posts the embed with the raw strings.
3. The moderation-channel embed renders the injected structure; the same preset submitted via the web app would have been sanitized.

## Evidence
```ts
// apps/discord-worker/src/handlers/commands/preset.ts:967-973 — raw user content
description: [
  `**Name:** ${preset.name}`,
  `**Description:** ${preset.description}`,
  `**Author:** ${preset.author_name} (<@${preset.author_discord_id}>)`,
  ...
].join('\n'),
```
```ts
// apps/discord-worker/src/index.ts:187-189 — the webhook path does it right
// SECURITY: Sanitize user-provided content before display
const safeName = sanitizePresetName(preset.name);
const safeDescription = sanitizePresetDescription(preset.description);
```

## Suggested Fix
Apply `sanitizePresetName` / `sanitizePresetDescription` (already imported in `index.ts` from `../utils/sanitize.js`) inside `notifyModerationChannel`, `notifyEditModerationChannel`, and `notifySubmissionChannel` before interpolation — or extract one shared embed builder (see REFACTOR-025) so sanitization happens in exactly one place.

## Why It's Hidden
Two parallel implementations of the same notification exist; the hardened one (webhook) is the one that got the security review comment, and the Discord-path duplicate silently diverged. Tests assert embed structure with benign fixture names, never adversarial input.

## Status

**FIXED 2026-07-19** — sanitization (`sanitizePresetName`/`sanitizePresetDescription`) is applied unconditionally inside the shared moderation-embed builder used by all three paths.
