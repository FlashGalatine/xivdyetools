# [BUG-026]: Browser preset sends raw, unredacted context and error to the error tracker, bypassing the redaction pipeline

## Severity
MEDIUM

## Type
Security / redaction bypass (third-party exfiltration path)

## Location
`packages/logger/src/presets/browser.ts:106-127` (`createBrowserLogger` error/warn wrappers)

## Description
`createBrowserLogger` monkey-patches `logger.error` and `logger.warn` so that, in production, events are also forwarded to a configured `errorTracker` (the documented use case is Sentry, browser.ts:72-80). The console path goes through `BaseLogger.createEntry → mergeContext → redactSensitiveFields` and `sanitizeErrorMessage` — but the tracker path passes the **caller's raw `context` object** and the **raw `Error`** (unsanitized message + full stack) straight to `errorTracker.captureException(error, context)`. None of the package's redaction machinery runs for the one path where data actually leaves the user's origin to a third-party service.

## Reproduction Scenario
1. Web app configures `createBrowserLogger({ errorTracker: sentryAdapter })` and runs a production build (`isDevMode === false`, so the wrappers are installed).
2. Code logs `logger.error('auth refresh failed', err, { token: userJwt })`.
3. Console output shows `token: '[REDACTED]'` (redacted pipeline).
4. The Sentry event contains the real JWT in `extra`, plus `err.message` unsanitized — persisted on a third-party server, visible to anyone with Sentry project access.

## Evidence
```ts
// packages/logger/src/presets/browser.ts:106-120
if (errorTracker && !isDevMode) {
  const originalError = logger.error.bind(logger);
  logger.error = (message: string, error?: unknown, context?: LogContext): void => {
    // Still log to console
    originalError(message, error, context);       // console path: redacted

    // Send to error tracker
    if (error instanceof Error) {
      errorTracker.captureException(error, context);   // RAW context + raw error
    } else if (error) {
      errorTracker.captureMessage(`${message}: ...`, 'error');
    ...
```

## Suggested Fix
Redact before forwarding. Expose the existing machinery on `BaseLogger` (e.g. make `redactSensitiveFields`/`sanitizeErrorMessage` callable via a public `redact(context)` helper, or have the wrapper build a `LogEntry` via `createEntry` and forward its already-redacted fields):

```ts
logger.error = (message, error?, context?) => {
  originalError(message, error, context);
  const safeContext = context ? logger.redact(context) : undefined;
  if (error instanceof Error) {
    const safeError = new Error(logger.sanitizeMessage(error.message));
    safeError.name = error.name;
    safeError.stack = error.stack; // stack is desirable for trackers; keep deliberate
    errorTracker.captureException(safeError, safeContext);
  } else { ... }
};
```
At minimum, run `redactSensitiveFields` on `context` — that closes the highest-value leak (structured secrets) with a two-line change.

## Why It's Hidden
Logger tests assert on console/JSON output, which *is* redacted — so the suite is green. The tracker is a consumer-supplied callback that tests stub without inspecting payload secrecy. In production the leak is invisible from the app side: the console looks clean, and the unredacted data only exists inside the third-party dashboard, which nobody diff-checks against the console output.

## Status

**FIXED 2026-07-19** — the browser preset's errorTracker path now redacts context via `redactContext` and sanitizes error/warn messages via `sanitizeMessage` before forwarding; the tracker receives a sanitized Error clone (name/stack preserved deliberately). Regression test added asserting no secret reaches the tracker payload.
