# [BUG-059]: Discord request body limit measured in UTF-16 code units, not bytes

## Severity
LOW

## Type
Boundary condition / unit mismatch (size-limit under-enforcement)

## Location
`packages/auth/src/discord.ts:88-97` (body read + size check; Content-Length pre-check at 66-73)

## Description
`verifyDiscordRequest` enforces `maxBodySize` (default 100,000 — documented as "100KB", discord.ts:33-34) in two places. The first checks the spoofable `Content-Length` header. The authoritative second check, after `await request.text()`, compares `body.length > maxBodySize` — but `String.length` counts **UTF-16 code units**, not bytes. CJK characters are 3 bytes each in UTF-8 but 1 code unit; astral characters (emoji) are 4 bytes but 2 units. A payload of 100,000 CJK characters (~300 KB on the wire) passes the "100KB" check; the practical byte cap is up to ~4× the intended limit.

Secondary observation: the entire body is buffered by `request.text()` *before* the check runs, so the limit is a post-hoc filter, not memory protection — the Workers runtime's own request-size limits are the real backstop. The check's purpose (bounding what gets passed to Ed25519 verification and JSON.parse) still holds, just at the wrong threshold.

## Reproduction Scenario
1. Send a POST to the discord-worker interactions endpoint with a body of 100,000 CJK characters (≈293 KB UTF-8) and an honest `Content-Length` — wait, honest Content-Length (293k) is caught by the header pre-check; so send `Content-Length: 90000` (spoofed) or chunked encoding.
2. `body.length === 100_000` → not `> 100_000` → passes the byte-limit intended as 100 KB with a ~293 KB payload, which then goes through signature verification and (if signed) `JSON.parse`.

## Evidence
```ts
// packages/auth/src/discord.ts:87-97
// Get the raw body
const body = await request.text();

// Verify actual body size (Content-Length can be spoofed)
if (body.length > maxBodySize) {        // UTF-16 code units, not bytes
  return {
    isValid: false,
    body: '',
    error: 'Request body too large',
  };
}
```

## Suggested Fix
Compare bytes:

```ts
const bodyBytes = new TextEncoder().encode(body);
if (bodyBytes.byteLength > maxBodySize) {
  return { isValid: false, body: '', error: 'Request body too large' };
}
```
(The encoded bytes can be reused for `verifyKey` if the underlying library accepts a `Uint8Array`, avoiding double encoding.) For true streaming enforcement, read `request.body` through a counting reader and abort past the limit before buffering — optional given Workers' platform caps.

## Why It's Hidden
Tests exercise ASCII payloads, where code units equal bytes and the check behaves exactly as documented. The divergence requires multi-byte content at boundary sizes, which nothing legitimate sends; and since the consequence is merely a permissive limit (not a rejection), there is no failure signal — only a slightly larger attack surface that never shows up in functional testing.

## Status

**FIXED 2026-07-19** — the authoritative post-read size check measures UTF-8 bytes (`TextEncoder`) instead of UTF-16 code units, closing the ~4x permissive gap for CJK/emoji payloads.
