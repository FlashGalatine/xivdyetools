# [BUG-039]: Server change does not invalidate in-flight price fetch (stale-cache repopulation race)

## Severity
MEDIUM

## Type
Race condition (async response vs. config change)

## Location
- `apps/web-app/src/services/market-board-service.ts:165-192` (config subscription)
- `apps/web-app/src/services/market-board-service.ts:304-390` (`fetchPricesForDyes`)

## Description
The service's request-versioning only bumps `requestVersion` when a **new fetch starts** (`:310-311`). The server-change subscription clears the price cache but does **not** bump the version. If a fetch for the old server is in flight when the user switches servers, and no tool immediately issues a new fetch (e.g. the active tool has no results displayed, or its re-fetch is gated behind `showPrices`/`matchedDyes` checks like `swatch-tool.ts:470-473`), the in-flight response still passes the `requestVersion === this.requestVersion` check at `:348` and writes **old-server prices** into the just-cleared cache (`:356-361`), then emits `prices-updated`. Consumers render those prices labeled with the new server name (`getWorldNameForPrice` falls back to `this.selectedServer`, `:267-272`).

Secondary issue in the same function: the `total === 0` early return (`:317-320`) and the stale-response returns (`:348-353`, error path `:379`) never reset `this.isFetching`, so `getIsFetching()` (`:215-217`) can be stuck `true` forever after a superseding call had nothing to fetch. No component currently consumes `getIsFetching()`, so this part is latent.

## Reproduction Scenario
Throttle the network; open Comparison with 4 dyes and prices on (fetch starts); immediately switch server in the ConfigSidebar. If the old-server fetch resolves after the config event but before (or without) a new versioned fetch, old-server prices land in the cache attributed to the new server.

## Evidence
```ts
// market-board-service.ts:169-181
if (serverChanged) {
  const previousServer = this.selectedServer;
  this.selectedServer = config.selectedServer;

  // Clear prices on server change (prices are server-specific)
  this.priceData.clear();                       // ← cache cleared…

  this.emitEvent('server-changed', {            // ← …but requestVersion NOT incremented
    server: this.selectedServer,
    previousServer,
  });
}

// market-board-service.ts:347-353 — guard only helps if a NEW fetch bumped the version
if (requestVersion !== this.requestVersion) {
  logger.info(`[MarketBoardService] Discarding stale price response ...`);
  return new Map();
}
```

## Suggested Fix
```ts
if (serverChanged) {
  const previousServer = this.selectedServer;
  this.selectedServer = config.selectedServer;
  this.requestVersion++;        // invalidate any in-flight fetch for the old server
  this.isFetching = false;
  this.priceData.clear();
  ...
}
```
Additionally move the `isFetching = false` reset into a `finally` that applies when `requestVersion === this.requestVersion`.

## Why It's Hidden
Requires a slow in-flight request crossing a config change; unit tests resolve mocks instantly and E2E rarely switches servers mid-fetch. The stale prices look like real numbers — nothing errors.

> Source: evidence/web-frontends-analysis.md (2026-07-18 deep-dive, web-frontends area)

## Status

**FIXED 2026-07-19** — the server-change subscription bumps `requestVersion` and resets `isFetching` before clearing the cache, so an in-flight old-server response is discarded; the `total === 0` early return also resets `isFetching`.
