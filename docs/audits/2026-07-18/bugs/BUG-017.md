# [BUG-017]: Env-validation "fail fast" only fails the first request per isolate — later requests run misconfigured (systemic, 3 workers)

## Severity
MEDIUM

## Type
State Management / Logic Error (module-level flag set before failure path)

## Location
Systemic — the same middleware pattern is copy-pasted across three workers:
- `apps/presets-api/src/index.ts:34, 50-68`
- `apps/oauth/src/index.ts:26, 41-59`
- `apps/discord-worker/src/index.ts:94` (same `envValidated` flag pattern, found independently)
- Function: env-validation middleware (`app.use('*', ...)`) in each worker's entry point

## Description
The module-level `envValidated` flag is set to `true` *before* the invalid-config branch returns 500. On the next request the entire validation block is skipped, so in production a misconfigured worker serves exactly one 500 and then processes all subsequent traffic in that isolate with the invalid environment (e.g. short `JWT_SECRET`, missing `DISCORD_CLIENT_SECRET`, absent bindings) — the opposite of the stated "fail fast on misconfiguration" intent. Because the flag is per-isolate, the fleet degrades into "one 500 per isolate spin-up, then silent misconfigured operation everywhere."

## Reproduction Scenario
1. Deploy any of the three workers with an invalid config (e.g. `JWT_SECRET` of 8 chars — fails the 32-char check).
2. Request 1 to a fresh isolate → 500 "Service misconfigured".
3. Requests 2..N to the same isolate → processed normally; e.g. oauth signs JWTs with the weak secret.

## Evidence
```ts
// apps/oauth/src/index.ts:43-49 (identical shape in presets-api index.ts:50-68 and discord-worker index.ts:94)
if (!envValidated) {
  const result = validateEnv(c.env);
  envValidated = true;              // <-- set before failing
  if (!result.valid) {
    logValidationErrors(result.errors);
    if (c.env.ENVIRONMENT === 'production') {
      return c.json({ error: 'Service misconfigured' }, 500);
    }
    ...
```

## Suggested Fix
Cache the *result*, not just "ran once", so every request in a bad isolate fails (apply in all three workers):
```ts
let envResult: EnvValidationResult | null = null;

app.use('*', async (c, next) => {
  if (!envResult) {
    envResult = validateEnv(c.env);
    if (!envResult.valid) logValidationErrors(envResult.errors);
  }
  if (!envResult.valid && c.env.ENVIRONMENT === 'production') {
    return c.json({ error: 'Service misconfigured' }, 500);
  }
  return next();
});
```

## Why It's Hidden
Unit tests hit the middleware once per fresh module import, so the second-request-in-same-isolate path is never exercised. In production the single 500 per isolate is indistinguishable from a transient blip in logs, while the ongoing misconfigured operation looks healthy. Being copy-pasted into three entry points, the bug also survives fixes applied to only one worker.
