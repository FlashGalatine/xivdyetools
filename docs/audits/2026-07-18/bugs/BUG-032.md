# [BUG-032]: /budget find accepts a Facewear entry as target dye, sending a negative itemID to the Universalis proxy

## Severity
MEDIUM

## Type
Edge case / input validation gap

## Location
- `apps/discord-worker/src/handlers/commands/budget.ts:121-127` (target resolution — no Facewear guard)
- `apps/discord-worker/src/services/budget/budget-calculator.ts:108-113` (target market ID seeded into fetch set) and `:244-248` (`getDyeByName` searches *all* dyes)
- `packages/core/src/config/consolidated-ids.ts:113` (`getMarketItemID` passes negative IDs through)

## Description
The 2026-02 fix filtered *candidate* dyes with `itemID > 0` (`budget-calculator.ts:96`), but the **target** dye is resolved without any Facewear exclusion. `getDyeByName` matches against `dyeService.getAllDyes()` unfiltered, `getMarketItemID` returns negative synthetic Facewear IDs unchanged (`if (dye.itemID < 0) return dye.itemID;`), and `findCheaperAlternatives` seeds the price-fetch set with the target's market ID. `fetchPrices` then builds a path like `/api/v2/aggregated/{world}/-1127,5729,...` (`universalis-client.ts:213-214`). The universalis-proxy's item-ID validation (`^[\d,]+$`, per the 2026-02 incident record) rejects the **whole batch**, so the entire command fails with a generic API error — not just the one bad item.

Autocomplete cannot produce this input (it filters `itemID > 0` and emits numeric IDs), but `target_dye` is a free-text option: a user can type a Facewear color name (or a negative number) directly.

## Reproduction Scenario
1. Run `/budget find target_dye:` and manually type any Facewear color name (bypassing autocomplete).
2. The command defers ("thinking…"), then edits in the generic `budget.errors.apiError` message.

## Evidence
```ts
// apps/discord-worker/src/handlers/commands/budget.ts:121-124 — no itemID sign check
const targetDyeId = parseInt(targetDyeInput, 10);
const targetDye = !isNaN(targetDyeId)
  ? getDyeById(targetDyeId)
  : getDyeByName(targetDyeInput);
```
```ts
// apps/discord-worker/src/services/budget/budget-calculator.ts:108-109
const targetMarketId = getMarketItemID(targetDye);
const marketIdSet = new Set<number>([targetMarketId]);
```
```ts
// packages/core/src/config/consolidated-ids.ts:113
if (dye.itemID < 0) return dye.itemID; // Facewear
```

## Suggested Fix
In `handleFindSubcommand`, right after resolution:
```ts
if (!targetDye || targetDye.itemID < 0) {
  return ephemeralResponse(t.t('budget.errors.dyeNotFound', { name: targetDyeInput }));
}
```
Also exclude Facewear inside `getDyeByName` (budget-calculator.ts:244-248), and consider requiring a full-string numeric match before treating input as an ID (`parseInt("255 Brown")` currently parses as `255`).

## Why It's Hidden
All test fixtures and autocomplete-driven manual testing use dyes that came through the filtered autocomplete path. Free-text entry of a Facewear name is an input path with no coverage, and the 2026-02 fixes created a reasonable (but wrong) assumption that negative-ID filtering was complete.

> Source: evidence/bot-workers-analysis.md (2026-07-18 deep-dive, bot-workers area)

## Status

**FIXED 2026-07-18** — `/budget find` target resolution now rejects Facewear entries: the handler guards `targetDye.itemID <= 0` (returning the standard dyeNotFound message), `getDyeByName` in budget-calculator only matches market dyes (`itemID > 0`), and ID-vs-name parsing requires a fully numeric string (`parseInt("255 Brown")` no longer resolves as ID 255; negative numeric input falls through to the name path and is rejected). Regression test added.
