# [BUG-006]: Cross-request locale race in `LocalizationService` mutable singleton — wrong-language responses, cacheable for 24h via api-worker

## Severity

HIGH

## Type

Race condition / shared mutable module state across concurrent requests (merged finding: core root cause + api-worker amplification)

## Location

- **Root cause (core):** `packages/core/src/services/LocalizationService.ts:170-229` — `private currentLocale` on the eagerly-created `defaultInstance` singleton (line 175); `setLocale()` mutates it (lines 207-221); all static getters (`getDyeName` 329-338, `getLabel`, `getCategory`, ...) read it.
- **Core-side consumers of the singleton state:** `packages/core/src/services/DyeService.ts:364-386` (`searchByLocalizedName` reads `LocalizationService.isLocaleLoaded()` / `LocalizationService.getDyeName()`), plus `getLocalizedDyeById` (402-414), `getLocalizedDyeByStainId` (425-437), `getAllLocalizedDyes` (454-465).
- **Worst amplification (api-worker):** `apps/api-worker/src/middleware/locale.ts:20-25` (`await LocalizationService.setLocale(locale)` per request); readers in `apps/api-worker/src/routes/dyes.ts:56-64, 111-114, 180-182, 216-218, 295-300` and `apps/api-worker/src/routes/match.ts:80-82, 136-141`.

## Description

The static `LocalizationService` API delegates to a module-scope singleton whose `currentLocale` field is plain mutable state. Cloudflare Worker isolates serve many requests **concurrently**, and every `await` is an interleaving point. api-worker's `localeMiddleware` sets the singleton locale per request, then handlers later read `LocalizationService.getDyeName(...)` (and `dyeService.searchByLocalizedName(...)`, which reads the same singleton internally). Between a request's `setLocale` and its reads, another concurrent request with a different `?locale=` can complete its own `setLocale`, so the first request serializes dye names in the **wrong language** while `meta.locale` still reports the requested one.

Amplification: every affected api-worker handler emits `Cache-Control: public, max-age=3600, s-maxage=86400` (e.g. `routes/dyes.ts:67`, `routes/match.ts:84`). A wrong-language body can therefore be cached up to 24 hours by shared caches (1 hour by browsers) **under the correct-locale URL**, converting a transient race into persistent wrong-language responses.

The root cause lives in core, so any consumer using the static/singleton API in a concurrent runtime is exposed; og-worker already avoided it via the stateless `LocaleLoader`/`LocaleRegistry`/`TranslationProvider` trio (REFACTOR-001, 2026-04-28), which is the proven in-repo fix pattern.

## Reproduction Scenario

Steady state (all 6 locales already registered — `setLocale` takes the cached synchronous path):

1. Request A: `GET /v1/dyes/5729?locale=ja` — middleware runs `await LocalizationService.setLocale('ja')`; the promise resolves and A's continuation is queued as a microtask.
2. Request B (same isolate): `GET /v1/dyes/5729?locale=de` — its `setLocale('de')` synchronously sets `currentLocale = 'de'` before A's continuation runs.
3. A's handler runs → `LocalizationService.getDyeName(5729)` reads `currentLocale === 'de'` → A's response contains the German name with `meta.locale: "ja"`.
4. Any shared cache stores the German body under the `?locale=ja` URL for up to 24h.

## Suggested Fix

Adopt og-worker's stateless pattern (already exported by core for exactly this purpose):

```ts
// api-worker: lib/translator.ts (module scope, per-isolate, immutable after init)
import { LocaleLoader, LocaleRegistry, TranslationProvider } from '@xivdyetools/core';
export const apiTranslator: TranslationProvider = (() => {
  const loader = new LocaleLoader();
  const registry = new LocaleRegistry();
  for (const lc of ['en', 'ja', 'de', 'fr', 'ko', 'zh'] as const) {
    registry.registerLocale(loader.loadLocale(lc));
  }
  return new TranslationProvider(registry);
})();
```

- Handlers call `apiTranslator.getDyeName(dye.itemID, locale)` with the request's locale (from `c.get('locale')`).
- Replace `dyeService.searchByLocalizedName(q)` with an explicit-locale search (add a `locale` parameter to a core helper, or filter locally against `apiTranslator.getDyeName(itemID, locale)`).
- Delete the `setLocale` call from `localeMiddleware` (keep `c.set('locale', ...)`).
- Core-side hardening: document that the static singleton API is unsafe for concurrent multi-locale servers, and/or make `setLocale`-style APIs accept an explicit locale on read (`getDyeName(itemID, locale?)`).

## Why It's Hidden

- Unit/integration tests issue requests sequentially — the race needs two in-flight requests with **different** locales interleaved inside one isolate.
- Load tests typically hammer a single locale, which never trips it.
- In production it surfaces as rare, unreproducible "the API returned Japanese names on my German request" reports; the 24h `s-maxage` then persists the corrupted body under the correct URL, making the cause look like bad cached data rather than a race.

## Status

**FIXED 2026-07-19** — core: new `ensureLocaleLoaded()` registers locale data without mutating the singleton's current locale; every getter (and DyeService's localized methods) accepts an explicit `locale` param; `setLocale` documents the concurrency hazard. api-worker: middleware uses `ensureLocaleLoaded`, and every handler passes `c.get('locale')` explicitly via the new `localizedNameFor` helper — no read of shared locale state remains on the request path.
