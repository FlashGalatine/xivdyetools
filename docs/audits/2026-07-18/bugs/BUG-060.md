# [BUG-060]: `truncateText` can split surrogate pairs, emitting a broken character before the ellipsis

## Severity
LOW

## Type
Boundary condition / Unicode handling (UTF-16 slicing)

## Location
`packages/svg/src/base.ts:309-312` (`truncateText`); callers: `packages/svg/src/preset-swatch.ts:152, 216`, `packages/svg/src/gradient.ts:104`, `packages/svg/src/contrast-matrix.ts:197, 228`, `packages/svg/src/random-dyes-grid.ts:195`

## Description
`truncateText` slices with `text.slice(0, maxLength - 1)`, which operates on UTF-16 code units. If position `maxLength - 1` lands inside a surrogate pair — emoji (all astral plane) or rare CJK Extension-B+ ideographs — the output ends with a lone high surrogate. When rendered, that becomes U+FFFD (�) or a missing-glyph box in the generated PNG, immediately before the ellipsis.

The most realistic inputs are community **preset names and descriptions** (user-authored free text, frequently containing emoji), truncated at `preset-swatch.ts:152` (descriptions, cap 60) and `:216` (dye names). Localized dye names run through the same helper in the other generators.

Note the helper is also inconsistent with its sibling `estimateTextWidth` (base.ts:322-334), which correctly iterates code points via `for (const char of text)` — so the two functions disagree about what "a character" is.

## Reproduction Scenario
```ts
truncateText('Cool Preset 🎨🎨🎨', 14)
// '🎨' = "🎨"; slice(0, 13) cuts between \uD83C and \uDFA8
// → 'Cool Preset 🎨\uD83C' + '…'  → lone high surrogate renders as � in the PNG
```
Any `/preset show` where the description's truncation point bisects an emoji reproduces it in the posted image.

## Evidence
```ts
// packages/svg/src/base.ts:309-312
export function truncateText(text: string, maxLength: number): string {
  if (text.length <= maxLength) return text;
  return text.slice(0, maxLength - 1) + '…';   // UTF-16 unit slice — can bisect a pair
}
```

## Suggested Fix
Slice by code points:

```ts
export function truncateText(text: string, maxLength: number): string {
  const chars = [...text];                      // code-point iteration
  if (chars.length <= maxLength) return text;
  return chars.slice(0, maxLength - 1).join('') + '…';
}
```
This also aligns the length semantics with `estimateTextWidth`. (Grapheme clusters — e.g. ZWJ emoji sequences — could still split visually; acceptable for this use, or use `Intl.Segmenter` if it matters.)

## Why It's Hidden
Test fixtures use ASCII and BMP CJK names, where code units equal code points and slicing is always safe. The bug needs three coincidences: astral characters in the input, a string long enough to truncate, and the cut landing exactly on the pair boundary. Even then it's a cosmetic artifact inside a rendered image — users see a stray � and shrug; no exception, log, or test signal is ever produced.

## Status

**FIXED 2026-07-19** — `truncateText` slices by code points (`[...text]`), so truncation can no longer bisect a surrogate pair and emit U+FFFD before the ellipsis; length semantics now match `estimateTextWidth`.
