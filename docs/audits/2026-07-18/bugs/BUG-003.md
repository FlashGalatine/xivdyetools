# [BUG-003]: A rejected preset permanently poisons its dye combination — resubmission and edits 500

## Severity
HIGH

## Type
Edge Case / Schema-Code Contract Mismatch (UNIQUE index vs status-filtered checks)

## Location
- File: `apps/presets-api/schema.sql:81` and `apps/presets-api/migrations/0004_unique_dye_signature.sql:22` (full UNIQUE index)
- File: `apps/presets-api/src/services/preset-service.ts:241-247, 368-374` (status-filtered duplicate checks), 430-436 (unguarded UPDATE)
- File: `apps/presets-api/src/handlers/presets.ts:458-476` (constraint recovery that cannot recover), 261 (edit path)
- Function: `findDuplicatePreset`, `findDuplicatePresetExcluding`, `createPreset`, `updatePreset`

## Description
The UNIQUE index on `presets(dye_signature)` covers **all** rows regardless of status, but both duplicate checks filter `status IN ('approved', 'pending')`. When a preset is `rejected` (or `hidden`), its row keeps its `dye_signature`. Consequences:

1. `POST /presets` with the same dye combo: `findDuplicatePreset` returns null → `createPreset` throws a UNIQUE constraint error → the catch block calls `findDuplicatePreset` again (still null, rejected rows excluded) → falls through to `throw error` → generic 500 for the user, forever, for that combination.
2. `PATCH /presets/:id` changing `dyes` to a combo held by a rejected preset: `findDuplicatePresetExcluding` passes → `updatePreset`'s UPDATE violates the UNIQUE index with **no** try/catch → raw 500.

## Reproduction Scenario
1. Submit a preset with dyes `[1, 2]`; a moderator rejects it.
2. Any user submits a new preset with dyes `[1, 2]` → 500 `INTERNAL_ERROR`.
3. Repeats indefinitely — the dye combination is permanently unusable.

## Evidence
```sql

```
```ts

WHERE dye_signature = ? AND status IN ('approved', 'pending')
```
```ts
// presets.ts:461-475 — recovery looks up with the same status filter, finds nothing, rethrows
if (errorMessage.includes('UNIQUE constraint failed') && errorMessage.includes('dye_signature')) {
  const existingPreset = await findDuplicatePreset(c.env.DB, body.dyes); // excludes rejected → null
  ...
}
throw error; // → 500
```

## Suggested Fix
Make the index match the check (SQLite/D1 supports partial unique indexes):
```sql
DROP INDEX idx_presets_dye_signature;
CREATE UNIQUE INDEX idx_presets_dye_signature
  ON presets(dye_signature)
  WHERE status IN ('approved', 'pending');
```
Alternatively, NULL out `dye_signature` when a preset is rejected/hidden and restore it on approve. Also wrap the `updatePreset` dye-change path in the same constraint-recovery logic as create.

## Why It's Hidden
Tests cover the approved-duplicate race (constraint → converted to a vote). No test creates a *rejected* preset and then resubmits its signature; D1 test mocks may not even enforce the unique index. The bug only surfaces in production after a real rejection followed by a resubmission.
