# [BUG-065]: universalis-proxy 5 MB upstream response cap bypassed when `Content-Length` is absent (chunked responses)

## Severity

LOW

## Type

Resource exhaustion guard gap (incomplete PROXY-HIGH-002 mitigation)

## Location

`apps/universalis-proxy/src/services/cached-fetch.ts:119-138` (`fetchFromUpstream`); unguarded consumption at `:100` (`response.json()`)

## Description

`fetchFromUpstream` enforces `MAX_RESPONSE_SIZE_BYTES` (5 MB) by inspecting the `Content-Length` header and throwing `ResponseTooLargeError` when it exceeds the cap. Upstream responses delivered with `Transfer-Encoding: chunked` (no `Content-Length`) skip the check entirely, after which `response.json()` buffers an arbitrarily large body into isolate memory — the exact OOM class PROXY-HIGH-002 was written to prevent. Mitigating factors keep this LOW: the route caps requests at 100 item IDs and appends `?listings=5&entries=5` (`index.ts:204`), so realistic Universalis payloads stay small.

Secondary note: `ResponseTooLargeError` is not an `UpstreamError`, so when it does fire it falls into the generic 502 catch branch (`index.ts:240-250`) with an unhelpful message.

## Reproduction Scenario

1. Upstream (or any future `UNIVERSALIS_API_BASE` target) returns a large chunked JSON body without `Content-Length`.
2. `fetchFromUpstream` returns the response unchecked; `response.json()` buffers the full body.
3. A handful of concurrent requests like this can exceed the 128 MB isolate memory limit → worker exceptions unrelated in appearance to the actual cause.

## Evidence

```ts
// cached-fetch.ts:128-135
const contentLength = response.headers.get('Content-Length');
if (contentLength) {                                   // ← absent header = no check at all
  const size = parseInt(contentLength, 10);
  if (!isNaN(size) && size > MAX_RESPONSE_SIZE_BYTES) {
    throw new ResponseTooLargeError(size);
  }
}
return response;
```

## Suggested Fix

Stream with a byte budget instead of trusting the header:

```ts
async function readBounded(response: Response): Promise<string> {
  const reader = response.body!.getReader();
  const decoder = new TextDecoder();
  let text = '', bytes = 0;
  for (;;) {
    const { done, value } = await reader.read();
    if (done) return text;
    bytes += value.byteLength;
    if (bytes > MAX_RESPONSE_SIZE_BYTES) {
      await reader.cancel();
      throw new ResponseTooLargeError(bytes);
    }
    text += decoder.decode(value, { stream: true });
  }
}
// callers: JSON.parse(await readBounded(response))
```

Keep the Content-Length fast-fail as a cheap pre-check. Optionally map `ResponseTooLargeError` to a dedicated 502 message in `index.ts`.

## Why It's Hidden

Tests mock `fetch` with `Content-Length` always present, so the guarded path is the only one ever exercised; real chunked upstream behavior never appears in unit tests, and Universalis currently sends small enough bodies that production never probes the gap.
