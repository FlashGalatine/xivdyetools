# [BUG-025]: `sanitizeErrorMessage` bypassed by JSON-style quoted keys and `key = value` spacing

## Severity
MEDIUM

## Type
Security / redaction bypass (regex edge case)

## Location
`packages/logger/src/core/base-logger.ts:140-170` (`sanitizeErrorMessage`)

## Description
Every key/value redaction pattern requires the separator to *immediately* follow the key name: `` new RegExp(`token[=:]\\s*${V}`) `` — whitespace is only allowed *after* `[=:]`. Two extremely common error-message shapes therefore bypass sanitization entirely:

1. **JSON embedded in error messages.** Upstream API clients routinely echo response bodies into error strings: `Request failed: {"access_token":"eyJ…"}`. Here the key is quoted, so the character after `token` is `"`, not `[=:]` — no pattern matches and the secret passes through verbatim.
2. **Space before the separator.** `token = eyJ…` (common in config-dump or pretty-printed messages) has a space after `token`, so `token[=:]` never matches.

Since `sanitizeErrors: true` is the default for worker loggers (`JsonAdapter`, `presets/worker.ts:71`), these messages land unredacted in Cloudflare Logs.

## Reproduction Scenario
```ts
logger.error('Discord token exchange failed',
  new Error('HTTP 401: {"access_token":"SECRET_A","refresh_token":"SECRET_B"}'));
```
With default `sanitizeErrors: true`, the emitted log entry's `error.message` still contains `SECRET_A` and `SECRET_B`. Same for `new Error('token = SECRET_C rejected')`.

## Evidence
```ts
// packages/logger/src/core/base-logger.ts:147-162
const V = `(?:["']([^"']*?)["']|[^\\s,;]+)`;

return message
  .replace(/Bearer\s+\S+/gi, 'Bearer [REDACTED]')
  // requires separator IMMEDIATELY after the key name:
  .replace(new RegExp(`token[=:]\\s*${V}`, 'gi'), 'token=[REDACTED]')
  .replace(new RegExp(`secret[=:]\\s*${V}`, 'gi'), 'secret=[REDACTED]')
  ...
```
- `'{"token":"abc"}'` — char after `token` is `"` → no match → unchanged.
- `'token = abc'` — char after `token` is a space → no match → unchanged.

## Suggested Fix
Allow optional surrounding quotes and whitespace around the separator in the shared value pattern:

```ts
// key may be quoted, whitespace allowed on both sides of = or :
const K = (name: string) => `["']?${name}["']?\\s*[=:]\\s*`;
.replace(new RegExp(`${K('token')}${V}`, 'gi'), 'token=[REDACTED]')
```
Optionally add one dedicated JSON-shaped pass to catch all sensitive-suffixed keys at once:

```ts
.replace(/"([a-z0-9_-]*(?:token|secret|password|key))"\s*:\s*"[^"]*"/gi, '"$1":"[REDACTED]"')
```

## Why It's Hidden
The existing LOG-ERR-001 test suite covers `key=value`, `key: value`, and quoted-*value* forms — exactly the shapes the regex was written for, so all tests pass. Quoted *keys* (JSON) were never tested, yet JSON-serialized bodies inside error messages are precisely how third-party HTTP clients and the Discord/Universalis APIs format failures. The sanitized output for common cases looks correct, creating false confidence that "error sanitization is on."

## Status

**FIXED 2026-07-19** — key patterns now allow quoted keys and whitespace on both sides of the separator (catching `{"token":"…"}` and `token = …`), plus a dedicated JSON-shaped pass that redacts every sensitive-suffixed quoted key in one sweep (compound names like sessionToken/webhook_secret included).
