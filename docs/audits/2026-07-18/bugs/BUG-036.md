# [BUG-036]: KV read-modify-write on single-blob user data loses concurrent updates

## Severity
MEDIUM

## Type
Race condition / state management (lost updates)

## Location
- `apps/discord-worker/src/services/user-storage.ts:108-138` (`addFavorite`), `:273-333` (`createCollection`), `:380-427` (`renameCollection`), `:439-477` (`addDyeToCollection`), `:489-521` (`removeDyeFromCollection`)
- `apps/discord-worker/src/services/preferences.ts:128-196` (`setPreference`), `:238-245`
- `apps/discord-worker/src/services/preset-favorites.ts` (same pattern)

## Description
All favorites, collections, preset-favorites, and preferences for a user live in **one JSON value per user**, updated with get → mutate → put and no concurrency control. Cloudflare KV is last-write-wins and eventually consistent across colos (up to ~60 s propagation). Two interactions in flight for the same user — rapid successive `/favorites add` commands, Discord plus a second device, or two interactions landing on different colos — silently drop one write.

The existing comment at `apps/discord-worker/src/index.ts:729-736` (DISCORD-CRITICAL-002) acknowledges staleness for *autocomplete reads*, but the lossy *write* path is the substantive issue: a user adds two dyes to a collection and one vanishes.

## Reproduction Scenario
1. Fire `/collection add name:X dye:A` and `/collection add name:X dye:B` back-to-back (faster than the KV round-trip).
2. The second invocation reads the pre-A state, appends B, and overwrites.
3. Final collection contains only B; the user got a success message for A.

## Evidence
```ts
// apps/discord-worker/src/services/user-storage.ts:115-129 — non-atomic RMW
const favorites = await getFavorites(kv, userId, logger);

if (favorites.includes(dyeId)) {
  return { success: false, reason: 'alreadyExists' };
}
if (favorites.length >= MAX_FAVORITES) {
  return { success: false, reason: 'limitReached' };
}

favorites.push(dyeId);
await kv.put(`${FAVORITES_KEY_PREFIX}${userId}`, JSON.stringify(favorites));
```

## Suggested Fix
Cheapest: per-item keys (one key per favorite / collection membership), matching the `usertrack:` approach already adopted in `analytics.ts` (BUG-007 fix) — a single `put`/`delete` becomes atomic and there is nothing to race:
```
xivdye:favorites:v2:{userId}:{dyeId} -> "1"
```
(Reads become a prefix `list()`.) Structural alternative: a per-user Durable Object serializing mutations. Interim: keep single-blob but document the limitation and add a read-back verify (as `incrementCounter` does) so lost writes are at least detectable.

## Why It's Hidden
Mock-KV tests are sequential and strongly consistent, so the race is unreachable in CI. In production it needs overlapping requests from the same user and/or cross-colo replication lag; the symptom (an item missing later) is far removed in time from the cause and looks like user error.

## Status

**DEFERRED (documented) 2026-07-19** — the single-blob KV read-modify-write race is now prominently documented at both storage services. The durable fix (per-item keys or a per-user Durable Object) requires a data migration of existing user blobs and is deliberately not rushed into this sprint; treat as open backlog.
