# [BUG-023]: MemoryRateLimiter cleanup uses the current request's window to purge ALL keys (cross-config state loss)

## Severity
MEDIUM

## Type
State-management / logic error (shared mutable state governed by per-call config)

## Location
`packages/rate-limiter/src/backends/memory.ts:110-124` (cleanup triggers in `check()`), `164-179` (`cleanupOldEntries`)

## Description
`check(key, config)` triggers `cleanupOldEntries(config.windowMs * 2)` every `cleanupInterval` requests and again under LRU pressure — where `config` is the config of *this* request. `cleanupOldEntries` then iterates **every key in the map** and deletes timestamps older than `now - 2 * windowMs`. The API is explicitly designed for per-call configs (each `check` receives its own `RateLimitConfig`), so a single shared `MemoryRateLimiter` instance serving endpoints with different windows will have a short-window request purge the history of keys governed by a long window, silently resetting those limits.

This is latent today: the deployed users (`apps/oauth/src/services/rate-limit.ts:35`, `apps/presets-api/src/middleware/rate-limit.ts:17`) use uniform 60s windows across all their configs. It breaks the day any preset or endpoint adds a longer window to a shared instance.

## Reproduction Scenario
1. One limiter instance; endpoint A uses `{ windowMs: 3_600_000, maxRequests: 5 }` (5/hour), endpoint B uses `{ windowMs: 60_000, maxRequests: 100 }`.
2. A client exhausts A's 5/hour budget at t=0.
3. 100 requests to endpoint B arrive (any clients); at the 100th, cleanup runs with `maxAge = 120_000` ms and deletes every timestamp older than 2 minutes — including the client's endpoint-A history.
4. The client immediately gets 5 more requests on A. Repeatable every ~2 minutes → the "5/hour" limit is effectively 5 per 2 minutes.

## Evidence
```ts
// packages/rate-limiter/src/backends/memory.ts:110-124
this.requestCount++;
if (this.requestCount % this.cleanupInterval === 0) {
  this.cleanupOldEntries(config.windowMs * 2);   // current request's window applied to ALL keys
}

if (this.requestLog.size > this.maxEntries) {
  this.cleanupOldEntries(config.windowMs * 2);   // same problem under LRU pressure
  ...
}

// :164-179 — iterates the whole map with that single cutoff
private cleanupOldEntries(maxAge: number): void {
  const cutoff = Date.now() - maxAge;
  this.requestLog.forEach((timestamps, key) => { ... delete/splice by cutoff ... });
}
```

## Suggested Fix
Track each key's own window so cleanup is per-key correct:

```ts
private requestLog = new Map<string, { windowMs: number; timestamps: number[] }>();

// in check(): entry.windowMs = Math.max(entry.windowMs, config.windowMs);
// in cleanupOldEntries(): use per-entry cutoff = Date.now() - entry.windowMs * 2
```

Alternative (smaller change): make the cleanup max-age a constructor option (`maxWindowMs`) decoupled from per-call configs, defaulting to the largest window the app uses.

## Why It's Hidden
Unit tests exercise one config per limiter instance, so cleanup's cutoff always matches the only window in play. In production all current configs happen to share `windowMs: 60_000`, so behavior is coincidentally correct — the bug is invisible until a heterogeneous-window config lands, at which point it manifests as "long-window limits mysteriously reset," far from the code that caused it.

> Source: evidence/shared-packages-analysis.md (2026-07-18 deep-dive, shared-packages area)

## Status

**FIXED 2026-07-19** — `requestLog` entries now carry the largest `windowMs` each key has been checked with, and cleanup computes a per-key cutoff (`now - 2 * entry.windowMs`) instead of applying the current request's window to every key. Heterogeneous-window configs on a shared instance can no longer purge each other's history.
