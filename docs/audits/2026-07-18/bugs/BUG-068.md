# [BUG-068]: og-worker `/og/default.png` "7 days" comment actually yields a 49-day edge TTL

## Severity

LOW

## Type

Cache header miscalculation / misleading API contract in `renderOGImage`

## Location

- `apps/og-worker/src/index.ts:545` (call site with "Cache for 7 days" comment)
- `apps/og-worker/src/services/renderer.ts:109-126` (`renderOGImage(svgString, cacheMaxAge = 86400)` multiplies by 7 for edge TTL)

## Description

`renderOGImage` treats its `cacheMaxAge` parameter as the **browser** TTL and derives the edge TTL as `cacheMaxAge * 7`. The default-image route passes `604800` (7 days) intending "cache for 7 days", producing `s-maxage=4233600` and `CDN-Cache-Control: max-age=4233600` — **49 days** at the edge. Harmless while the branding card never changes; when it does change (logo/theme/text update), the old card can linger on Cloudflare's edge for up to 7 weeks with no URL-versioning to bust it.

## Reproduction Scenario

1. `curl -I https://og.xivdyetools.app/default.png` → `Cache-Control: public, max-age=604800, s-maxage=4233600`, `CDN-Cache-Control: max-age=4233600`.
2. Change the default card design and deploy; edge PoPs keep serving the old PNG for up to 49 days.

## Evidence

```ts
// renderer.ts:123-124
'Cache-Control': `public, max-age=${cacheMaxAge}, s-maxage=${cacheMaxAge * 7}`,
'CDN-Cache-Control': `max-age=${cacheMaxAge * 7}`,
```

```ts
// index.ts:545
return renderOGImage(svg, 604800); // Cache for 7 days   ← edge actually gets 49 days
```

## Suggested Fix

Make the TTL contract explicit instead of an implicit multiplier:

```ts
export async function renderOGImage(
  svgString: string,
  ttl: { browser: number; edge: number } = { browser: 86400, edge: 604800 }
): Promise<Response> { ... }
```

Then pass `{ browser: 86400, edge: 604800 }` (or just the default) for `/og/default.png`. If keeping the ×7 convention, pass `86400` at the call site and fix the comment.

## Why It's Hidden

The multiplier lives in the renderer while the comment lives at the call site; both read plausibly in isolation. Nothing asserts response cache headers in tests, and a too-long TTL produces no error — only a weeks-later "why is the old image still showing" mystery.
