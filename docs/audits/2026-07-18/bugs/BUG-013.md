# [BUG-013]: Cached rejected init promise permanently poisons the isolate (resvg-wasm renderers + CharacterColorService lazy loaders)

## Severity

MEDIUM

## Type

Resource/state management — module-level promise cache never reset on rejection (systemic pattern, 3 instances)

## Location

1. `apps/og-worker/src/services/renderer.ts:20-52` (`wasmInitPromise` for `initWasm(resvgWasm)`)
2. `apps/discord-worker/src/services/svg/renderer.ts:39` (same `wasmInitPromise` pattern, lines 21-61)
3. `packages/core/src/services/CharacterColorService.ts:174-184` (`hairColorsLoading`) and `:196-206` (`skinColorsLoading`) — lazy `import()` of race-specific color JSON

## Description

All three sites cache an in-flight initialization promise at module/instance scope to deduplicate concurrent init, but **never clear the cached promise when it rejects**:

- **og-worker / discord-worker renderers:** `initRenderer()` stores `wasmInitPromise` before awaiting. If the first `initWasm()` call fails (realistic under isolate memory pressure at cold start for these multi-MiB-bundle workers), `wasmInitialized` stays `false` and every subsequent request `await`s the **same rejected promise**. The isolate then serves 500 (`Image generation failed`) / failed Discord card renders for its entire lifetime, until Cloudflare happens to recycle it.
- **CharacterColorService:** `loadHairColors()` / `loadSkinColors()` store the `import(...).then(...)` promise in `hairColorsLoading` / `skinColorsLoading`. The `.then` callback (which resets the field to `null`) only runs on **success**; on rejection the field keeps the rejected promise forever, so every later call to `getHairColors`/`getSkinColors` rethrows the original failure with no retry. This propagates into og-worker's swatch OG images and discord-worker preference features.

One transient failure thus becomes a persistent per-isolate outage — the worst kind of intermittent production incident.

## Reproduction Scenario

1. Stub `initWasm` (or the dynamic `import`) to reject once, then succeed on subsequent calls.
2. Request 1 → 500, as expected.
3. Requests 2..N (same isolate) → still 500 forever, even though a retry would have succeeded; a fresh isolate works fine — classic "some users broken, redeploy fixes it" symptom.

## Evidence

```ts
// og-worker renderer.ts:28-51 (discord-worker renderer.ts:31-61 identical shape)
if (wasmInitPromise) {
  await wasmInitPromise;   // ← rejected promise re-awaited forever
  return;
}
wasmInitPromise = (async () => {
  try {
    await initWasm(resvgWasm);
    wasmInitialized = true;
  } catch (error) {
    ...
    throw new Error(...);  // promise rejects; wasmInitPromise never reset
  }
})();
await wasmInitPromise;
```

```ts
// packages/core/src/services/CharacterColorService.ts:174-184
if (!this.hairColorsLoading) {
  this.hairColorsLoading = import('../data/character_colors/race_specific/hair_colors.json')
    .then((module) => {
      this.hairColorsData = module.default;
      this.hairColorsLoading = null;   // ← only reset on SUCCESS
      return this.hairColorsData;
    });                                 // no .catch → rejected promise cached forever
}
return this.hairColorsLoading;
```

## Suggested Fix

One pattern, applied at all three sites — reset the cached promise on failure so the next request retries:

```ts
// renderers
wasmInitPromise = (async () => { ... })();
wasmInitPromise.catch(() => { wasmInitPromise = null; });  // retry path; keep wasmInitialized guard
await wasmInitPromise;

// CharacterColorService
this.hairColorsLoading = import('...').then((m) => {
  this.hairColorsData = m.default;
  this.hairColorsLoading = null;
  return this.hairColorsData;
}).catch((err) => {
  this.hairColorsLoading = null;   // allow retry
  throw err;
});
```

Note for the renderers: keep the `wasmInitialized` flag check — `initWasm` throws "Already initialized" if invoked twice after a successful init, so only the *failure* path must reset.

## Why It's Hidden

- Tests either mock the renderer/imports or run where init always succeeds; the failure mode only exists **across multiple requests to the same isolate after one bad init** — a shape no unit test exercises.
- In production it presents as an intermittent per-PoP outage that "self-heals" on isolate recycling or redeploy, so it is chronically misattributed to Cloudflare flakiness.
