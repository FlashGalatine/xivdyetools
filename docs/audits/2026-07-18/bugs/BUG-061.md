# [BUG-061]: worker-middleware rate-limit backend factory silently breaks stateful limiters; caught errors dropped

## Severity
LOW

## Type
API foot-gun (per-request state reset) + error-handling gap (swallowed exception details)

## Location
`packages/worker-middleware/src/rate-limit.ts:36` (backend factory option), `:114-119` (per-request factory invocation), `:123-135` (catch that discards the error)

## Description
Two related defects in `rateLimitMiddleware`:

1. **Factory foot-gun.** The `backend` option accepts `RateLimiter | ((c: Context) => RateLimiter)`, with the factory form intended for request-time bindings (KV namespaces live on `c.env`). The middleware invokes the factory on **every request** (`:118-119`). If a worker passes `backend: (c) => new MemoryRateLimiter(...)` — a perfectly natural reading of the factory shape — a fresh limiter with an empty map is constructed per request, and rate limiting is **silently disabled**: every check sees zero prior requests and allows. Nothing warns, no type distinction exists, and the option's JSDoc ("for lazy initialization when the backend needs request-time bindings") doesn't say stateful backends must not be constructed there. (`KVRateLimiter` survives per-request construction only because its state lives in KV — api-worker relies on exactly that at `apps/api-worker/src/middleware/rate-limit.ts:24-26`.)

2. **Swallowed error.** The `catch` around `backend.check()` logs `'Rate limiter backend error'` with `onError/key/path/method` — but binds no error variable, so the actual failure (KV outage, bad binding name, JSON corruption) is discarded. Backend failures become undiagnosable from logs; operators see only that *something* failed.

## Reproduction Scenario
1. Worker configures `rateLimitMiddleware({ backend: (c) => new MemoryRateLimiter(), keyExtractor, config: { maxRequests: 5, windowMs: 60_000 } })`.
2. Send 100 requests/min from one IP → all allowed; every response says `X-RateLimit-Remaining: 4`. No error anywhere.
3. Separately: point a `KVRateLimiter` at a nonexistent binding → every request logs the generic warn with no cause; fail-open admits all traffic while the misconfiguration stays invisible.

## Evidence
```ts
// packages/worker-middleware/src/rate-limit.ts:114-135
return async (c, next): Promise<Response | void> => {
  ...
  const backend =
    typeof options.backend === 'function' ? options.backend(c) : options.backend; // per-request

  let result: RateLimitResult;
  try {
    result = await backend.check(key, config);
  } catch {                                    // error binding dropped entirely
    const logger = c.get('logger');
    if (logger) {
      logger.warn('Rate limiter backend error', {
        onError, key, path: c.req.path, method: c.req.method,   // no error info
      });
    }
```

## Suggested Fix
```ts
// 1) Memoize the factory result per isolate + document the constraint:
let cached: RateLimiter | undefined;
const resolveBackend = (c: Context) =>
  typeof options.backend === 'function' ? (cached ??= options.backend(c)) : options.backend;
// JSDoc on `backend`: "Factory results are cached; must be safe to reuse across requests.
//  Never construct a MemoryRateLimiter inside the factory."

// 2) Keep the error:
} catch (err) {
  logger?.warn('Rate limiter backend error', {
    onError, key, path: c.req.path, method: c.req.method,
    error: err instanceof Error ? err.message : String(err),
  });
```

## Why It's Hidden
A per-request MemoryRateLimiter produces *plausible* headers (`remaining = limit - 1`) and never blocks — indistinguishable from light traffic unless someone load-tests the limit. Current first-party workers happen to use the safe patterns (module-scope Memory instance, or KV via factory), so nothing is broken today; the trap awaits the next integrator. The swallowed error is invisible by construction: fail-open means requests succeed, and the warn line carries no cause to grep for.
