# [BUG-043]: Changelog modal misrenders when `APP_VERSION` is missing from CHANGELOG-laymans.md

## Severity
MEDIUM

## Type
Edge-case logic error (v4.11.0 "What's New" feature, PR #72)

## Location
- `apps/web-app/src/components/changelog-modal.ts:83-102` (`getRelevantEntries`)
- `apps/web-app/src/components/changelog-modal.ts:182-217` (non-full render; unchecked `highlights[0]` at `:211`)
- Parser: `apps/web-app/vite-plugin-changelog-parser.ts:82-133` (header regex `:88-89`, sections guard `:122`, `extractHighlights` `:142-158`)

## Description
The auto-popup path assumes the current version exists in the parsed changelog. If a release bumps `APP_VERSION` but the laymans changelog entry is missing — or its header does not match the parser regex (e.g. wrong dash character before the date, or an entry with no `###` sections, which the `sections.length > 0` guard drops) — then `findIndex` returns `-1` and `slice(0, 2)` returns the **two newest old releases**. The non-full renderer then presents `entries[0]` — an *old* release — as the current version **with no version heading** (`:184-188` renders sections only), listing the second-newest under "Previous updates". Users are shown the wrong release notes as "what's new".

Additionally, `entry.highlights[0]` is interpolated unchecked at `:211`; the parser can produce entries with `sections.length > 0` but **empty** `highlights` (every `###` header shorter than 3 chars is skipped in `extractHighlights`), rendering the literal text "— undefined".

Minor related note: the auto-popup (`showChangelogIfUpdated`, `:352-360`) and the header singleton (`showChangelogModal`, `:372-377`) are independent instances with independent `modalId` guards, so the full-history modal can stack on top of the auto-popup.

## Reproduction Scenario
Bump `APP_VERSION` to 4.12.0 without adding a `## Web-App Version 4.12.0 — …` block to CHANGELOG-laymans.md (or typo the header, e.g. `Version 4.12` or a colon instead of a dash). Returning users get the v4.11.0 notes presented unlabeled as the new release; closing the modal marks 4.12.0 as viewed, so the mistake is shown exactly once per user and then hidden.

## Evidence
```ts
// changelog-modal.ts:89-101
const currentEntry = changelogEntries.find((e) => e.version === APP_VERSION);
const currentIndex = changelogEntries.findIndex((e) => e.version === APP_VERSION); // -1 if absent
const previousEntries = changelogEntries.slice(currentIndex + 1, currentIndex + 3); // slice(0, 2)!

const entries: ChangelogEntry[] = [];
if (currentEntry) entries.push(currentEntry);
entries.push(...previousEntries);        // two newest OLD releases become "current + previous"

// changelog-modal.ts:203-212 — unchecked highlights[0]
item.appendChild(document.createTextNode(` — ${entry.highlights[0]}`)); // "— undefined" possible
```

## Suggested Fix
In `getRelevantEntries`, when `currentEntry` is undefined, either return `[]` (falls through to the existing `changelog.noChanges` fallback at `:154-159`) or render the entries with explicit version headings (reuse the `full` layout). Guard the highlights line:

```ts
const summary = entry.highlights[0] ?? entry.sections[0]?.header ?? '';
item.appendChild(document.createTextNode(summary ? ` — ${summary}` : ''));
```
Optionally add a CI assertion that `APP_VERSION` parses out of CHANGELOG-laymans.md at build time.

## Why It's Hidden
The changelog-modal unit tests mock `virtual:changelog` with entries that include the current version; the mismatch only occurs from a release-process slip (missing/typoed laymans entry), exactly when nobody is testing the modal.
