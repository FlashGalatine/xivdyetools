# [BUG-045]: getNonMetallicDyes silently returns metallic dyes when no locale was ever loaded

## Severity
LOW

## Type
Silent misbehavior / hidden initialization-order dependency

## Location
- `packages/core/src/services/DyeService.ts:480-483`
- `packages/core/src/services/localization/TranslationProvider.ts:232-242` (`getMetallicDyeIds` empty-registry fallback)

## Description
`DyeService.getNonMetallicDyes()` sources its exclusion list from the LocalizationService singleton. If `setLocale()` was never called, the locale registry is empty, `TranslationProvider.getMetallicDyeIds` returns `[]` (the English fallback is also unloaded), and the method returns **all** dyes including metallics — with no warning and no error. The method's documented contract ("Excludes: Metallic Silver, Metallic Brass, etc.") silently fails based on an unrelated initialization order.

Underlying smell: the metallic ID list is locale-*independent* data that happens to live in the locale JSON files, coupling a dye-filtering concern to localization state.

## Reproduction Scenario
```ts
// Fresh process, no LocalizationService.setLocale() call anywhere:
const dyeService = new DyeService(dyeDatabase);
const result = dyeService.getNonMetallicDyes();
// → metallic dyes ARE present in the result; no warning logged
```

## Evidence
```ts
// DyeService.ts:480-483
getNonMetallicDyes(): Dye[] {
  const metallicIds = new Set(LocalizationService.getMetallicDyeIds());
  return this.getAllDyes().filter((dye) => !metallicIds.has(dye.itemID));
}
```
```ts
// TranslationProvider.ts:232-242
getMetallicDyeIds(locale: LocaleCode): number[] {
  const localeData = this.registry.getLocale(locale);
  if (localeData?.metallicDyeIds) {
    return localeData.metallicDyeIds;
  }
  // Fallback to English
  const englishData = this.registry.getLocale('en');
  return englishData?.metallicDyeIds || [];   // ← [] when nothing loaded → filter removes nothing
}
```

## Suggested Fix
Preferred: remove the localization dependency entirely — dyes already carry an `isMetallic` flag in `colors_xiv.json`:
```ts
getNonMetallicDyes(): Dye[] {
  return this.getAllDyes().filter((dye) => !dye.isMetallic);
}
```
Alternative: log a warning (or lazily load `en`) when `getMetallicDyeIds` is called with an empty registry.

## Why It's Hidden
Test setups (and most apps) call `setLocale` early, so the registry is always populated in practice. Only a consumer that never touches localization hits the empty-registry path — and the failure mode is "extra items in a list", which nothing asserts against.

## Status

**FIXED 2026-07-19** — `getNonMetallicDyes` filters on the dye data's own `isMetallic` flag, removing the localization-state dependency entirely.
