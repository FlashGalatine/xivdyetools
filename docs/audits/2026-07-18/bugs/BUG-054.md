# [BUG-054]: presets-api Content-Type enforcement trusts Content-Length — chunked bodies bypass it (and the JSON depth limit)

## Severity
LOW

## Type
Edge Case / Defense-in-Depth Erosion (header-trusting validation)

## Location
- File: `apps/presets-api/src/index.ts`
- Line(s): 144-163 (Content-Type middleware, `hasBody` gate at 149-150)
- Related: `apps/presets-api/src/middleware/body-validation.ts:50-52` (`jsonDepthLimit` also gates on content-type)
- Function: global Content-Type validation middleware

## Description
The 415 gate only fires when a `content-length` header is present and > 0 (`hasBody`). A request with `Transfer-Encoding: chunked` (no Content-Length) and a wrong/absent Content-Type skips the check entirely. Downstream, `jsonDepthLimit` *also* returns early for non-JSON content types, so such a body reaches route handlers unvetted for nesting depth and prototype-pollution keys — Hono's `c.req.json()` parses the body regardless of the header. This is defense-in-depth erosion only: handlers wrap `c.req.json()` in try/catch, and prototype-pollution keys are only a risk if some future code merges parsed bodies into objects — but two layered protections (SEC-003 depth limit, Content-Type gate) can both be skipped by the same trivial header choice.

## Reproduction Scenario
1. `POST /api/v1/presets` with `Transfer-Encoding: chunked`, `Content-Type: text/plain`, and a deeply-nested (>10 levels) JSON body plus valid auth headers.
2. The Content-Type middleware sees no `content-length` → `hasBody` false → no 415.
3. `jsonDepthLimit` sees non-JSON content-type → skips depth/`__proto__` validation.
4. The handler's `c.req.json()` parses it and processing proceeds on a payload both guards were designed to reject.

## Evidence
```ts
// index.ts:149-152
const contentLength = c.req.header('content-length');
const hasBody = contentLength && parseInt(contentLength) > 0;
if (hasBody && (!contentType || !contentType.includes('application/json'))) {
  return c.json({ error: 'Unsupported Media Type', ... }, 415);
```
```ts
// body-validation.ts:50-52 — same header-trusting gate
const contentType = c.req.header('content-type');
if (!contentType?.includes('application/json')) return next();
```

## Suggested Fix
Drop the `hasBody` inference: for POST/PATCH/PUT, require `application/json` outright (empty-body requests can still send the header or be exempted by method+route):
```ts
if (['POST', 'PATCH', 'PUT'].includes(method)) {
  const contentType = c.req.header('content-type');
  if (!contentType || !contentType.includes('application/json')) {
    return c.json({ error: 'Unsupported Media Type', message: 'Content-Type must be application/json' }, 415);
  }
}
```
This also restores `jsonDepthLimit` coverage, since the depth check then only ever sees JSON-labeled bodies that were not rejected.

## Why It's Hidden
Test clients (and browsers/fetch) always send Content-Length for JSON bodies, so the chunked path never occurs in CI; the two middlewares individually look correct and only their combined gating logic leaves the gap.
