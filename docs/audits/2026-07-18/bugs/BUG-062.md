# [BUG-062]: test-utils MockD1 `exec()` desynchronizes `_queries` from `_bindings`

## Severity
LOW

## Type
Test-infrastructure correctness (parallel-array misalignment)

## Location
`packages/test-utils/src/cloudflare/d1.ts:284-288` (`exec` on the db object), `:308-312` (`exec` on the session object), vs `:158-163` (`enforceMaxHistory`)

## Description
Every prepared-statement method (`first`/`all`/`run`/`raw`) pushes to `queries` **and** `bindings` in lockstep, keeping the two arrays index-aligned — which is exactly what the mock's own documented assertion pattern relies on (`d1.ts:27-28`: `expect(db._queries).toContain(...)`; `expect(db._bindings[0]).toEqual(['param1'])`). But both `exec()` implementations push only to `queries`. After any `exec` call, `_bindings[i]` no longer corresponds to `_queries[i]`: every subsequent statement's bindings appear shifted one slot earlier relative to its query.

`enforceMaxHistory` compounds this: it loops `while (queries.length > maxQueryHistory)` shifting **both** arrays, but since `bindings` is shorter (exec never fed it), eviction removes bindings for queries that are still retained, skewing alignment further in long suites.

## Reproduction Scenario
```ts
const db = createMockD1Database();
await db.exec('PRAGMA foreign_keys = ON');                 // queries: 1, bindings: 0
await db.prepare('SELECT * FROM presets WHERE id = ?')
        .bind('preset-1').first();                          // queries: 2, bindings: 1

db._queries[1];   // 'SELECT * FROM presets WHERE id = ?'
db._bindings[1];  // undefined  ← positional assertion silently checks the wrong slot
db._bindings[0];  // ['preset-1'] — actually belongs to _queries[1]
```
A test asserting `expect(db._bindings[1]).toEqual(['preset-1'])` fails confusingly; worse, a test asserting `_bindings[0]` may pass *incorrectly* against the wrong query.

## Evidence
```ts
// statement methods keep the arrays aligned (e.g. :184-187):
first: async <T = unknown>() => {
  queries.push(query);
  bindings.push(boundValues);
  enforceMaxHistory();
  ...

// :284-288 — exec pushes only queries:
exec: async (query: string) => {
  queries.push(query);              // no bindings.push
  enforceMaxHistory();
  return { count: 1, duration: 0 };
},
// same at :308-312 in withSession()'s exec
```

## Suggested Fix
One line in each `exec` (db object and session object):

```ts
exec: async (query: string) => {
  queries.push(query);
  bindings.push([]);                // keep parallel arrays aligned
  enforceMaxHistory();
  return { count: 1, duration: 0 };
},
```

## Why It's Hidden
`exec` is used for schema/PRAGMA setup, and most tests assert on `_queries` content (`toContain`) rather than positional `_bindings` — so alignment rot goes unnoticed. When a positional assertion does misfire, it looks like a flaky/wrong test expectation, and authors "fix" the index instead of the mock. Nothing in the mock itself can fail; the damage is confined to the diagnostic quality of other packages' tests.

> Source: evidence/shared-packages-analysis.md (2026-07-18 deep-dive, shared-packages area)

## Status

**FIXED 2026-07-19** — both `exec()` implementations (db and session) push `[]` to `bindings`, keeping `_queries`/`_bindings` index-aligned for positional assertions and `enforceMaxHistory` eviction.
