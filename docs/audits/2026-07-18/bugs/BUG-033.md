# [BUG-033]: Universalis aggregated parsing always reads DC-scope prices, even for world-scoped queries

## Severity
MEDIUM

## Type
Logic error / wrong data scope

## Location
`apps/discord-worker/src/services/budget/universalis-client.ts:226-244` (parsing), `:32-47` (response typing omits the `world` key entirely)

## Description
The Universalis v2 `/aggregated/{worldDcRegion}/{ids}` response nests each statistic under `world` / `dc` / `region` keys, where `world` is populated when the query scope is a single world. The client reads **only `.dc`** for min listing, average sale price, and daily velocity.

Consequences when the user has set a **specific home world** (via `/budget set_world` or the `world:` option):
- Prices shown are the *datacenter-wide minimum*, which may not exist on the user's world at all.
- Savings math and the "on {world}" label attribute DC prices to the world.
- Listings-empty detection is also evaluated at the wrong scope.

## Reproduction Scenario
1. Pick a dye with a large price spread across a datacenter (cheap on world A, expensive on world B).
2. Run `/budget find` with `world:<world B>`.
3. The embed reports world A's (DC-minimum) price, labeled as world B.

## Evidence
```ts
// apps/discord-worker/src/services/budget/universalis-client.ts:226-233
const minListingPrice = item.nq?.minListing?.dc?.price;
if (minListingPrice == null) {
  continue;
}
const averagePrice = item.nq.averageSalePrice?.dc?.price ?? minListingPrice;
const lastUpload = item.worldUploadTimes?.[0]?.timestamp ?? Date.now();
const velocity = item.nq.dailySaleVelocity?.dc?.quantity ?? 0;
```
```ts
// :35 — the interface has no `world` key at all
minListing: { dc?: { price: number; worldId: number }; region?: { price: number; worldId: number } };
```

## Suggested Fix
Add `world?: { price: number; worldId: number }` (etc.) to the response typing and prefer world scope with fallback:
```ts
const listing = item.nq?.minListing;
const minListingPrice = listing?.world?.price ?? listing?.dc?.price ?? listing?.region?.price;
```
(Same cascade for `averageSalePrice` and `dailySaleVelocity`.) Alternatively, if DC-scope pricing is *intended* ("cheapest on your DC — travel to buy"), label it as such instead of attributing it to the queried world. Verify field availability against the live aggregated schema before shipping.

## Why It's Hidden
Mocked Universalis responses in tests are hand-written to match the parser's expectations (dc-only), so the scope mismatch never surfaces. Detecting it requires comparing live per-world output against the in-game market board — and DC-min prices are always ≤ world prices, so the output looks plausible.

## Status

**FIXED 2026-07-19** — the aggregated-response typing gained the `world` scope and all three statistics cascade world → dc → region, so world-scoped queries report the queried world's prices instead of the DC minimum.
