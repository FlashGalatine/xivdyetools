# [BUG-055]: UpstashRateLimiter reports `resetAt`/`retryAfter` as a full window regardless of actual remaining TTL

## Severity
LOW

## Type
Logic error (incorrect client-facing rate-limit headers)

## Location
`packages/rate-limiter/src/backends/upstash.ts:66-90` (esp. lines 82 and 89); surfaced via `getRateLimitHeaders` in `packages/rate-limiter/src/headers.ts:37-51`

## Description
The Upstash backend implements a fixed window: `INCR` plus `EXPIRE key ttl NX`, so the TTL is anchored at the *first* request of the window. But every subsequent `check()` computes `resetAt = new Date(Date.now() + ttlSeconds * 1000)` and, when the request is blocked, `retryAfter: ttlSeconds` ŌĆö i.e. always a full `windowMs` measured from *now*, never the key's actual remaining TTL. Consequences:

- `Retry-After` and `X-RateLimit-Reset` overstate the wait by up to one full window.
- The advertised reset time drifts forward on every request even though the real window boundary is fixed.
- Well-behaved clients that honor `Retry-After` back off far longer than necessary.

## Reproduction Scenario
Config `{ maxRequests: 2, windowMs: 60_000 }`:
1. t=0s ŌĆö request 1: `INCR` ŌåÆ 1, TTL set to 60s. Allowed.
2. t=50s ŌĆö request 2: allowed.
3. t=55s ŌĆö request 3: `count 3 > 2` ŌåÆ blocked with `retryAfter: 60` and `resetAt = t+60s`. The key actually expires at t=60s ŌĆö real wait is 5 seconds, client is told 60.

## Evidence
```ts
// packages/rate-limiter/src/backends/upstash.ts:73-89
const pipeline = this.redis.pipeline();
pipeline.incr(redisKey);
pipeline.expire(redisKey, ttlSeconds, 'NX');   // TTL anchored at FIRST request
const results = await pipeline.exec<[number, number]>();

const count = results[0];
const allowed = count <= effectiveLimit;
const remaining = Math.max(0, effectiveLimit - count);
const resetAt = new Date(Date.now() + ttlSeconds * 1000);  // ignores actual TTL

return {
  allowed, remaining, resetAt,
  limit: effectiveLimit,
  retryAfter: allowed ? undefined : ttlSeconds,            // always full window
};
```

## Suggested Fix
Pipeline a `TTL` command alongside ŌĆö same single round-trip:

```ts
const pipeline = this.redis.pipeline();
pipeline.incr(redisKey);
pipeline.expire(redisKey, ttlSeconds, 'NX');
pipeline.ttl(redisKey);
const [count, , ttlLeft] = await pipeline.exec<[number, number, number]>();

const remainingTtl = ttlLeft > 0 ? ttlLeft : ttlSeconds;
const resetAt = new Date(Date.now() + remainingTtl * 1000);
...
retryAfter: allowed ? undefined : remainingTtl,
```

## Why It's Hidden
Unit tests assert that `retryAfter` equals the configured window ŌĆö which is exactly what the buggy formula returns ŌĆö so the tests encode the bug. At runtime nothing fails: 429 responses are still valid, just pessimistic, and clients waiting too long looks like conservative behavior rather than an error.

## Status

**FIXED 2026-07-19** ŌĆö the pipeline gained a `TTL` command (same single round-trip); `resetAt`/`retryAfter` now report the key's actual remaining TTL instead of a full window from now. Tests updated (they previously encoded the bug).
