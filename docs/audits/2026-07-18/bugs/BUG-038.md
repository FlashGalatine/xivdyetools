# [BUG-038]: stoat-worker reaction context stored under the wrong message ID, overwritten in multi-match loops — and no reaction listener exists

## Severity
MEDIUM

## Type
Logic error / state management (latent, feature-blocking)

## Location
- `apps/stoat-worker/src/commands/info.ts:106-125` (`sendDyeInfoResponse`)
- `apps/stoat-worker/src/commands/info.ts:71-79` (`multiple` branch overwrite)
- `apps/stoat-worker/src/index.ts:38-89` (only `ready` + `messageCreate` handlers registered)

## Description
Three compounding problems in the reactions-as-buttons design:

1. **Wrong key.** The bot's reply is sent and its return value discarded; context is stored under the **user's** message ID (`ctx.message.id`). But the reactions (`DYE_INFO_REACTIONS`, `restrict_reactions: true`) live on the **bot's reply**. Any future reaction handler will receive the *bot message's* ID and find nothing in the `MessageContextStore`.
2. **Overwrite on multi-match.** In the `multiple` branch every `sendDyeInfoResponse` call `set()`s the same `ctx.message.id`, so only the last dye's context survives even under the wrong-key scheme.
3. **No listener.** `index.ts` registers no reaction event handler at all, so the reaction affordances rendered on every dye-info reply do nothing today.

## Reproduction Scenario
1. `!xd info snow` → the reply appears with restricted reaction options, inviting clicks.
2. Click any reaction → nothing happens (no listener).
3. Latent parts 1–2 bite whoever implements the listener: lookups by the reacted (bot) message ID always miss, and multi-match responses share one clobbered context entry.

## Evidence
```ts
// apps/stoat-worker/src/commands/info.ts:106-125
await ctx.message.channel?.sendMessage({
  embeds: [embed],
  replies: [{ id: ctx.message.id, mention: false }],
  ...
  interactions: {
    reactions: DYE_INFO_REACTIONS,
    restrict_reactions: true,          // reactions live on the BOT's reply
  },
});                                     // <- returned Message (with its id) discarded

// Track message context for reaction handling
ctx.messageContextStore.set(ctx.message.id, {   // <- keyed by the USER's message id
  command: 'dye-info',
  dyeId: result.dye.id,
  dyeHex: result.dye.hex,
  createdAt: Date.now(),
});
```
```ts
// apps/stoat-worker/src/index.ts — only these handlers exist
client.on('ready', () => { ... });
client.on('messageCreate', async (message) => { ... });
```

## Suggested Fix
Capture the sent message and key by it (revolt.js `sendMessage` resolves to the created `Message`):
```ts
const sent = await ctx.message.channel?.sendMessage({ ... });
if (sent) {
  ctx.messageContextStore.set(sent.id, { command: 'dye-info', dyeId: result.dye.id, dyeHex: result.dye.hex, createdAt: Date.now() });
}
```
Then either register `client.on('messageReactionAdd', ...)` now, or remove `interactions.reactions` from the reply until the handler ships so the UI stops advertising dead controls.

## Why It's Hidden
`info.test.ts` asserts the store was populated — it was, just under a key that will never be queried — and no test spans the send → reaction → lookup round trip. With no listener registered, nothing ever performs the lookup that would expose the wrong key.

## Status

**FIXED 2026-07-19** — the bot reply's Message is captured and context is keyed by ITS id (per reply — no more multi-match overwrites), and the dead reaction affordances were removed until a reaction listener ships (re-add `interactions.reactions` with the listener).
