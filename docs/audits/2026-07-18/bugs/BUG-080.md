# [BUG-080]: DyeSearchBox renders "All" category button as active even when a category filter is set

## Severity
LOW

## Type
UI state logic error (initial render)

## Location
- `apps/web-app/src/components/dye-search-box.ts:170` (faulty condition)
- Context: constructor options `:27-39`, click handler that fixes it `:248-279`

## Description
The active-state check for category chips unconditionally styles the "All" chip as active. When the component is constructed with `initialCategory` (restored filter state passed via options, `:33-37`), the initial render highlights **both** "All" and the actual category chip. The state is misleading until the user clicks any chip, at which point the click handler rebuilds the classes correctly.

## Reproduction Scenario
Instantiate `new DyeSearchBox(container, { initialCategory: 'Red' })` (as done when restoring persisted filter state) and render: both the "All" chip and the "Red" chip show the blue active styling, implying no filter is applied while the list is actually filtered to Red.

## Evidence
```ts
// dye-search-box.ts:170-183
const isActive = cat === 'all' || cat === this.currentCategory;  // 'all' is ALWAYS active

if (isActive) {
  btn.classList.remove('border-gray-300', ...);
  btn.classList.add('bg-blue-500', 'text-white');
}
```

## Suggested Fix
```ts
const isActive =
  this.currentCategory === null ? cat === 'all' : cat === this.currentCategory;
```

## Why It's Hidden
The default construction path passes no `initialCategory` (so "All" being active is correct), and manual testing usually starts from the default state; the wrong dual-highlight appears only when filter state is restored, and self-corrects on the first click.

> Source: evidence/web-frontends-analysis.md (2026-07-18 deep-dive, web-frontends area)

## Status

**FIXED 2026-07-19** — the "All" chip is active only when `currentCategory` is null; restoring a persisted category filter no longer dual-highlights.
