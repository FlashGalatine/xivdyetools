# [BUG-073]: MODERATOR_IDS parsed with different grammars by the two bot workers

## Severity
LOW

## Type
Configuration-parsing inconsistency (fails closed, silently)

## Location
`apps/discord-worker/src/services/preset-api.ts:207-211` vs. `apps/moderation-worker/src/services/preset-api.ts:304-312`

## Description
The same conceptual secret (`MODERATOR_IDS`) is parsed two different ways:

- **discord-worker:** `env.MODERATOR_IDS.split(',').map(id => id.trim())` — comma-only. A newline- or space-separated list leaves whitespace-embedded tokens that never match a real user ID.
- **moderation-worker:** validated `getModerators()` parsing with snowflake validation (documented in its CLAUDE.md as supporting whitespace/comma separators), plus `isValidDiscordSnowflake(userId)` on the incoming ID.

One secret value, two grammars. A `MODERATOR_IDS` formatted the way the moderation bot documents (e.g. newline-separated for readability) works on the moderation bot and **silently fails closed** in discord-worker's `/preset` moderator checks — moderators just appear to lack permission, with no log signal.

## Reproduction Scenario
1. Set `MODERATOR_IDS` to a newline-separated list (valid per moderation-worker's docs) on both workers.
2. Moderation-worker commands work.
3. Any discord-worker code path calling its own `isModerator()` returns `false` for every moderator (tokens like `"123\n456"` never equal a real snowflake).

## Evidence
```ts
// apps/discord-worker/src/services/preset-api.ts:207-211 — comma-only, no validation
export function isModerator(env: Env, userId: string): boolean {
  if (!env.MODERATOR_IDS) return false;
  const moderatorIds = env.MODERATOR_IDS.split(',').map((id) => id.trim());
  return moderatorIds.includes(userId);
}
```
```ts
// apps/moderation-worker/src/services/preset-api.ts:304-312 — validated variant
export function isModerator(env: Env, userId: string): boolean {
  if (!env.MODERATOR_IDS) return false;
  if (!isValidDiscordSnowflake(userId)) {
    return false;
  }
  const moderators = getModerators(env);
  ...
```

## Suggested Fix
Extract a single `parseModeratorIds()` helper (natural home: the shared bot client package proposed in REFACTOR-010) using `value.split(/[\s,]+/).filter(Boolean)` plus snowflake validation, and use it in both workers.

## Why It's Hidden
Both implementations pass their own unit tests, each written against its own grammar. The divergence only manifests with a specific secret formatting, and the failure mode (permission denied) looks like intended access control rather than a parsing bug.
