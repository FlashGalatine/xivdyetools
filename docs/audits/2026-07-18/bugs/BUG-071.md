# [BUG-071]: api-worker returns `marketItemID` values that 404 when round-tripped to `GET /v1/dyes/:id`

## Severity

LOW

## Type

API contract gap (Patch 7.5 consolidation IDs not resolvable)

## Location

- `apps/api-worker/src/lib/dye-serializer.ts:51` (`marketItemID: getMarketItemID(dye)`)
- `apps/api-worker/src/routes/dyes.ts:192-222` (`GET /:id` auto-detect lookup)
- `apps/api-worker/src/lib/validation.ts:50-55` (`resolveIdType` — `id >= 5729` ⇒ item lookup)

## Description

Every serialized dye exposes `marketItemID` — for the 105 consolidated dyes this is one of the Patch 7.5 consolidated itemIDs (Type-A=52254 / B=52255 / C=52256). Those values fall in the `id >= 5729` range, so `GET /v1/dyes/52254` resolves as an item lookup — and returns 404, because the consolidated itemIDs are **not** entries in the dye database (legacy itemIDs remain the per-dye keys). A consumer that round-trips the field the API itself just returned gets `NOT_FOUND` with no hint. `/v1/dyes/consolidation-groups` contains exactly the needed mapping, but nothing links the 404 to it.

## Reproduction Scenario

1. `GET /v1/dyes/5729` → response includes `"marketItemID": 52254` (Type-A example).
2. `GET /v1/dyes/52254` → `404 NOT_FOUND: "No dye found with ID 52254."` — the ID the API emitted one call earlier.

## Evidence

```ts
// dye-serializer.ts:51 — emitted on every dye
marketItemID: getMarketItemID(dye),

// validation.ts:50-55 — 52254 classified as an item ID...
export function resolveIdType(id: number): IdResolution {
  if (id < 0) return { type: 'facewear', id };
  if (id >= 1 && id <= 125) return { type: 'stain', stainId: id };
  if (id >= 5729) return { type: 'item', itemId: id };   // ← ...but not present in dyesByIdMap
  return { type: 'invalid', id };
}
```

## Suggested Fix

In the `/:id` not-found branch, special-case the consolidated IDs:

```ts
import { CONSOLIDATED_IDS } from '@xivdyetools/core';
const consolidatedType = (Object.entries(CONSOLIDATED_IDS)
  .find(([, cid]) => cid === id)?.[0]) as 'A' | 'B' | 'C' | undefined;
if (consolidatedType) {
  throw new ApiError(ErrorCode.NOT_FOUND,
    `ID ${id} is the consolidated market itemID for Type-${consolidatedType} dyes; ` +
    `see /v1/dyes/consolidation-groups.`, 404, { consolidatedType });
}
```

(Or return the consolidation group directly.) Also document in `apps/api-docs` that `marketItemID` is a market-board identifier, not a lookup key.

## Why It's Hidden

Route tests use legacy itemIDs; nobody tests "feed an API output field back into the API". The gap only bites third-party consumers building market integrations — exactly the audience of the public API — who then file confusing 404 reports.

## Status

**FIXED 2026-07-19** — `GET /v1/dyes/:id` special-cases the three consolidated market itemIDs with an explanatory 404 pointing at /v1/dyes/consolidation-groups; api-docs documents `marketItemID` as a market-board identifier, not a lookup key.
