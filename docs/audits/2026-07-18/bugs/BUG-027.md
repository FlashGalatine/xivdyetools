# [BUG-027]: universalis-proxy serves per-origin CORS headers on publicly cacheable responses without `Vary: Origin`

## Severity

MEDIUM

## Type

Cache correctness / CORS

## Location

- `apps/universalis-proxy/src/index.ts:53-87` (CORS middleware — per-origin `Access-Control-Allow-Origin`, no `Vary`)
- `apps/universalis-proxy/src/services/cached-fetch.ts:206-217` (`buildCacheHeaders` → `Cache-Control: public, max-age=...`)

## Description

The CORS middleware emits a **request-dependent** `Access-Control-Allow-Origin`: it echoes the request `Origin` when allowed, otherwise falls back to `allowedOrigins[0]`. Success responses simultaneously carry `Cache-Control: public, max-age=300` (24 h for `/worlds` and `/data-centers`). No `Vary: Origin` header is ever set, so any HTTP cache between the worker and the user is licensed to replay a response — including its `Access-Control-Allow-Origin` — for a request from a **different** origin.

This is a realistic failure because production allows two origins (`https://xivdyetools.app` and `https://xivdyetools.projectgalatine.com`, wrangler.toml `ALLOWED_ORIGINS`). A cached response whose ACAO names one origin, replayed to the other, causes a spurious CORS block — the exact error class this proxy exists to eliminate.

## Reproduction Scenario

1. From `https://xivdyetools.projectgalatine.com`, the budget tool fetches `/api/v2/aggregated/aether/5729` through a shared caching proxy (corporate proxy / any URL-keyed intermediary). Response cached with `Access-Control-Allow-Origin: https://xivdyetools.projectgalatine.com`, `Cache-Control: public, max-age=300`.
2. Within 5 minutes, a user behind the same cache loads the tool on `https://xivdyetools.app` and requests the same URL.
3. The cache replays the stored response; the browser sees an ACAO for the other origin → fetch blocked, market prices fail to load.

## Evidence

```ts
// index.ts:65, 84-86
const corsOrigin = isAllowed ? origin : allowedOrigins[0];   // response varies by request Origin
...
await next();
c.header('Access-Control-Allow-Origin', corsOrigin);         // but no Vary: Origin anywhere
c.header('Access-Control-Allow-Methods', 'GET, OPTIONS');
```

```ts
// cached-fetch.ts:211-216
return {
  'X-Cache': source === 'upstream' ? 'MISS' : 'HIT',
  ...
  'Cache-Control': `public, max-age=${config.cacheTtl}`,     // downstream caches key on URL only
};
```

## Suggested Fix

```ts
// in the CORS middleware, both branches:
// OPTIONS preflight response headers:
'Vary': 'Origin',
// and after next():
c.header('Vary', 'Origin');
```

One header, standard practice, no downside.

## Why It's Hidden

Requires two distinct Origins sharing one HTTP cache. Unit tests, curl checks, and single-origin smoke tests never exercise it; when it fires in the field it looks like a flaky CORS misconfiguration that "fixes itself" when the cache entry expires.

> Source: evidence/edge-workers-analysis.md (2026-07-18 deep-dive, edge-workers area)

## Status

**FIXED 2026-07-19** — `Vary: Origin` emitted on both the preflight response and every post-handler response, so shared caches can no longer replay one allowed origin's ACAO to the other.
