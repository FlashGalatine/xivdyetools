# [BUG-079]: SecureStorage size-index mutex is bypassed by two of three writers (BUG-007 fix incomplete)

## Severity
LOW

## Type
Race condition (lost update on shared index)

## Location
- `apps/web-app/src/services/storage-service.ts:459-495` (mutexed `updateSizeIndex`)
- `apps/web-app/src/services/storage-service.ts:500-505` (`removeFromSizeIndex` — no mutex)
- `apps/web-app/src/services/storage-service.ts:608-651` (`enforceSizeLimit` — no mutex)

## Description
The BUG-007 fix introduced `sizeIndexMutex` so concurrent `setItem` calls serialize their size-index writes. However, `removeFromSizeIndex` and `enforceSizeLimit` load/mutate/save the **same index** synchronously, outside the mutex.

Interleaving: `SecureStorage.setItem(A)` queues an index update on the mutex; before that microtask runs, `SecureStorage.removeItem(A)` executes `removeFromSizeIndex(A)` synchronously; the queued mutex closure then runs and **re-adds** A's entry. The index now tracks a deleted key, permanently inflating `getCachedTotalSize()` and causing premature LRU evictions against the 5 MB limit (`:611-615`). The same lost-update shape exists between `enforceSizeLimit`'s deletions and any queued `updateSizeIndex` continuation.

## Reproduction Scenario
In the same tick: `await SecureStorage.setItem('k', bigValue)` resolves (queuing the index write on the mutex) followed synchronously by `SecureStorage.removeItem('k')`. Inspect the persisted `__secure_storage_size_index__`: it still contains an entry for `'k'` although the key is gone. Repeat to accumulate phantom bytes until eviction fires with plenty of real space free.

## Evidence
```ts
// storage-service.ts:488-495 — mutexed writer
private static updateSizeIndex(key: string, size: number, timestamp: number): void {
  this.sizeIndexMutex = this.sizeIndexMutex.then(() => {
    const index = this.loadSizeIndex();
    index[key] = { size, timestamp };      // may resurrect a just-deleted key
    this.sizeIndexCache = index;
    this.saveSizeIndex();
  });
}

// storage-service.ts:500-505 — synchronous writer, no mutex
private static removeFromSizeIndex(key: string): void {
  const index = this.loadSizeIndex();
  delete index[key];
  this.sizeIndexCache = index;
  this.saveSizeIndex();
}
```

## Suggested Fix
Route all three writers through the same chain:

```ts
private static removeFromSizeIndex(key: string): void {
  this.sizeIndexMutex = this.sizeIndexMutex.then(() => {
    const index = this.loadSizeIndex();
    delete index[key];
    this.sizeIndexCache = index;
    this.saveSizeIndex();
  });
}
```
(and perform `enforceSizeLimit`'s index mutation as a mutex continuation). Alternatively drop the async mutex entirely — every operation involved is synchronous, so a strict synchronous read-modify-write discipline suffices.

## Why It's Hidden
Needs an interleaving of a queued mutex callback with a synchronous writer in the same tick; deterministic tests do not produce it, and the symptom (slightly inflated size accounting) is invisible until LRU eviction misbehaves much later.

## Status

**FIXED 2026-07-19** — `removeFromSizeIndex` and the whole of `enforceSizeLimit` now run as `sizeIndexMutex` continuations, closing the lost-update interleavings with queued `updateSizeIndex` closures.
