# [BUG-051]: Refresh grace-period arithmetic breaks on tokens without `exp`; `isJWTExpired` inverts for the same case

## Severity
LOW

## Type
Edge Case / NaN Propagation (JWT claim optionality divergence)

## Location
- File: `apps/oauth/src/handlers/refresh.ts:82` (grace-period check)
- File: `apps/oauth/src/services/jwt-service.ts:277-316` (`verifyJWTSignatureOnly` requires `sub` but not `exp`), 321-327 (`isJWTExpired`)
- Function: `tokenRouter.post('/refresh')`, `verifyJWTSignatureOnly`, `isJWTExpired`

## Description
In the refresh fallback path, `decoded.exp + gracePeriod < now` with `exp === undefined` evaluates `NaN < now` → `false`, so a validly-signed token with **no** `exp` claim is treated as *within* the 24h grace period and refreshed into a fresh full-lifetime token — repeatable indefinitely. `verifyJWTSignatureOnly` was hardened during REFACTOR-001 alignment to require `sub` but not `exp`, unlike `verifyJWT`, which guards it (`jwt-service.ts:239-241`). Similarly, `isJWTExpired` returns `false` (not expired) when `exp` is missing. Not exploitable without the signing key today — every current mint path sets `exp` — but it silently converts a hypothetical mis-minted eternal token into a self-renewing credential and is a latent divergence between the two verifiers.

## Reproduction Scenario
1. (Requires signing key / a legacy mis-minted token) Craft a token signed with `JWT_SECRET` whose payload has `sub` but no `exp`.
2. `POST /auth/refresh` with it: `verifyJWT` throws ("JWT has expired" — its missing-exp guard), the fallback `verifyJWTSignatureOnly` accepts it, the grace check computes `NaN < now === false` → passes.
3. A brand-new fully-valid token is issued; repeat forever.

## Evidence
```ts
// refresh.ts:80-82
const gracePeriod = 24 * 60 * 60;
if (decoded.exp + gracePeriod < now) {   // NaN < now === false when exp is undefined
```
```ts
// jwt-service.ts:307-311 — requires sub, but not exp
if (!payload.sub) return null;
return payload;
```
```ts
// jwt-service.ts:321-327
export function isJWTExpired(token: string): boolean {
  const payload = decodeJWT(token);
  if (!payload) return true;
  const now = Math.floor(Date.now() / 1000);
  return payload.exp < now;   // undefined < now === false → "not expired"
}
```

## Suggested Fix
```ts
// verifyJWTSignatureOnly — match verifyJWT's claim requirements:
if (!payload.sub || !payload.exp) return null;

// isJWTExpired — fail closed:
return !payload.exp || payload.exp < now;
```

## Why It's Hidden
All tokens the service mints include `exp`, so no test or production flow ever exercises the missing-`exp` branch; the NaN comparison fails silently in the permissive direction rather than throwing.
