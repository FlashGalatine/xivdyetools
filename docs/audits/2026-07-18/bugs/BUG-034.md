# [BUG-034]: getPresetByName fetches limit: 1, defeating its own exact-match logic

## Severity
MEDIUM

## Type
Logic error

## Location
`apps/discord-worker/src/services/preset-api.ts:270-285`

## Description
`getPresetByName` requests a single search result from the presets API, then attempts exact-match-then-partial selection over that result set. With `limit: 1`, `response.presets` has at most one element — chosen by the API's search **ranking**, not by exactness. If a query for "Rose" ranks "Rose Gold Dream" first, the preset literally named "Rose" is unreachable through this path; the wrong preset is returned silently, with no error for the user to notice.

## Reproduction Scenario
1. Have two approved presets: "Rose Gold Dream" (popular) and "Rose".
2. Trigger any flow that resolves a preset by typed name via `getPresetByName` (e.g. `/preset show` with a typed name rather than an autocomplete-selected ID).
3. The higher-ranked partial match is returned instead of the exact-named preset.

## Evidence
```ts
// apps/discord-worker/src/services/preset-api.ts:274-285
const response = await getPresets(env, {
  search: name,
  status: 'approved',
  limit: 1,          // <-- only one candidate ever comes back
});

// Find exact match first, then partial match
const exactMatch = response.presets.find(
  (p) => p.name.toLowerCase() === name.toLowerCase()
);
return exactMatch || response.presets[0] || null;
```

## Suggested Fix
Fetch one full page and keep the existing selection logic:
```ts
const response = await getPresets(env, { search: name, status: 'approved', limit: 25 });
```
Longer term, add an exact-name lookup parameter/endpoint to presets-api so this client-side scan is unnecessary.

## Why It's Hidden
Tests mock `getPresets` to return exactly the preset the test expects, so the `find()` always succeeds. The ranking-vs-exactness interaction only appears with a realistic search backend and colliding preset names — and even then the failure mode is "plausible-looking wrong preset", not an error.
