# [BUG-005]: ColorManipulator mutates HSV objects returned by reference from LRU caches (cache poisoning)

## Severity
HIGH

## Type
State management / shared mutable cache references

## Location
- `packages/core/src/services/color/ColorManipulator.ts:20-24, 30-34, 40-45` (mutation site)
- `packages/core/src/services/color/ColorConverter.ts:401-404, 407-409` (`hexToHsv` returns/stores live references)
- `packages/core/src/services/color/ColorConverter.ts:285-288` (same object also stored in `rgbToHsvCache`)

## Description
`ColorConverter.hexToHsv()` returns the *cached object itself* (no copy), and the same object instance is stored in **two** caches: `rgbToHsvCache` (created inside `rgbToHsv`) and `hexToHsvCache`. `ColorManipulator.adjustBrightness`, `adjustSaturation`, and `rotateHue` mutate that returned object in place (`hsv.v = ...`, `hsv.s = ...`, `hsv.h = ...`).

After one `adjustBrightness('#336699', 20)` call, both `hexToHsv('#336699')` and `rgbToHsv(51, 102, 153)` permanently return the *brightened* HSV to every caller sharing the default singleton converter. Repeated calls compound (non-idempotent: second `adjustBrightness(..., 20)` on the same hex effectively applies +40).

The same hazard class applies to every conversion cache (`hexToRgb`, `hsvToRgb`, `rgbToLab`, `rgbToOklab`, and `ColorblindnessSimulator.colorblindCache`): any consumer mutating a returned `RGB`/`HSV`/`LAB` object poisons a process-wide cache. `DyeSearch`, `HarmonyGenerator`, the web app, and the SVG package all funnel through these singleton caches.

## Reproduction Scenario
```ts
ColorService.adjustBrightness('#336699', 20);
ColorService.hexToHsv('#336699');              // v is +20 off — wrong for every future caller
ColorService.adjustBrightness('#336699', 20);  // compounds: now effectively +40
```

## Evidence
```ts
// ColorManipulator.ts:20-24
static adjustBrightness(hex: string, amount: number): HexColor {
  const hsv = ColorConverter.hexToHsv(hex);
  hsv.v = clamp(hsv.v + amount, 0, 100);   // ← mutates the cached object
  return ColorConverter.hsvToHex(hsv.h, hsv.s, hsv.v);
}
```
```ts
// ColorConverter.ts:401-409 — returns cached reference, no copy;
// same object also lives in rgbToHsvCache (stored at line 287)
const cached = this.hexToHsvCache.get(cacheKey);
if (cached) {
  return cached;
}
const rgb = this.hexToRgb(hex);
const result = this.rgbToHsv(rgb.r, rgb.g, rgb.b);
this.hexToHsvCache.set(cacheKey, result);
```

## Suggested Fix
Short-term, safest: return defensive copies on cache hits *and* on first computation in `hexToRgb`, `rgbToHsv`, `hsvToRgb`, `hexToHsv`, `rgbToLab`, `rgbToOklab`, and `ColorblindnessSimulator.simulateColorblindness`:
```ts
if (cached) return { ...cached };
...
this.hexToHsvCache.set(cacheKey, result);
return { ...result };
```
Additionally fix `ColorManipulator` to build a new object instead of mutating:
```ts
const { h, s, v } = ColorConverter.hexToHsv(hex);
return ColorConverter.hsvToHex(h, s, clamp(v + amount, 0, 100));
```
Optionally `Object.freeze()` cached values in dev builds to surface any other offenders. Copy-on-return costs one small allocation per call — negligible against the conversion math it caches.

## Why It's Hidden
Unit tests call each manipulation once on fresh colors and assert the return value — which is correct. Corruption only shows when the *same hex* is later converted again in the same process (long-lived browser session, warm Cloudflare Worker isolate), and it manifests as subtly wrong downstream colors, not an exception.
