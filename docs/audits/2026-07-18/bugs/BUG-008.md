# [BUG-008]: Default `wrangler deploy` overwrites the production worker with development vars (all three edge workers)

## Severity

HIGH

## Type

Deployment configuration hazard (env name collision + prod routes in default env)

## Location

- `apps/universalis-proxy/wrangler.toml:1-26` + `apps/universalis-proxy/package.json` (`"deploy": "wrangler deploy"`)
- `apps/api-worker/wrangler.toml:1-28` + `apps/api-worker/package.json`
- `apps/og-worker/wrangler.toml:1-37` + `apps/og-worker/package.json`

## Description

In each of the three workers, `[env.production].name` is **identical** to the top-level `name`, and the top-level (default) environment carries the **production routes / custom domains** but **development vars**. `pnpm run deploy` — documented in each CLAUDE.md as "deploy to staging" — therefore deploys to the *same* production worker and its custom domains, replacing production configuration with dev configuration:

- **universalis-proxy:** top-level has `ENVIRONMENT="development"`, `ALLOWED_ORIGINS="http://localhost:5173,..."`, `RATE_LIMIT_REQUESTS="60"` — while routes bind `proxy.xivdyetools.app` and `proxy.xivdyetools.projectgalatine.com`. After a default deploy, the dev CORS branch (`src/index.ts:59-61`) allows any localhost origin against production, and browser calls from `https://xivdyetools.app` receive `Access-Control-Allow-Origin: http://localhost:5173` (fallback `allowedOrigins[0]`) → the production web-app's market-board features break outright.
- **api-worker:** top-level `ENVIRONMENT="development"` enables verbose 500 bodies with `stack` (`src/index.ts:159-168`) and disables HSTS, and binds `RATE_LIMIT` to the **dev KV namespace** (`8f2028...` vs prod `a57d95...`) — all on `data.xivdyetools.app`.
- **og-worker:** same name collision; top-level vars happen to equal production values so the damage is limited, but "staging deploy" is still silently a production deploy.

## Reproduction Scenario

1. `pnpm --filter xivdyetools-universalis-proxy run deploy` (no `--env`).
2. `curl https://proxy.xivdyetools.app/` → health JSON reports `"environment": "development"`.
3. Open `https://xivdyetools.app` budget tool → all proxy fetches fail CORS (`Access-Control-Allow-Origin: http://localhost:5173`).

## Evidence

```toml
# apps/universalis-proxy/wrangler.toml
name = "xivdyetools-universalis-proxy"
routes = [
  { pattern = "proxy.xivdyetools.app", custom_domain = true },
  { pattern = "proxy.xivdyetools.projectgalatine.com", custom_domain = true }
]
[vars]
ENVIRONMENT = "development"
ALLOWED_ORIGINS = "http://localhost:5173,http://localhost:3000,http://127.0.0.1:5173"
...
[env.production]
name = "xivdyetools-universalis-proxy"   # ← SAME worker name as the default env
```

```json
// each worker's package.json
"deploy": "wrangler deploy",
"deploy:production": "wrangler deploy --env production",
```

## Suggested Fix

Make the default env a genuinely separate target, per worker:

```toml
name = "xivdyetools-universalis-proxy-dev"   # distinct name; gets a workers.dev URL
# no routes at top level

[env.production]
name = "xivdyetools-universalis-proxy"
routes = [
  { pattern = "proxy.xivdyetools.app", custom_domain = true },
  { pattern = "proxy.xivdyetools.projectgalatine.com", custom_domain = true }
]
```

Move routes/custom domains under `[env.production]` only (repeat for api-worker and og-worker). Alternatively, drop the default deploy target entirely and point `"deploy"` at a dedicated `[env.staging]`.

## Why It's Hidden

- No test exercises wrangler configuration; CI deploy workflows always pass `--env production`, so the hazard only fires on a **manual local** `pnpm deploy` — exactly the "quick staging check" the scripts and CLAUDE.md deployment checklists invite.
- The deploy succeeds with no warning (same worker name = silent overwrite); breakage appears minutes later as production CORS/ratelimit anomalies with no obvious link to the "staging" deploy.

> Source: evidence/edge-workers-analysis.md (2026-07-18 deep-dive, edge-workers area)

## Status

**FIXED 2026-07-18** — all three edge workers now use a distinct `-dev`-suffixed top-level worker name with NO routes; production name, routes/custom domains, vars, and per-env bindings (api-worker prod KV, og-worker analytics) live exclusively under `[env.production]`. A plain `wrangler deploy` now targets a separate dev worker on workers.dev and cannot overwrite production. Validated with `wrangler deploy --dry-run` on all six targets (default + production per worker); CI workflows already pass `--env production` and are unaffected.
