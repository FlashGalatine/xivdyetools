# [BUG-050]: moderation_log 7-day stats compare mismatched datetime formats

## Severity
LOW

## Type
Edge Case / Data Format Mismatch (SQLite TEXT datetime comparison)

## Location
- File: `apps/presets-api/src/handlers/moderation.ts`
- Line(s): 192 (stats query) vs 75, 139 (writers use `new Date().toISOString()`)
- Function: `moderationRouter.get('/stats')`

## Description
`moderation_log` rows are written with JS ISO format `2026-07-18T12:34:56.789Z`; the stats query compares against SQLite's `datetime('now','-7 days')`, which renders `2026-07-11 12:34:56` (space separator, no `Z`). SQLite compares TEXT lexicographically: for rows dated exactly on the boundary day, `'T' (0x54) > ' ' (0x20)` makes every ISO timestamp on that calendar day sort as "after" the cutoff even when its time-of-day is earlier — `actions_last_week` over-counts by up to one day's worth of boundary-day actions. Note `presets.created_at` written by `createPreset` uses the same ISO format, so any future `datetime('now')` comparisons against presets inherit the same hazard.

## Reproduction Scenario
1. Insert a `moderation_log` row at `2026-07-11T02:00:00.000Z` (via a moderation action).
2. On `2026-07-18` at 12:00 UTC, call `GET /api/v1/moderation/stats`.
3. The cutoff is `2026-07-11 12:00:00`; the 02:00 row (older than 7 days) still counts because `"2026-07-11T02..." > "2026-07-11 12..."` lexicographically.

## Evidence
```ts
// moderation.ts:192
(SELECT COUNT(*) FROM moderation_log WHERE created_at > datetime('now', '-7 days')) as actions_last_week
```
```ts
// moderation.ts:75 — writer
const now = new Date().toISOString(); // "2026-07-18T12:34:56.789Z"
```

## Suggested Fix
Use one format on both sides. Simplest: compare with an ISO-formatted cutoff:
```sql
... WHERE created_at > strftime('%Y-%m-%dT%H:%M:%fZ', 'now', '-7 days')
```
or stop passing JS timestamps and rely on the column's `DEFAULT (datetime('now'))` consistently.

## Why It's Hidden
Both formats sort correctly against each other whenever the date components differ, so the error is confined to the single boundary day and is a small over-count in a moderator-facing statistic — invisible unless someone reconciles the number by hand.
