# [BUG-019]: Vote insert and vote_count update are not atomic — counter drift on partial failure

## Severity
MEDIUM

## Type
Race Condition / Missing Transaction (D1 multi-statement mutation)

## Location
- File: `apps/presets-api/src/handlers/votes.ts`
- Line(s): 33-77 (`addVote`), 91-131 (`removeVote`)
- Also: `apps/presets-api/src/handlers/presets.ts:479` (auto-vote after create uses the same path)
- Function: `addVote`, `removeVote`

## Description
`addVote` runs two independent statements: `INSERT ... ON CONFLICT DO NOTHING` then `UPDATE presets SET vote_count = vote_count + 1`. The TOCTOU on *duplicate* votes is correctly solved by the conflict clause, but the pair is not batched: if the worker is evicted, the request is cancelled, or D1 errors between the two statements, a `votes` row exists whose increment never landed (or, for `removeVote`, a deleted row whose decrement never landed). Drift is permanent — nothing reconciles `vote_count` with `COUNT(votes)`, and `sort=popular` ordering silently degrades over time. `D1Database.batch()` is already used in the sibling delete handler (`presets.ts:201-204`), so the primitive is available.

## Reproduction Scenario
Not deterministically reproducible — requires a failure between statements (worker CPU-limit kill, D1 transient error, isolate eviction). Over months of traffic the denormalized counter skews from the `votes` table truth; visible as presets whose displayed count disagrees with `SELECT COUNT(*) FROM votes WHERE preset_id = ?`.

## Evidence
```ts
// votes.ts:36-41 then 59-64 — two separate round trips, no batch
const insertResult = await db
  .prepare('INSERT INTO votes (preset_id, user_discord_id, created_at) VALUES (?, ?, ?) ON CONFLICT DO NOTHING')
  .bind(presetId, userDiscordId, now).run();
...
const updateResult = await db
  .prepare('UPDATE presets SET vote_count = vote_count + 1, updated_at = ? WHERE id = ? RETURNING vote_count')
  .bind(now, presetId).first<{ vote_count: number }>();
```

## Suggested Fix
Make the counter update recompute truth and run both statements in one atomic batch (also self-heals prior drift):
```ts
const [_, updated] = await db.batch([
  db.prepare('INSERT INTO votes (preset_id, user_discord_id, created_at) VALUES (?, ?, ?) ON CONFLICT DO NOTHING')
    .bind(presetId, userDiscordId, now),
  db.prepare(`UPDATE presets
              SET vote_count = (SELECT COUNT(*) FROM votes WHERE preset_id = ?), updated_at = ?
              WHERE id = ? RETURNING vote_count`)
    .bind(presetId, now, presetId),
]);
```
Apply the mirrored change in `removeVote`.

## Why It's Hidden
Both statements succeed in every test run and in the vast majority of production requests; drift needs an induced partial failure, and once it happens there is no error — just a slightly wrong number that nothing cross-checks.

## Status

**FIXED 2026-07-19** — `addVote`/`removeVote` run the vote mutation and counter update in one `db.batch()`, with the counter recomputed as `(SELECT COUNT(*) FROM votes WHERE preset_id = ?)` so prior drift self-heals. The shared D1 mock's `run()`/`batch()` were updated to honor RETURNING rows like real D1.
