# [BUG-052]: `previous_values` audit snapshot is overwritten on each flagged edit despite "append-only audit log" comment

## Severity
LOW

## Type
Logic Error / Code-Comment Contract Mismatch (audit-trail loss)

## Location
- File: `apps/presets-api/src/handlers/presets.ts`
- Line(s): 296-311 (flagged-edit snapshot + PRESETS-CRITICAL-004 comment)
- Also: `apps/presets-api/src/services/preset-service.ts:417-420` (overwrite), 458-463 (revert clears the column)
- Function: `presetsRouter.patch('/:id')`, `updatePreset`, `revertPreset`

## Description
The comment block (presets.ts:306-310, "PRESETS-CRITICAL-004 ... The previous_values field now serves as an append-only audit log") claims audit history is preserved for compliance and pattern detection. In reality the column holds a single JSON object: each new flagged edit *replaces* `previous_values` with only the immediately-prior state, and a moderator revert nulls the column entirely. Two successive flagged edits leave only the middle state recoverable — the original content is gone. The code implements "last-known-good snapshot for revert" semantics, not the documented append-only audit semantics; one of the two is wrong, and moderators relying on the documented behavior will make incorrect assumptions about what can be recovered/reviewed.

## Reproduction Scenario
1. Create preset with name `Original` (approved).
2. Edit to a profane name `BadWord1` → flagged; `previous_values` = snapshot of `Original`. ✔
3. While pending, edit again to `BadWord2` → flagged again; `previous_values` is **overwritten** with the `BadWord1` snapshot. The `Original` state is unrecoverable.
4. Moderator revert restores `BadWord1` (itself flagged content), and the column is set to NULL — no history remains.

## Evidence
```ts
// presets.ts:306-310 — the claim
// PRESETS-CRITICAL-004: Do NOT clear previous_values when moderation passes
// ... The previous_values field now serves as an append-only audit log
```
```ts
// preset-service.ts:417-420 — the overwrite
if (previousValues !== undefined) {
  setClauses.push('previous_values = ?');
  params.push(previousValues ? JSON.stringify(previousValues) : null);
}
```

## Suggested Fix
Either fix the comment (snapshot semantics are arguably fine for the revert feature), or make it actually append-only: only write `previous_values` when it is currently NULL (preserve the oldest clean state), e.g. in the handler: `if (!preset.previous_values) previousValues = {...}` — or persist superseded snapshots into `moderation_log` before overwriting.

## Why It's Hidden
Tests cover a single flagged edit + revert; the double-flagged-edit sequence that loses history is never exercised, and the misleading comment steers reviewers away from questioning the overwrite.

## Status

**FIXED 2026-07-19** — Chose snapshot semantics and made them write-once: a flagged edit only captures `previous_values` when the column is empty, so successive flagged edits can't overwrite the oldest known-good state. The misleading append-only comment was corrected.
