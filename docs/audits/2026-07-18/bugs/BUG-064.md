# [BUG-064]: KVRateLimiter `check()` can read one window and increment the next across a window boundary

## Severity
LOW

## Type
Race condition / time-of-check-time-of-use across a fixed-window boundary

## Location
`packages/rate-limiter/src/backends/kv.ts:115-123` (`check()` composing the two ops), `:128-136` (`checkOnly` — `now` at T1), `:194-197` (`increment` — fresh `now` at T2), key derivation at `:301-304`

## Description
`check()` calls `checkOnly()` then `increment()` as separate operations, and **each takes its own `Date.now()`**. The KV key embeds the fixed window number (`Math.floor(timestamp / windowMs)`), so if the window boundary falls between T1 (checkOnly's read) and T2 (increment's write) — a gap that includes a full KV read round-trip, typically tens of milliseconds — the two operations address **different KV keys**:

- The allow/deny decision and the returned `remaining`/`resetAt` describe window N.
- The recorded request lands in window N+1 as its first entry (`count: 1`).

Effects: the reported headers describe a window that was never charged; the new window starts pre-consumed by a request that was actually admitted under the old window's budget; and near boundaries the effective count in either window can be off by the number of straddling requests. It self-corrects within one window and cannot be exploited deliberately at scale, but it is essentially free to fix.

## Reproduction Scenario
`windowMs = 60_000`; window boundary at t=120.000s.
1. Request arrives; `checkOnly()` runs at t=119.990s → key `…|1`, count 5 of 10 → allowed, `remaining: 5`, `resetAt: 120s`.
2. KV read latency pushes `increment()` to t=120.010s → key `…|2` → writes `{count: 1}`.
3. Result: window 1 never recorded the request (its final count stays 5); window 2 opens with 1 already consumed; the response's `remaining`/`resetAt` referred to window 1.

## Evidence
```ts
// packages/rate-limiter/src/backends/kv.ts:115-123
async check(key: string, config: RateLimitConfig): Promise<RateLimitResult> {
  const result = await this.checkOnly(key, config);   // Date.now() #1 → window N key
  if (result.allowed && !result.backendError) {
    await this.increment(key, config);                // Date.now() #2 → may be window N+1 key
    result.remaining = Math.max(0, result.remaining - 1);
  }
  return result;
}

// :301-304 — key embeds the window number derived from the timestamp
private buildKey(key: string, timestamp: number, windowMs: number): string {
  const window = Math.floor(timestamp / windowMs);
  return `${this.keyPrefix}${key}|${window}`;
}
```

## Suggested Fix
Capture `now` once in `check()` and thread it through both operations:

```ts
async check(key: string, config: RateLimitConfig): Promise<RateLimitResult> {
  const now = Date.now();
  const result = await this.checkOnly(key, config, now);
  if (result.allowed && !result.backendError) {
    await this.increment(key, config, now);           // same window key guaranteed
    result.remaining = Math.max(0, result.remaining - 1);
  }
  return result;
}
// checkOnly(key, config, now = Date.now()) / increment(key, config, now = Date.now())
// — optional trailing parameter keeps the public API backward compatible.
```

## Why It's Hidden
The straddle window is tens of milliseconds once per `windowMs` per key — a tiny hit probability that unit tests (fake or fast clocks, no KV latency) never land in. When it does occur, the symptom is a one-off counter discrepancy at a window edge, indistinguishable in logs from the backend's other documented non-atomicity (see BUG-022), so it is written off as "KV being KV."

## Status

**FIXED 2026-07-19** — `check()` captures `Date.now()` once and threads it through `checkOnly`/`increment` (optional trailing parameter, backward compatible), so both operations always address the same fixed-window KV key.
