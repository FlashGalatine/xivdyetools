# [BUG-012]: fetchBatchPriceData — invalid-ID validation throw escapes the never-throw contract

## Severity
MEDIUM

## Type
Error-handling gap / unguarded throw path

## Location
- `packages/core/src/services/APIService.ts:667` (URL build outside the try starting at `:669`)
- `packages/core/src/services/APIService.ts:896-904` (`buildBatchApiUrl` validation throws)
- Callers without catch: `packages/core/src/services/APIService.ts:989` (`getPricesForItems`), `:1048` (`getPricesForDataCenter`)

## Description
`fetchBatchPriceData` is designed as a never-throw method — fetch/parse failures return an empty `Map` (`:683-688`), and the prior audit's BUG-001 fix (chunking at `:646-660`) removed the >100-item throw. But the URL builder is still invoked **before** the try block, and `buildBatchApiUrl` throws `AppError(INVALID_INPUT)` for any non-positive-integer ID.

So `getPricesForItems([...ids, -1477])` — a Facewear synthetic negative ID leaking past a missing `itemID > 0` filter, exactly the mistake already made once in the Discord budget command (fixed there at the *caller* level) — produces an **uncaught AppError** from a code path callers reasonably treat as non-throwing, instead of "no price for that item". The comment at `:666` ("safe: itemIDs.length ≤ 100 guaranteed above") is accurate but incomplete: length is not the only reason `buildBatchApiUrl` throws.

## Reproduction Scenario
```ts
await apiService.getPricesForItems([5729, -1477]); // -1477 uncached
// → throws AppError: "Invalid item IDs: all IDs must be positive integers (invalid: -1477)"
//   instead of returning the price for 5729
```

## Suggested Fix
Defense-in-depth at the library layer — filter and warn before building the URL:
```ts
const validIDs = itemIDs.filter((id) => Number.isInteger(id) && id > 0);
if (validIDs.length < itemIDs.length) {
  this.logger.warn(`Skipping ${itemIDs.length - validIDs.length} invalid item IDs in batch fetch`);
}
if (validIDs.length === 0) return new Map();
```
(or, minimally, move the `buildBatchApiUrl` call inside the try block so the existing catch returns an empty `Map`).

## Why It's Hidden
Tests pass valid positive IDs; the cached path never reaches URL building. It takes a cold cache **plus** an unfiltered synthetic/zero ID to trigger — the same latent-ID class of bug that previously shipped in the budget command.
