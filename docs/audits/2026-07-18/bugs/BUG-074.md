# [BUG-074]: Webhook endpoints ignore Discord API outcomes; no app.onError in discord-worker; changelog fetch has no timeout

## Severity
LOW

## Type
Error-handling gap / resource hang risk

## Location
- `apps/discord-worker/src/index.ts:192` and `:240` (`await sendMessage(...)` — Response discarded)
- `apps/discord-worker/src/index.ts:341` (`fetch(changelogUrl)` — no `AbortSignal`)
- `apps/discord-worker/src/index.ts` (no `app.onError` anywhere) vs. `apps/moderation-worker/src/index.ts:515-530` (hardened sibling)

## Description
Three small gaps in the non-interaction HTTP surface of discord-worker:

1. `/webhooks/preset-submission` returns `{ success: true }` to presets-api even when `sendMessage` came back 403/404 (bad channel ID, missing bot permission) — the moderation notification is lost with no signal to the caller and no log entry.
2. When `sendMessage` *throws* (its 5 s `AbortSignal.timeout`, `discord-api.ts:272`), the route has no try/catch and discord-worker registers no `app.onError`, so Hono's default 500 handler responds — inconsistent with moderation-worker, which has an explicit `onError` that logs and shapes the response (SEC-001 there).
3. `/webhooks/github`'s `fetch(changelogUrl)` is the only outbound fetch in the worker without a timeout; a hung raw.githubusercontent.com response holds the request open until the platform kills it.

## Reproduction Scenario
1. Point `MODERATION_CHANNEL_ID` at a channel the bot cannot post in; submit a web preset. presets-api receives `{ success: true }`; no embed appears; nothing is logged.
2. Simulate Discord latency > 5 s on the same path: the timeout throws, and the webhook caller gets an unshaped Hono 500.
3. Stall the changelog fetch: the GitHub webhook request hangs.

## Evidence
```ts
// apps/discord-worker/src/index.ts:192 — outcome ignored, route later returns success unconditionally
await sendMessage(env.DISCORD_TOKEN, env.MODERATION_CHANNEL_ID, { embeds: [...], components: [...] });
...
return c.json({ success: true });   // :260
```
```ts
// apps/discord-worker/src/index.ts:341 — no timeout
const changelogResponse = await fetch(changelogUrl);
```

## Suggested Fix
- Check `.ok` on both `sendMessage` calls; on failure log status/body and return 502 so presets-api can retry or alert.
- Add `signal: AbortSignal.timeout(10_000)` to the changelog fetch.
- Add the same `app.onError` handler moderation-worker uses (log via the request logger, return a shaped 500, dev-only detail).

## Why It's Hidden
Webhook paths are exercised far less than interactions and almost always succeed; a lost moderation notification just looks like "no submissions today". The 500-shape inconsistency and the missing timeout only appear during Discord/GitHub degradation windows.
