# [BUG-002]: CJK profanity lists can never match — `\b` word boundaries don't exist around CJK characters

## Severity
HIGH

## Type
Logic Error / Dead Code Path (content moderation)

## Location
- File: `apps/presets-api/src/services/moderation-service.ts`
- Line(s): 85 (pattern construction), 151-172 (`checkLocalFilter`)
- Word data: `apps/presets-api/src/data/profanity/{ja,ko,zh}.ts`
- Function: `compileProfanityPatterns`, `checkLocalFilter`

## Description
The combined filter regex is `` new RegExp(`\\b(${escapedWords.join('|')})\\b`, 'i') ``. JavaScript `\b` is defined in terms of `\w` = `[A-Za-z0-9_]`. CJK characters are non-word characters, so there is never a word boundary between a CJK character and an adjacent non-word character (another CJK char, a space, or start/end of string). Any list entry that ends (or begins) with a CJK character — every entry in `ja.ts` (e.g. `'aiのガラクタ'` ends in `タ`), `ko.ts` (`'ai 쓰레기'` ends in `기`), and `zh.ts` — can never satisfy the trailing `\b` in real text. The Japanese/Korean/Chinese local profanity lists are effectively dead code; only Perspective API (optional, fail-open on timeout or when `PERSPECTIVE_API_KEY` is unset) covers those languages.

## Reproduction Scenario
1. Ensure `PERSPECTIVE_API_KEY` is unset (or Perspective times out — the code fails open).
2. Submit a preset whose name is exactly a listed Korean entry, e.g. `ai 쓰레기`.
3. `checkLocalFilter('ai 쓰레기', ...)` returns `null` (clean) because `/\b(ai 쓰레기)\b/i` cannot match — the char after `기` (space/end) is non-word, so no boundary exists.
4. The preset is auto-approved despite containing a listed blocked term.

## Evidence
```ts
// moderation-service.ts:85
combinedPattern = new RegExp(`\\b(${escapedWords.join('|')})\\b`, 'i');
```
```ts
// ja.ts:13 — 'aiのガラクタ'; /\b(aiのガラクタ)\b/i.test('これはaiのガラクタです') === false
// ko.ts:13 — 'ai 쓰레기';   /\b(ai 쓰레기)\b/i.test('ai 쓰레기 preset') === false
```

## Suggested Fix
Split compiled data into two matchers: ASCII-ish words keep `\b(...)\b`; entries containing any character outside `[\w\s]` use a boundary-less alternation:
```ts
const [ascii, cjk] = partition(allWords, (w) => /^[\w\s'-]+$/.test(w));
const asciiPattern = ascii.length ? new RegExp(`\\b(${ascii.map(escapeRegex).join('|')})\\b`, 'i') : null;
const cjkPattern = cjk.length ? new RegExp(cjk.map(escapeRegex).join('|'), 'i') : null;
```
Alternatively use Unicode-aware boundaries: `(?<![\p{L}\p{N}])(...)(?![\p{L}\p{N}])` with the `u` flag.

## Why It's Hidden
`tests/data/profanity.test.ts` and the moderation tests inject/assert ASCII words (which do have `\w` neighbors, so `\b` works). No test asserts that a CJK list entry actually matches through the compiled combined pattern, so the filter passes all tests while silently ignoring three languages.

> Source: evidence/d1-workers-analysis.md (2026-07-18 deep-dive, d1-workers area)

## Status

**FIXED 2026-07-19** — `compileProfanityPatterns` now splits words by script: ASCII-ish words keep the ``-anchored pattern; words containing characters outside `[\w\s'-]` (all ja/ko/zh entries) compile into a boundary-less `cjkPattern` tested alongside it in `checkLocalFilter`.
