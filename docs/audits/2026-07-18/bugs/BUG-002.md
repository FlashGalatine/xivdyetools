# [BUG-002]: CJK profanity lists can never match вЂ” `\b` word boundaries don't exist around CJK characters

## Severity
HIGH

## Type
Logic Error / Dead Code Path (content moderation)

## Location
- File: `apps/presets-api/src/services/moderation-service.ts`
- Line(s): 85 (pattern construction), 151-172 (`checkLocalFilter`)
- Word data: `apps/presets-api/src/data/profanity/{ja,ko,zh}.ts`
- Function: `compileProfanityPatterns`, `checkLocalFilter`

## Description
The combined filter regex is `` new RegExp(`\\b(${escapedWords.join('|')})\\b`, 'i') ``. JavaScript `\b` is defined in terms of `\w` = `[A-Za-z0-9_]`. CJK characters are non-word characters, so there is never a word boundary between a CJK character and an adjacent non-word character (another CJK char, a space, or start/end of string). Any list entry that ends (or begins) with a CJK character вЂ” every entry in `ja.ts` (e.g. `'aiгЃ®г‚¬гѓ©г‚Їг‚ї'` ends in `г‚ї`), `ko.ts` (`'ai м“°л €кё°'` ends in `кё°`), and `zh.ts` вЂ” can never satisfy the trailing `\b` in real text. The Japanese/Korean/Chinese local profanity lists are effectively dead code; only Perspective API (optional, fail-open on timeout or when `PERSPECTIVE_API_KEY` is unset) covers those languages.

## Reproduction Scenario
1. Ensure `PERSPECTIVE_API_KEY` is unset (or Perspective times out вЂ” the code fails open).
2. Submit a preset whose name is exactly a listed Korean entry, e.g. `ai м“°л €кё°`.
3. `checkLocalFilter('ai м“°л €кё°', ...)` returns `null` (clean) because `/\b(ai м“°л €кё°)\b/i` cannot match вЂ” the char after `кё°` (space/end) is non-word, so no boundary exists.
4. The preset is auto-approved despite containing a listed blocked term.

## Evidence
```ts
// moderation-service.ts:85

```
```ts
// ja.ts:13 вЂ” 'aiгЃ®г‚¬гѓ©г‚Їг‚ї'; /\b(aiгЃ®г‚¬гѓ©г‚Їг‚ї)\b/i.test('гЃ“г‚ЊгЃЇaiгЃ®г‚¬гѓ©г‚Їг‚їгЃ§гЃ™') === false
// ko.ts:13 вЂ” 'ai м“°л €кё°';   /\b(ai м“°л €кё°)\b/i.test('ai м“°л €кё° preset') === false
```

## Suggested Fix
Split compiled data into two matchers: ASCII-ish words keep `\b(...)\b`; entries containing any character outside `[\w\s]` use a boundary-less alternation:
```ts
const [ascii, cjk] = partition(allWords, (w) => /^[\w\s'-]+$/.test(w));
const asciiPattern = ascii.length ? new RegExp(`\\b(${ascii.map(escapeRegex).join('|')})\\b`, 'i') : null;
const cjkPattern = cjk.length ? new RegExp(cjk.map(escapeRegex).join('|'), 'i') : null;
```
Alternatively use Unicode-aware boundaries: `(?<![\p{L}\p{N}])(...)(?![\p{L}\p{N}])` with the `u` flag.

## Why It's Hidden
`tests/data/profanity.test.ts` and the moderation tests inject/assert ASCII words (which do have `\w` neighbors, so `\b` works). No test asserts that a CJK list entry actually matches through the compiled combined pattern, so the filter passes all tests while silently ignoring three languages.
