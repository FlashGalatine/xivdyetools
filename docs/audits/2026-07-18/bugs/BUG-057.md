# [BUG-057]: `JWTPayload.type` (`'access' | 'refresh'`) declared but never validated anywhere

## Severity
LOW

## Type
Latent security gap / type-reality mismatch (token-confusion risk)

## Location
- `packages/auth/src/jwt.ts:35-36` (declaration), `:153-176` (`verifyJWT` — no `type` check)
- Consumer that ignores it: `apps/presets-api/src/middleware/auth.ts:29` (optional in `ExtendedJWTPayload`), `:39-49` (never checked)
- Issuer that never emits it: `apps/oauth/src/services/jwt-service.ts:99-110, 158-178` (payloads contain no `type` claim)

## Description
The shared `JWTPayload` interface models a `type: 'access' | 'refresh'` discriminator ("Token type: 'access' or 'refresh'"), but `verifyJWT` accepts any signed, unexpired token carrying a `sub` — it never inspects `type`. No consumer checks it either. Two problems follow:

1. **Type-reality mismatch:** the oauth issuer's payloads have no `type` claim at all, so the shared interface misdescribes every real token. Code that reads `payload.type` gets `undefined` despite the non-optional type declaration.
2. **Latent token confusion:** the field and `verifyJWTSignatureOnly`'s refresh-token docstring (jwt.ts:179-198) clearly anticipate long-lived refresh JWTs signed with the same secret. If those are ever introduced, every existing `verifyJWT` call site (e.g. presets-api web auth) will accept a refresh token as an access token — a lifetime-escalation bug that requires zero attacker capability beyond possessing their own refresh token.

## Reproduction Scenario
(Latent — requires refresh-JWT issuance to exist.) Mint a token with `type: 'refresh'`, `exp` = 30 days out, signed with `JWT_SECRET`. Present it as `Authorization: Bearer <token>` to presets-api → `sharedVerifyJWT` returns the payload (signature ok, exp ok, sub ok, `type` ignored) → full web-auth access for 30 days.

## Evidence
```ts
// packages/auth/src/jwt.ts:34-36 — declared…
/** Token type: 'access' or 'refresh' */
type: 'access' | 'refresh';

// :153-176 — …never enforced
export async function verifyJWT(token: string, secret: string): Promise<JWTPayload | null> {
  const payload = await verifyJWTSignature(token, secret);
  if (!payload) return null;
  if (!payload.exp || payload.exp < now) return null;
  if (!payload.sub) return null;
  return payload;                       // no payload.type check
}
```
```ts
// apps/oauth/src/services/jwt-service.ts:99-110 — issuer emits no `type` at all
const payload: JWTPayload = { sub, iat, exp, iss, jti, username, ... };
```

## Suggested Fix
Pick one:
1. **Honest typing (today):** make `type` optional (`type?: 'access' | 'refresh'`) or remove it until real tokens carry it.
2. **Enforcement (forward-proof):** add an option and enforce when present:
```ts
export async function verifyJWT(token: string, secret: string,
  opts?: { expectedType?: 'access' | 'refresh' }): Promise<JWTPayload | null> {
  ...
  if (opts?.expectedType && payload.type !== opts.expectedType) return null;
  ...
}
```
Document the invariant either way so refresh-JWT work can't land without tripping over it.

## Why It's Hidden
Nothing is functionally wrong today because refresh JWTs don't exist — every token in circulation is an access token. TypeScript's structural checks don't catch it: verification code parses JSON and casts, so the missing claim never fails a compile or a test. The trap only springs when someone implements the refresh flow the interface already advertises, likely far from this code.

## Status

**FIXED 2026-07-19** — `JWTPayload.type` is now optional (honest typing: no issuer emits it), and `verifyJWT` accepts `{ expectedType }` which fails closed on mismatch — the enforcement hook refresh-JWT work must use.
