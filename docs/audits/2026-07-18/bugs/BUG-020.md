# [BUG-020]: Moderation status change — audit log written before (and regardless of) the update; action derived from stale status

## Severity
MEDIUM

## Type
Race Condition / TOCTOU + Missing Transaction (moderation flow)

## Location
- File: `apps/presets-api/src/handlers/moderation.ts`
- Line(s): 67-91 (status route), 132-146 (revert route, inverse ordering)
- Function: `moderationRouter.patch('/:presetId/status')`, `moderationRouter.patch('/:presetId/revert')`

## Description
The status handler reads the preset (line 68), inserts a `moderation_log` row (78-83), then updates the status (86) — three separate, un-batched statements:

1. **Log-before-write:** if the UPDATE fails, an audit record exists for an action that never happened.
2. **TOCTOU on action naming:** two moderators acting concurrently both read the same old status, so `getActionFromStatusChange(preset.status, body.status)` (line 76) can log the wrong action (e.g. both log `unflag`), and last-write-wins on status with no conflict signal.
3. The revert handler has the inverse ordering: revert first (132), log after (141-146) — a failed log insert loses the audit trail for a revert that *did* happen.

## Reproduction Scenario
- Induce a D1 error (or worker kill) between the log insert and `updatePresetStatus` → `moderation_log` claims an approve/reject that was never applied.
- Or: two moderators approve/reject the same pending preset within the same second; compare `moderation_log.action` values against the actual single transition — one entry is mislabeled.

## Evidence
```ts
// moderation.ts:76-86
const action = getActionFromStatusChange(preset.status, body.status); // stale read
await c.env.DB.prepare(
  `INSERT INTO moderation_log (id, preset_id, moderator_discord_id, action, reason, created_at)
   VALUES (?, ?, ?, ?, ?, ?)`
).bind(logId, presetId, auth.userDiscordId!, action, body.reason || null, now).run();
// Update preset status
const updatedPreset = await updatePresetStatus(c.env.DB, presetId, body.status);
```

## Suggested Fix
Batch the pair for atomicity and make the update conditional on the observed status so the concurrent-moderator case is detectable:
```ts
const update = c.env.DB.prepare(
  'UPDATE presets SET status = ?, updated_at = ? WHERE id = ? AND status = ?'
).bind(body.status, now, presetId, preset.status);
const log = c.env.DB.prepare('INSERT INTO moderation_log (...) VALUES (?, ?, ?, ?, ?, ?)')
  .bind(logId, presetId, auth.userDiscordId!, action, body.reason || null, now);
const [updateResult] = await c.env.DB.batch([update, log]);
if (!updateResult.meta.changes) return c.json({ success: false, error: 'CONFLICT', message: 'Preset status changed concurrently' }, 409);
```
Apply the same batch treatment to the revert route (update + log together).

## Why It's Hidden
Sequential single-moderator tests always see all statements succeed in order, so the log always matches reality in CI. The mislabeled-action and orphan-log cases require either concurrency or induced partial failure — neither appears until real multi-moderator use.

## Status

**FIXED 2026-07-19** — Status route batches a conditional `UPDATE ... WHERE id = ? AND status = ? RETURNING *` with the log INSERT (gated on `changes() > 0` so no orphan log rows); concurrent moderation returns 409. Revert route likewise batches revert + log via `prepareRevert`.
