## Description
This recently-churned module is production-referenced only by its own test (`component-context.test.ts`) â€” no handler imports it, so nothing is broken *today* (presumed KEEP-status V4 infrastructure; not re-reported as dead code). Two design flaws will bite the moment it is wired up, plus one hardening gap:

1. **Token lifetime mismatch.** `ComponentContext.interactionToken` is stored for later message edits with `CONTEXT_TTL.STANDARD = 3600` s, but Discord interaction tokens are valid for **15 minutes**. For 45 of the 60 minutes, a retrieved context contains a token guaranteed to 404.
2. **Cache API is per-colo.** `caches.default` is datacenter-local, not global storage. A button click that Discord delivers through a different Cloudflare datacenter than the original command cannot find the context â†’ `getContext` returns null â†’ "context expired" UX for perfectly fresh interactions. KV (or a Durable Object) is the correct substrate; the migration rationale (KV write quota) trades correctness for quota here.
3. **Hardening:** the hash is SHA-256 truncated to 32 bits â€” a collision swaps users' contexts â€” and the read path never verifies `context.userId` against the clicking user.

## Reproduction Scenario
(When wired up.) 1. Run a command that stores context and renders a component. 2. Wait 20+ minutes, click the component: context is found (TTL 1 h) but the stored token 404s on edit. Or: 2'. Click immediately from a session Discord routes via a different CF colo: context lookup misses and the user is told their interaction expired.
