# [BUG-021]: Token refresh never revokes the old token and has no absolute session lifetime

## Severity
MEDIUM

## Type
Auth/Session Logic Error (token rotation without invalidation; unbounded renewal)

## Location
- File: `apps/oauth/src/handlers/refresh.ts`
- Line(s): 96-137 (`POST /auth/refresh`)
- Function: `tokenRouter.post('/refresh')`

## Description
`/auth/refresh` checks whether the *old* token's `jti` was revoked (lines 97-108) but never revokes it after minting the replacement, and the new token copies claims with a fresh `iat`/`exp` and no original-issuance marker carried forward. Consequences:

1. **Rotation without invalidation:** after a refresh, both the old and new tokens are simultaneously valid until the old one's natural expiry.
2. **Unbounded session:** a stolen token can be kept alive forever by refreshing at least once per `exp + 24h` grace window — there is no maximum session age.
3. **No user re-validation:** nothing checks D1; a deleted user's token refreshes indefinitely.

## Reproduction Scenario
1. Obtain a valid JWT; call `POST /auth/refresh` → new token issued.
2. Call `GET /auth/me` with the *old* token → still 200 until its exp (both valid).
3. Chain refreshes daily using the newest token → the session never terminates, even years later, and even if the user row was deleted.

## Evidence
```ts
// refresh.ts:110-133 — new token minted; no revokeToken(payload.jti, ...) call anywhere in the handler
const newJti = crypto.randomUUID();
const newPayload: JWTPayload = {
  sub: payload.sub,
  iat: now,
  exp: newExpiry,
  ...
};
const { token: newToken, expires_at } = await createJWTFromPayload(newPayload, c.env);
```

## Suggested Fix
```ts
// after minting the new token, best-effort revoke the old one:
if (payload.jti && c.env.TOKEN_BLACKLIST) {
  await revokeToken(payload.jti, payload.exp, c.env.TOKEN_BLACKLIST);
}
// carry an absolute-session anchor and enforce it:
const origIat = payload.orig_iat ?? payload.iat;
const MAX_SESSION_SECONDS = 30 * 24 * 60 * 60;
if (now - origIat > MAX_SESSION_SECONDS) {
  return c.json({ success: false, error: 'Session expired, please log in again' }, 401);
}
newPayload.orig_iat = origIat;
// optionally: const user = await findUserById(c.env.DB, payload.sub); if (!user) return 401;
```

## Why It's Hidden
`refresh.test.ts` verifies the new token works and that expired-beyond-grace tokens are rejected — not that the old token *stops* working, nor that refresh chains terminate. Functionally everything appears correct; the gap only matters in a token-theft or account-deletion scenario.

> Source: evidence/d1-workers-analysis.md (2026-07-18 deep-dive, d1-workers area)

## Status

**FIXED 2026-07-19** — `/auth/refresh` now: (1) best-effort revokes the old token's jti after minting the replacement, (2) enforces a 30-day absolute session lifetime anchored by a new `orig_iat` claim carried across refreshes (tokens minted before the claim fall back to their own iat), and (3) re-validates the user still exists in D1.
