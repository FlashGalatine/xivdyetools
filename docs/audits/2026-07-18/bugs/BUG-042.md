# [BUG-042]: Comparison persisted-dye restore filters `undefined` but `getDyeById` returns `null`

## Severity
MEDIUM

## Type
Type-predicate logic error / crash on stale persisted data

## Location
- `apps/web-app/src/components/comparison-tool.ts:249-272` (`loadPersistedDyes`, faulty filter at `:253-255`)
- Contract: `packages/core/src/services/DyeService.ts:103-105` (`getDyeById(id): Dye | null`)

## Description
`DyeService.getDyeById` returns `Dye | null` (backed by `dyesByIdMap.get(id) || null`, `DyeDatabase.ts:396-399`). The comparison tool's restore path filters with `d !== undefined`, so stale IDs pass the filter as `null` values that the type predicate then blesses as `Dye`. The very next calls (`this.dyeSelector.setSelectedDyes(dyes)`, `this.calculateHSVValues()` at `:258-260`) dereference `.hex`/`.itemID` on `null` and throw. The error boundary converts this into the component-level error UI — and since the bad IDs remain persisted in localStorage, the Comparison tool breaks on **every** load until storage is cleared.

The correct pattern exists elsewhere in the codebase: `dye-palette-drawer.ts:779-781` filters `dye !== null`.

## Reproduction Scenario
Persist a comparison containing a Facewear entry (its synthetic negative itemID is a hash of the entry's *name characters*), then change that entry's name upstream in `colors_xiv.json` — the hash changes and the stored ID no longer resolves. Or simply hand-edit the stored `v3_comparison_dyes` array to include a retired/unknown ID. Reload → Comparison tool shows its error boundary on every visit.

## Evidence
```ts
// comparison-tool.ts:253-255
const dyes = savedIds
  .map((id) => dyeService.getDyeById(id))          // (Dye | null)[]
  .filter((d): d is Dye => d !== undefined);       // null !== undefined → null passes as "Dye"

// comparison-tool.ts:258-260 — dereferences the nulls
this.dyeSelector.setSelectedDyes(dyes);
this.selectedDyes = dyes;
this.calculateHSVValues();
```

## Suggested Fix
```ts
const dyes = savedIds
  .map((id) => dyeService.getDyeById(id))
  .filter((d): d is Dye => d !== null);
// Optionally self-heal the stored list:
if (dyes.length !== savedIds.length) this.saveSelectedDyes();
```

## Why It's Hidden
Requires previously-persisted IDs that no longer resolve — a cross-version scenario that no fresh-profile test reproduces; the dye database is stable within any single test run, so the filter's wrong sentinel never gets exercised.

> Source: evidence/web-frontends-analysis.md (2026-07-18 deep-dive, web-frontends area)

## Status

**FIXED 2026-07-18** — the restore filter now uses `d != null` (matching `getDyeById`'s `Dye | null` contract), and the persisted list self-heals: stale IDs are pruned from localStorage whenever any fail to resolve, even if the dye selector is not yet mounted, so a bad ID can never brick the tool across reloads. Regression test added with the mock corrected to return `null` for misses (the old mock returned `undefined` via `.find()`, which is exactly why the wrong sentinel passed tests).
