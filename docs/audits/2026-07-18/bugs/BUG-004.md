# [BUG-004]: XIVAuth account-merge can hit UNIQUE(discord_id) and permanently 500 the login

## Severity
HIGH

## Type
Edge Case / Unhandled Constraint Violation (account merging)

## Location
- File: `apps/oauth/src/services/user-service.ts`
- Line(s): 42-67 (merge path), 129-169 (`updateUser`, no constraint handling)
- Schema: `apps/oauth/schema/users.sql:20-21` (partial unique indexes)
- Caller: `apps/oauth/src/handlers/xivauth.ts:445-472`
- Function: `findOrCreateUser`, `updateUser`

## Description
`findOrCreateUser` looks up by `xivauth_id` first. If found (user row B), it calls `updateUser` with `discord_id: existingUser.discord_id || discord_id` — i.e. it stamps the XIVAuth-linked Discord ID onto row B. If that Discord ID already belongs to a *different* row A (the user previously logged in directly via Discord), the UPDATE violates the partial unique index `idx_users_discord_id`. `updateUser` has no constraint handling (the try/catch in `findOrCreateUser` only wraps the INSERT path), so the error propagates to the handler's generic catch → 500 "Authentication failed". Because the data condition persists, **every subsequent XIVAuth login for that user fails deterministically** until someone manually merges the rows in D1.

## Reproduction Scenario
1. Login via Discord → row A created (`discord_id = D`).
2. Login via XIVAuth with no Discord linked → row B created (`xivauth_id = X`).
3. Link Discord `D` inside XIVAuth, then login via XIVAuth again → lookup by `X` finds B → UPDATE sets `discord_id = D` → UNIQUE violation → 500.
4. Every retry hits the same path. Permanent login failure for that user via XIVAuth.

## Evidence
```ts
// user-service.ts:57-63 — merge path, no constraint handling
if (existingUser) {
  return await updateUser(db, existingUser.id, {
    discord_id: existingUser.discord_id || discord_id,  // may collide with another row
    xivauth_id: existingUser.xivauth_id || xivauth_id,
    ...
```
```sql
-- users.sql:20
CREATE UNIQUE INDEX IF NOT EXISTS idx_users_discord_id ON users(discord_id) WHERE discord_id IS NOT NULL;
```

## Suggested Fix
Detect the collision before merging, and perform an explicit atomic account merge:
```ts
if (!existingUser.discord_id && discord_id) {
  const owner = await db.prepare('SELECT id FROM users WHERE discord_id = ? AND id != ?')
    .bind(discord_id, existingUser.id).first<{ id: string }>();
  if (owner) {
    // Merge: keep existingUser (has xivauth_id), move characters, drop the discord-only row
    await db.batch([
      db.prepare('UPDATE xivauth_characters SET user_id = ? WHERE user_id = ?').bind(existingUser.id, owner.id),
      db.prepare('DELETE FROM users WHERE id = ?').bind(owner.id),
      db.prepare('UPDATE users SET discord_id = ?, updated_at = datetime(\'now\') WHERE id = ?')
        .bind(discord_id, existingUser.id),
    ]);
  }
}
```
At minimum, catch the UNIQUE error in `updateUser` and fall back to returning the row that owns `discord_id` without overwriting.

## Why It's Hidden
`user-service.test.ts` covers create, the simple merge (found-by-discord gains an xivauth_id), and the INSERT race. The two-pre-existing-rows collision requires a specific multi-session sequence (Discord-first login + XIVAuth-without-link login + later linking) that no test models, and it produces only a generic "Authentication failed" 500 in production logs.
