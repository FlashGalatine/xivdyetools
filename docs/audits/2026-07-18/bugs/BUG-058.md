# [BUG-058]: `verifyJWTSignatureOnly` max-age check silently skipped when `iat` is missing or 0

## Severity
LOW

## Type
Boundary condition / inverted guard (null/falsy propagation defeats caller intent)

## Location
`packages/auth/src/jwt.ts:213-220` (`verifyJWTSignatureOnly`)

## Description
The optional age limit is applied only when *both* the caller passed `maxAgeMs` *and* the token carries a truthy `iat`:

`if (maxAgeMs !== undefined && payload.iat) { ... }`

A signed token with a missing `iat` claim — or the falsy edge value `iat: 0` (the Unix epoch, a valid timestamp) — bypasses the age check entirely while still passing signature and `sub` validation. Since this function deliberately ignores `exp` (its documented purpose is refresh-style verification with a grace period), a signed token without `iat` is accepted **forever**, even when the caller explicitly asked for e.g. a 7-day cap. The guard inverts the caller's intent: the stricter the caller tries to be, the more the missing claim matters, yet it fails open.

Exploitation today requires a token signed with the real secret but lacking `iat` — the oauth issuer always sets it — so this is a hardening gap rather than a live hole. But the shared package is a public building block; future issuers or test tokens won't necessarily uphold that invariant.

## Reproduction Scenario
1. Construct `{ sub: '123', exp: 1 }` (no `iat`), sign with `JWT_SECRET` (any code path or tool with signing access).
2. `await verifyJWTSignatureOnly(token, secret, 7 * 24 * 60 * 60 * 1000)` → returns the payload — no age check ran, and `exp` is ignored by design → the token verifies indefinitely despite the 7-day cap.
3. Same result with `iat: 0` (falsy).

## Evidence
```ts
// packages/auth/src/jwt.ts:213-220
// Check max age if specified
if (maxAgeMs !== undefined && payload.iat) {   // no/zero iat → no age check at all
  const now = Date.now();
  const tokenAge = now - payload.iat * 1000;
  if (tokenAge > maxAgeMs) {
    return null;
  }
}
return payload;
```

## Suggested Fix
When the caller requests an age cap, treat a missing or non-numeric `iat` as a verification failure:

```ts
if (maxAgeMs !== undefined) {
  if (typeof payload.iat !== 'number') return null;  // fail closed; also fixes iat: 0
  const tokenAge = Date.now() - payload.iat * 1000;
  if (tokenAge > maxAgeMs) return null;
}
```

## Why It's Hidden
All real tokens have `iat`, so every integration test passes and production behavior is correct. The bypass only exists for tokens that violate an *unstated* issuer invariant, which no test fabricates — and a fail-open guard produces no error, no log line, and no behavioral difference until precisely the scenario the caller was defending against occurs.

## Status

**FIXED 2026-07-19** — when `maxAgeMs` is passed, a missing or non-numeric `iat` now fails verification (also fixes the falsy `iat: 0` epoch edge).
