# [BUG-041]: LanguageService English fallback never loads for non-English users

## Severity
MEDIUM

## Type
i18n logic gap / error-handling gap

## Location
- `apps/web-app/src/services/language-service.ts:159-178` (`t()`)
- `apps/web-app/src/services/language-service.ts:311-331` (`loadWebAppTranslations`)
- `apps/web-app/src/services/language-service.ts:356-362` (`preloadLocales` — has no callers)

## Description
`t()` is designed to fall back to English for keys missing from the active locale, but `en` is only ever inserted into the `webAppTranslations` cache when (a) the current locale *is* `en`, or (b) the current locale's file **failed to load** (`:320-329`). For a Japanese/German/etc. user whose locale file loads fine, `webAppTranslations.get('en')` is `undefined`, so any key that exists in `en.json` but not yet in the active locale renders as the raw dot-path key (`return key`, `:172-175`) instead of the English string. `preloadLocales` exists (`:356`) but is never called anywhere in `src/` (verified by grep).

## Reproduction Scenario
Set locale to `ja`; add a new UI string translated only in `en.json` (the normal state right after adding a feature): the UI shows `tools.newthing.title` literally instead of the English text. Any locale-file gap that slips past release validation ships raw keys to all non-English users.

## Evidence
```ts
// language-service.ts:160-175
const translations = webAppTranslations.get(this.currentLocale);
const englishTranslations = webAppTranslations.get('en');   // ← undefined for non-en users

let value = this.getNestedValue(translations, key);

if (value === undefined && this.currentLocale !== 'en') {
  value = this.getNestedValue(englishTranslations, key);    // ← always undefined
}

if (value === undefined) {
  logger.warn(`Translation not found: ${key}`);
  return key;                                               // raw dot-path shown in UI
}
```

## Suggested Fix
In `setLocale` (`:106-108`), ensure English is always cached:

```ts
if (!webAppTranslations.has(locale)) {
  await this.loadWebAppTranslations(locale);
}
if (locale !== 'en' && !webAppTranslations.has('en')) {
  await this.loadWebAppTranslations('en');   // enables the documented fallback
}
```
Cost: one extra lazily-loaded JSON chunk for non-English users.

## Why It's Hidden
Tests and dev usage run in `en`, where the fallback path is unreachable. The i18n validation script checks file completeness at release time, masking the mid-development window — the broken fallback only manifests for real non-English users when a key gap ships.
