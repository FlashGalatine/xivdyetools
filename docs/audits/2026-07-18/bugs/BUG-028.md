# [BUG-028]: universalis-proxy re-serves stale SWR responses with full `max-age`, exporting staleness to downstream caches

## Severity

MEDIUM

## Type

Cache correctness / SWR semantics

## Location

- `apps/universalis-proxy/src/services/cached-fetch.ts:206-217` (`buildCacheHeaders`)
- Callers: `apps/universalis-proxy/src/index.ts:212, 275, 312`

## Description

`buildCacheHeaders()` always emits `Cache-Control: public, max-age=${config.cacheTtl}` — **including when `isStale === true`**. A response served from the stale-while-revalidate window is already up to `cacheTtl + swrWindow` old (7 min for aggregated prices; 30 h for worlds/data-centers), yet it instructs browsers and intermediaries to treat it as fresh for another full TTL. Worst-case client-observed price age becomes `cacheTtl + swrWindow + cacheTtl` ≈ 12 minutes instead of the intended ≈ 7; for the 24 h endpoints it's an extra 24 h on top of up to 30 h. The `X-Cache-Stale: true` debug header is informational only — HTTP caches ignore it.

Related doc drift: `apps/universalis-proxy/CLAUDE.md` documents `X-Cache: HIT-STALE`, but the code emits `X-Cache: HIT` plus a separate `X-Cache-Stale` header.

## Reproduction Scenario

1. Prime the cache: `GET /api/v2/aggregated/aether/5729` (MISS → cached).
2. Wait 6 minutes (past the 5-min TTL, inside the 2-min SWR window).
3. `GET` again → edge serves the stale body with `Cache-Control: public, max-age=300`.
4. The browser will not revalidate for 5 more minutes, even though the edge itself already refreshed in the background — the user keeps rendering ~11-minute-old prices as "fresh".

## Evidence

```ts
// cached-fetch.ts:206-217
export function buildCacheHeaders(
  source: CacheSource,
  isStale: boolean,
  config: CacheConfig
): Record<string, string> {
  return {
    'X-Cache': source === 'upstream' ? 'MISS' : 'HIT',
    'X-Cache-Source': source,
    'X-Cache-Stale': isStale ? 'true' : 'false',
    'Cache-Control': `public, max-age=${config.cacheTtl}`,   // ← same full TTL even when stale
  };
}
```

## Suggested Fix

```ts
'Cache-Control': isStale
  ? 'public, max-age=0, must-revalidate'          // or a small value, e.g. max-age=30
  : `public, max-age=${config.cacheTtl}, stale-while-revalidate=${config.swrWindow}`,
```

Better still: compute remaining freshness from the stored `X-Cached-At` and emit `max-age=<remaining>`, letting downstream caches implement SWR natively via the `stale-while-revalidate` directive.

## Why It's Hidden

Service tests assert the `isStale` flag and header *presence*, not the semantic interaction between `max-age` and payload age across cache layers. Detecting it requires wall-clock, multi-layer cache observation — nothing a unit suite does. Users just see "prices lag sometimes", which is easy to blame on Universalis itself.
