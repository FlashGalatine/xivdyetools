# [BUG-009]: Approve/Reject buttons posted by the main bot route to a Discord application that cannot handle them

## Severity
HIGH

## Type
Logic error / cross-service routing defect (Discord component interactions)

## Location
- `apps/discord-worker/src/handlers/commands/preset.ts:979-999` (`notifyModerationChannel` button row) and `:1038-1076` (`notifyEditModerationChannel`)
- `apps/discord-worker/src/index.ts:209-229` (`/webhooks/preset-submission` button row)
- `apps/discord-worker/src/handlers/buttons/index.ts:64-81` (main bot's button dispatcher — copy buttons only)
- `apps/moderation-worker/src/handlers/buttons/index.ts:89-99` (the only handlers for these custom_ids)

## Description
Three code paths in **discord-worker** post moderation-channel embeds carrying `preset_approve_{id}` / `preset_reject_{id}` buttons, sent with `env.DISCORD_TOKEN` — the **main bot application's** token (client ID `1447108133020369048`).

Discord delivers component interactions to the **application that owns the message**. The main bot's button dispatcher only recognizes `copy_hex_` / `copy_rgb_` / `copy_hsv_` and answers everything else with "This button is not recognized." The handlers for `preset_approve_` / `preset_reject_` exist **only** in moderation-worker, which is a *separate Discord application* (client ID `1453806659708129374`, its own interactions endpoint). Interactions from buttons on main-bot messages will never reach it.

Result: every approve/reject button on webhook-notified and Discord-submitted pending presets is dead. Moderators must fall back to `/preset moderate` on the moderation bot.

*Caveat:* if in deployed reality both workers share one Discord application/token (contradicting both CLAUDE.mds and the distinct `DISCORD_CLIENT_ID` values), severity drops to LOW (the moderation-worker handlers become dead code instead). Verify which application posts in the production moderation channel before fixing.

## Reproduction Scenario
1. Submit a preset via the web app (presets-api → `POST /webhooks/preset-submission` on discord-worker → embed with buttons in the moderation channel), or via Discord `/preset submit` (pending path → `notifyModerationChannel`).
2. As a moderator, click **Approve** on the resulting embed.
3. The main bot replies ephemerally: "This button is not recognized." The preset stays pending.

## Suggested Fix
Pick one:
1. **Preferred:** post these embeds via the moderation bot's identity — either add a `MODERATION_BOT_TOKEN` secret to discord-worker, or forward the notification over a service binding and let moderation-worker post it. Buttons then route to moderation-worker, which already handles them.
2. Handle `preset_approve_` / `preset_reject_` in discord-worker's dispatcher (its `preset-api.ts` already has `approvePreset`/`rejectPreset` + `isModerator`; the reject modal would need porting — option 1 is simpler).
3. Minimum viable: drop the buttons from main-bot-posted embeds and add a "use `/preset moderate`" hint so the UI stops advertising a dead affordance.

## Why It's Hidden
Unit tests exercise moderation-worker's `handlePresetApproveButton` directly with mocked interactions, and discord-worker's tests only cover copy buttons. The failure lives at the Discord-side application-routing boundary, which no test (and no single-worker staging smoke test) crosses; it only shows when a real moderator clicks a real button on a message posted by the *other* application.

## Status

**FIXED 2026-07-19** — all three moderation-embed paths now go through one shared builder (`preset-notifications.ts`). When the new `MODERATION_BOT_TOKEN` secret (the moderation application's token) is configured, embeds are posted with it so approve/reject/revert buttons route to moderation-worker and work; when unset, buttons are omitted and a `/preset moderate` hint is shown instead of a dead affordance. Deploy note: set `MODERATION_BOT_TOKEN` on discord-worker to enable working buttons.
