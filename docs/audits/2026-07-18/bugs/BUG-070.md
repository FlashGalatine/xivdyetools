# [BUG-070]: api-worker `parseBooleanParam` silently drops invalid values — 200 with unfiltered data instead of 400

## Severity

LOW

## Type

Input validation inconsistency / silent contract violation

## Location

- `apps/api-worker/src/lib/validation.ts:231-236` (`parseBooleanParam`)
- Related inconsistency: `apps/api-worker/src/routes/dyes.ts:234-235` (empty-string numeric params → misleading `MISSING_PARAMETER`)

## Description

Every other parser in the api-worker validation layer throws a structured 400 on bad input (`parseEnumParam`, `parseIntParam`, `parseFloatParam`, `parseHex`), but `parseBooleanParam` returns `undefined` for anything outside `{'true','1','false','0'}`. A request like `?metallic=yes` therefore has its filter **silently ignored**: the client receives a 200 with the *unfiltered* result set and no signal that its filter was dropped — worse than an error, because downstream logic consumes wrong data confidently.

Same consistency theme: `?minPrice=` (parameter present but empty) reaches `parseIntParam('' , ...)` with no default and throws `MISSING_PARAMETER` — a misleading code for a parameter that *was* supplied, just empty.

## Reproduction Scenario

1. `GET /v1/dyes?metallic=yes` → 200, full dye list (filter ignored). Client believes it requested metallic-only dyes.
2. Compare `GET /v1/dyes?sort=bogus` → 400 `VALIDATION_ERROR` — demonstrating the API's own stricter contract elsewhere.

## Evidence

```ts
// validation.ts:231-236
export function parseBooleanParam(value: string | undefined): boolean | undefined {
  if (value === undefined || value === '') return undefined;
  if (value === 'true' || value === '1') return true;
  if (value === 'false' || value === '0') return false;
  return undefined;   // ← invalid value indistinguishable from "param not sent"
}
```

## Suggested Fix

```ts
export function parseBooleanParam(value: string | undefined, name?: string): boolean | undefined {
  if (value === undefined || value === '') return undefined;
  if (value === 'true' || value === '1') return true;
  if (value === 'false' || value === '0') return false;
  throw new ApiError(ErrorCode.VALIDATION_ERROR,
    `Parameter "${name ?? 'boolean'}" must be one of: true, false, 1, 0.`, 400,
    { parameter: name, received: value, expected: ['true', 'false', '1', '0'] });
}
```

Callers in `parseDyeFilters` / `routes/dyes.ts` pass the param name. Additionally map empty-but-present numeric params to `VALIDATION_ERROR` instead of `MISSING_PARAMETER`.

## Why It's Hidden

A dropped filter never errors — tests assert correct behavior for valid values and often for *missing* values, but "present and invalid" boolean inputs are the untested third state. API consumers see plausible (just unfiltered) data and rarely notice.
