# [BUG-007]: OAuth rate-limit prefix shadowing gives `/auth/xivauth/callback` the wrong (stricter) limit — duplicated in two places

## Severity
HIGH

## Type
Logic error / configuration shadowing (edge case in prefix matching)

## Location
- `packages/rate-limiter/src/presets/configs.ts:19-45` (`OAUTH_LIMITS` + `getOAuthLimit`)
- `apps/oauth/src/services/rate-limit.ts:52-58` (`getConfigForPath` — independent local copy with the same prefix-shadowing)

The same routing logic exists in two divergent copies, and **both** have the bug. The duplication itself worsens this bug class: a fix applied to the shared package alone would leave the oauth worker (the actual production consumer of these limits) still broken, and vice versa.

## Description
`getOAuthLimit` iterates `Object.entries(OAUTH_LIMITS)` in insertion order and returns the first entry where `path.startsWith(key)`. The key `'/auth/xivauth'` (10 req/min, login initiation) is inserted at line 22, *before* `'/auth/xivauth/callback'` (20 req/min, token exchange) at line 26. Any request to `/auth/xivauth/callback` matches the shorter `'/auth/xivauth'` prefix first and receives the stricter login-initiation limit (10/min) instead of the intended callback limit (20/min). The `'/auth/xivauth/callback'` config entry is unreachable dead configuration.

The oauth worker's own `getConfigForPath` replicates the identical mistake with explicit branches: it checks `path.startsWith('/auth/xivauth')` (returning the 10/min `/auth/discord` config) before its `path.startsWith('/auth/xivauth/callback')` branch, which is therefore also dead code.

## Reproduction Scenario
1. `getOAuthLimit('/auth/xivauth/callback')` → returns `{ maxRequests: 10, windowMs: 60_000 }`. Expected: 20/min.
2. In production (`apps/oauth`): a user completes XIVAuth login; browser redirects/retries or a double-submit hits `/auth/xivauth/callback` more than 10 times in a minute from a shared NAT/corporate IP → legitimate callbacks 429 at half the intended budget.

## Evidence
```ts
// packages/rate-limiter/src/presets/configs.ts:19-45
export const OAUTH_LIMITS: Record<string, RateLimitConfig> = {
  '/auth/discord': { maxRequests: 10, windowMs: 60_000 },
  '/auth/xivauth': { maxRequests: 10, windowMs: 60_000 },        // matches first…
  '/auth/callback': { maxRequests: 20, windowMs: 60_000 },
  '/auth/xivauth/callback': { maxRequests: 20, windowMs: 60_000 }, // …never reached
  '/auth/refresh': { maxRequests: 30, windowMs: 60_000 },
  default: { maxRequests: 30, windowMs: 60_000 },
};

export function getOAuthLimit(path: string): RateLimitConfig {
  for (const [key, config] of Object.entries(OAUTH_LIMITS)) {
    if (key !== 'default' && path.startsWith(key)) {
      return config;                                              // first prefix wins
    }
  }
  return OAUTH_LIMITS.default;
}
```

```ts
// apps/oauth/src/services/rate-limit.ts:52-58 — independent duplicate, same bug
if (path.startsWith('/auth/discord') || path.startsWith('/auth/xivauth')) {
  return OAUTH_LIMITS['/auth/discord'];                           // 10/min — shadows callback
}
if (path.startsWith('/auth/callback') || path.startsWith('/auth/xivauth/callback')) {
  return OAUTH_LIMITS['/auth/callback'];                          // unreachable for xivauth
}
```

## Suggested Fix
Match longest prefix first in the shared package, and delete the app-local copy in favor of the shared function:

```ts
// packages/rate-limiter/src/presets/configs.ts
export function getOAuthLimit(path: string): RateLimitConfig {
  const entries = Object.entries(OAUTH_LIMITS)
    .filter(([key]) => key !== 'default')
    .sort((a, b) => b[0].length - a[0].length); // longest prefix wins
  for (const [key, config] of entries) {
    if (path.startsWith(key)) return config;
  }
  return OAUTH_LIMITS.default;
}
```

Then in `apps/oauth/src/services/rate-limit.ts`, replace `getConfigForPath` with `getOAuthLimit(path)` from `@xivdyetools/rate-limiter` so the routing logic exists exactly once.

## Why It's Hidden
Unit tests exercise one exact path against one expected config at a time; the bug only manifests when a longer key shares a prefix with an earlier shorter key. The wrong result is still a *valid* `RateLimitConfig` — nothing throws, headers look plausible, and the difference (10 vs 20 per minute) only surfaces as sporadic 429s for XIVAuth users under bursty-but-legitimate callback traffic. The duplicated app-local copy also means a reviewer reading either implementation in isolation sees "callback handled" branches that are in fact dead.

## Status

**FIXED 2026-07-18** — `getOAuthLimit` in `@xivdyetools/rate-limiter` now matches the longest prefix (keys pre-sorted by length at module load). Both app-local copies in apps/oauth (`rate-limit.ts` `getConfigForPath` and the Durable Object variant in `rate-limit-do.ts`, a third copy found during the fix) were deleted in favor of the shared function, so the routing logic now exists exactly once. The package test that documented the shadowing as expected behavior was corrected; regression tests added in both the package and the oauth worker. Note: `@xivdyetools/rate-limiter` needs a version bump + npm publish for non-workspace consumers.
