# [BUG-076]: StorageService falsy-default handling drops legitimate `0`/`''`/`false` values

## Severity
LOW

## Type
Latent API-contract bug (`||` vs `??`)

## Location
- `apps/web-app/src/services/storage-service.ts:38-62` (`getItem`: `return defaultValue || null` in three places)
- `apps/web-app/src/services/storage-service.ts:285-303` (`getItemWithTTL`: `return data.value || defaultValue || null`)

## Description
All default-return paths use `||` instead of `??`. `getItem<T>(key, 0)`, `getItem(key, '')`, and `getItem(key, false)` all return `null` instead of the supplied default. Worse, `getItemWithTTL` discards a **stored** falsy value even when it is present and unexpired, returning the default (or `null`) instead — a stored counter of `0`, an empty string, or a `false` toggle is unrecoverable through the TTL API.

Current callers happen to survive: e.g. `WelcomeModal.shouldShow` (`welcome-modal.ts:110`) treats `null` and `false` alike, and `v4-layout.ts:71` patches over it with `?? false`. This is a latent contract bug that will bite the next caller storing a falsy value.

## Reproduction Scenario
`StorageService.setItemWithTTL('counter', 0, 60000)` then `StorageService.getItemWithTTL('counter', 42)` → returns `42` (or `null` without a default), not the stored `0`. Similarly `getItem('missing', false)` returns `null`, not `false`.

## Evidence
```ts
// storage-service.ts:40-48
if (!this.isAvailable()) {
  return defaultValue || null;      // getItem(k, false) → null, not false
}
...
if (item === null) {
  return defaultValue || null;
}

// storage-service.ts:293-298
if (data.expiresAt && Date.now() > data.expiresAt) {
  this.removeItem(key);
  return defaultValue || null;
}
return data.value || defaultValue || null;   // stored 0 / '' / false → silently dropped
```

## Suggested Fix
```ts
return defaultValue ?? null;                     // all three getItem paths
return data.value ?? defaultValue ?? null;       // getItemWithTTL
```

## Why It's Hidden
All present callers store truthy values or coerce the result themselves, so no visible behavior differs today; the bug only manifests when a falsy value is round-tripped, which no test exercises.

> Source: evidence/web-frontends-analysis.md (2026-07-18 deep-dive, web-frontends area)

## Status

**FIXED 2026-07-19** — all ten `defaultValue || null` returns and the `data.value || defaultValue` TTL read use `??`, so stored/default `0`/`''`/`false` values round-trip correctly.
