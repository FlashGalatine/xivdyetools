# [BUG-053]: Bot API secret compared with non-constant-time `===`

## Severity
LOW

## Type
Security Hardening Gap (timing side-channel on secret comparison)

## Location
- File: `apps/presets-api/src/middleware/auth.ts`
- Line(s): 99
- Function: `authMiddleware`

## Description
`token === c.env.BOT_API_SECRET` is an early-exit JS string comparison, so comparison time correlates with the length of the matching prefix. Remote timing exploitation across Cloudflare's edge is impractical (network jitter dwarfs the signal), but the codebase already standardized on constant-time comparison for HMAC verification elsewhere (oauth `state-signing.ts` FINDING-001 uses `crypto.subtle.verify`), and `@xivdyetools/auth` is already imported in this very file — making this the one remaining variable-time secret comparison in the worker.

## Reproduction Scenario
Theoretical: an attacker submits candidate bearer tokens and statistically measures response-time differences to recover the secret prefix byte-by-byte. Not practically achievable through CF edge jitter; documented as consistency/hardening, not an active exploit path.

## Evidence
```ts
// auth.ts:99
if (token === c.env.BOT_API_SECRET) {
```

## Suggested Fix
Compare fixed-length digests instead of raw strings:
```ts
async function timingSafeEqualStr(a: string, b: string): Promise<boolean> {
  const enc = new TextEncoder();
  const [da, db] = await Promise.all([
    crypto.subtle.digest('SHA-256', enc.encode(a)),
    crypto.subtle.digest('SHA-256', enc.encode(b)),
  ]);
  const ua = new Uint8Array(da), ub = new Uint8Array(db);
  let diff = 0;
  for (let i = 0; i < ua.length; i++) diff |= ua[i] ^ ub[i];
  return diff === 0;
}
```
(or use the Workers-supported `crypto.subtle.timingSafeEqual` on equal-length encodings).

## Why It's Hidden
Functionally identical to the correct implementation in every test and normal request; only visible to a reviewer specifically auditing secret-comparison call sites.
