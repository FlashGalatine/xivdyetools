# [BUG-049]: Daily submission limit is check-then-insert — concurrent requests exceed the cap

## Severity
LOW

## Type
Race Condition / TOCTOU (D1 read-modify-write without transaction)

## Location
- File: `apps/presets-api/src/services/rate-limit-service.ts`
- Line(s): 77-106 (`checkSubmissionRateLimit`)
- Caller: `apps/presets-api/src/handlers/presets.ts:394`
- Function: `checkSubmissionRateLimit`, `presetsRouter.post('/')`

## Description
`checkSubmissionRateLimit` does a `SELECT COUNT(*)` on the user's presets for today, and the INSERT happens several awaits later in the handler with no coordination. N parallel POSTs from one user at 9/10 quota all read `count=9`, all pass, and all insert — ending the day at 9+N. There is no cross-request transaction in D1 for this pattern and no constraint backstop (unlike the dye-signature race, which has a UNIQUE index to catch it).

## Reproduction Scenario
1. As one user, use 9 of the 10 daily submissions.
2. Fire 10 simultaneous `POST /api/v1/presets` with distinct dye combinations.
3. Several (up to all 10) succeed; the user ends the day well over the documented limit.

## Evidence
```ts
// rate-limit-service.ts:93-102 — pure read; insert occurs ~5 awaits later in the handler
const result = await db.prepare(query).bind(userDiscordId, today.toISOString(), tomorrow.toISOString())
  .first<{ count: number }>();
const submissionsToday = result?.count || 0;
return { allowed: submissionsToday < DAILY_SUBMISSION_LIMIT, ... };
```

## Suggested Fix
Options, in increasing effort:
1. Accept and document the small overshoot (abuse impact is bounded — a few extra presets/day).
2. Re-check the count *after* insert and delete-if-over.
3. Use the already-existing (currently unused) `rate_limits` table as an atomic counter:
```sql
INSERT INTO rate_limits (key, count, expires_at) VALUES (?, 1, ?)
ON CONFLICT(key) DO UPDATE SET count = count + 1
RETURNING count;
```
and reject when the returned count exceeds 10.

## Why It's Hidden
Sequential tests never race the check against the insert; single-user browsing behavior rarely does either. It requires deliberate parallel submission, and even then produces no error — just a quietly exceeded quota.

> Source: evidence/d1-workers-analysis.md (2026-07-18 deep-dive, d1-workers area)

## Status

**FIXED 2026-07-19** — After the INSERT, the submission handler re-counts via `getSubmissionCountToday`; if the count (which includes this insert) exceeds the limit, the request deletes its own preset/votes and returns 429. The same count feeds `remaining_submissions` (see OPT-016).
