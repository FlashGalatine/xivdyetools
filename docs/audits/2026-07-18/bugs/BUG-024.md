# [BUG-024]: Logger context redaction is exact, case-sensitive key match — trivially bypassed by common key spellings

## Severity
MEDIUM

## Type
Security / redaction bypass (error-handling gap in the logging pipeline)

## Location
- `packages/logger/src/core/base-logger.ts:178-211` (`redactSensitiveFields`, esp. 183-187; depth cap at 179, 190)
- `packages/logger/src/constants.ts:14-44` (`CORE_REDACT_FIELDS`, `WORKER_SPECIFIC_REDACT_FIELDS`)

## Description
`redactSensitiveFields` redacts a context key only when `field in redacted` — an exact, case-sensitive lookup against the redact list. The list contains specific lowercase spellings (`password`, `token`, `secret`, `authorization`, `cookie`, `api_key`, `apiKey`, `access_token`, `refresh_token`; worker extras are snake_case only: `jwt_secret`, `bot_api_secret`, `bot_signing_secret`, `discord_client_secret`).

All of these therefore leak unredacted:
- Alternate casing: `Token`, `TOKEN`, `Password`, `Authorization` (the canonical HTTP header casing)
- camelCase of the worker secrets: `jwtSecret`, `botApiSecret`, `discordClientSecret`
- Compound keys: `sessionToken`, `userToken`, `authToken`, `webhookSecret`

Additionally, the recursion depth cap of 3 (`MAX_REDACT_DEPTH`) means a secret nested 4+ levels deep is emitted verbatim even when its key *does* match the list exactly.

## Reproduction Scenario
1. `logger.error('login failed', err, { Authorization: 'Bearer eyJ…' })` → the JSON log line sent to Cloudflare Logs contains the full bearer token (console path shows `Authorization` untouched because `authorization` ≠ `Authorization`).
2. `logger.info('cfg', { config: { auth: { discord: { jwt_secret: 's3cret' } } } })` → key matches the list but sits at depth 4 → emitted verbatim.

## Evidence
```ts
// packages/logger/src/core/base-logger.ts:178-187
protected redactSensitiveFields(context: LogContext, depth: number = 0): LogContext {
  const MAX_REDACT_DEPTH = 3;
  const redacted = { ...context };
  const fieldsToRedact = this.config.redactFields || DEFAULT_REDACT_FIELDS;

  for (const field of fieldsToRedact) {
    if (field in redacted) {            // exact case-sensitive match only
      redacted[field] = '[REDACTED]';
    }
  }
```
```ts
// packages/logger/src/constants.ts:30-35 — snake_case only; camelCase variants leak
export const WORKER_SPECIFIC_REDACT_FIELDS = [
  'jwt_secret', 'bot_api_secret', 'bot_signing_secret', 'discord_client_secret',
] as const;
```

## Suggested Fix
Normalize keys before comparing, and match on normalized suffixes for the generic secret words:

```ts
const normalize = (k: string) => k.toLowerCase().replace(/[_-]/g, '');
const redactSet = new Set(fieldsToRedact.map(normalize)); // collapses api_key/apiKey too

for (const key of Object.keys(redacted)) {
  const n = normalize(key);
  if (redactSet.has(n) || /(token|secret|password|apikey)$/.test(n)) {
    redacted[key] = '[REDACTED]';
  }
}
```
Raise or remove `MAX_REDACT_DEPTH` (guard against cycles with a `WeakSet` of visited objects instead of a fixed depth).

## Why It's Hidden
The redaction tests in `packages/logger/src/types.test.ts` assert on the exact listed keys, which pass. Real leak paths use *other* spellings — header-cased `Authorization` objects spread into context, camelCase env-var mirrors, compound names like `sessionToken` — none of which any test exercises. Logs "look redacted" in the common case, so nobody inspects the aggregated output for the variant spellings.
