# [BUG-015]: `?status=pending|rejected|flagged` publicly lists unmoderated and rejected content

## Severity
MEDIUM

## Type
Information Disclosure / Insufficient Input Sanitization (auth-gate bypass)

## Location
- File: `apps/presets-api/src/services/preset-service.ts`
- Line(s): 131-137 (only `'hidden'` remapped)
- Also: `apps/presets-api/src/handlers/presets.ts:53-67` (raw cast of `status` query param)
- Function: `getPresets`, `presetsRouter.get('/')`

## Description
The status filter is only sanitized against `'hidden'` (`safeStatus = status === 'hidden' ? 'approved' : status`). Every other value is bound directly, so any anonymous caller can list `?status=pending` (content that *failed* profanity moderation and is awaiting review — exactly what the pending flow is supposed to keep private), `?status=rejected`, or `?status=flagged`. The moderator-only endpoint for the same data (`GET /api/v1/moderation/pending`) requires moderator auth, so this is an unauthenticated bypass of that gate. The value is also unvalidated, so `?status=banana` silently returns an empty list rather than a 400.

## Reproduction Scenario
1. `GET /api/v1/presets?status=pending` with no `Authorization` header.
2. Response: the full pending-moderation queue, including flagged names/descriptions and author Discord IDs — identical data to the moderator-gated `/moderation/pending`.

## Evidence
```ts
// preset-service.ts:132-137 — only 'hidden' is remapped
const safeStatus = status === 'hidden' ? 'approved' : status;
const conditions: string[] = ["status = ? AND status != 'hidden'"];
const params: (string | number)[] = [safeStatus];
```
```ts
// presets.ts:59 — raw query value cast straight into the filter type
status: status as PresetFilters['status'],
```

## Suggested Fix
Whitelist publicly listable statuses; let only moderators query others:
```ts
// in the GET / handler
const auth = c.get('auth');
const requested = status as PresetFilters['status'];
const effectiveStatus = auth.isModerator ? requested : 'approved';
```
(or return 400 for non-`approved` values from non-moderators).

## Why It's Hidden
Tests cover the `approved` listing and the `hidden` remap; the other enum values were assumed to be harmless filters. Because the endpoint is public and the parameter is free-form, nothing in CI ever exercised `?status=pending` without moderator credentials.

## Status

**FIXED 2026-07-18** — `GET /presets` now rejects unknown `status` values with 400 and non-approved status filters from non-moderators with 403; the service-level hidden remap remains as defense-in-depth. Regression tests added.
