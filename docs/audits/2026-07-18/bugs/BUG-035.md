# [BUG-035]: Deferred-interaction follow-up edits never check response.ok; error-path edits can reject unhandled inside waitUntil

## Severity
MEDIUM

## Type
Error-handling gap / async anti-pattern (systemic across deferred flows)

## Location
- `apps/discord-worker/src/handlers/commands/budget.ts:257-272` (success edit, result ignored) and `:293-295` (catch-path edit, not itself guarded)
- `apps/discord-worker/src/handlers/commands/preset.ts:178-231, 459-529, 589-609` (all `process*Command` bodies)
- `apps/moderation-worker/src/handlers/buttons/preset-moderation.ts:108-137` (success) and `:143-165` (catch)
- Root enabler: `apps/discord-worker/src/utils/discord-api.ts:137-163` returns the raw `Response`; **no caller in either worker checks `.ok`**

## Description
Two related gaps in the defer → `ctx.waitUntil(process...)` → `editOriginalResponse` pattern:

1. **Silent 4xx/5xx.** `editOriginalResponse` / `sendFollowUp` return the fetch `Response`; every call site discards it. If Discord returns 400 (embed limits: description > 4096 chars or total > 6000 — realistic for `/preset list` with long names or verbose-locale budget descriptions), 401/404 (interaction token expired — tokens die after 15 minutes), or 429, the user is left on "Bot is thinking…" forever and nothing is logged.
2. **Unguarded catch-path edit.** In every `catch` block the fallback `editOriginalResponse(...)` is awaited without its own try/catch. `AbortSignal.timeout(5000)` (`discord-api.ts:17, 72, 161`) makes a thrown `TimeoutError` a normal occurrence; when it throws, the `waitUntil` promise rejects — the error-report edit is lost *and* the original error context with it.

## Reproduction Scenario
1. `/preset list` in a category where 10 preset lines + tips exceed 4096 chars → Discord returns 400 → user sees eternal "thinking", no log entry.
2. Make Discord's webhook endpoint slow (>5 s) during a failing budget lookup → catch-path edit throws `TimeoutError` → unhandled rejection inside `waitUntil`, user sees nothing.

## Evidence
```ts
// apps/discord-worker/src/handlers/commands/budget.ts:257 — Response discarded
await editOriginalResponse(env.DISCORD_CLIENT_ID, interaction.token, { embeds: [...], file: {...} });
```
```ts
// budget.ts:293-295 — catch path, itself unguarded
await editOriginalResponse(env.DISCORD_CLIENT_ID, interaction.token, {
  embeds: [errorEmbed(t.t('common.error'), errorMessage)],
});
```
```ts
// apps/discord-worker/src/utils/discord-api.ts:155-162 — raw fetch Response returned; can also throw on timeout
return fetch(url, {
  method: 'PATCH',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(body),
  signal: AbortSignal.timeout(DISCORD_WEBHOOK_TIMEOUT),
});
```

## Suggested Fix
One shared helper used by all background processors, for both success and catch paths (the catch path becomes throw-safe by construction):
```ts
async function safeEdit(appId: string, token: string, opts: FollowUpOptions, logger?: ExtendedLogger) {
  try {
    const res = await editOriginalResponse(appId, token, opts);
    if (!res.ok) {
      logger?.error('Follow-up edit failed', undefined, {
        status: res.status,
        body: await res.text().catch(() => ''),
      });
    }
  } catch (e) {
    logger?.error('Follow-up edit threw', e instanceof Error ? e : undefined);
  }
}
```

## Why It's Hidden
Mocked `fetch` in tests always resolves `200 OK`, so the discarded-Response path never matters in CI. Embed-size limits, token expiry, and webhook timeouts only occur with real Discord payload sizes and real latency; the user-visible symptom ("thinking…" forever) produces no server-side error trail because the failure itself was never inspected or logged.
