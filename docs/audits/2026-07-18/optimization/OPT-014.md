# [OPT-014]: retry() retries deterministic 4xx failures with full exponential backoff

## Impact
LOW

## Category
I/O latency — non-discriminating retry policy

## Location
- `packages/core/src/utils/index.ts:848-875` (`retry` retries every error type)
- `packages/core/src/services/APIService.ts:706-708` (HTTP status folded into a generic `Error`)
- Timing constants: `packages/core/src/constants/index.ts:111-112` (`UNIVERSALIS_API_RETRY_COUNT = 3`, `UNIVERSALIS_API_RETRY_DELAY = 1000`)

## Current Performance
`fetchWithTimeout` throws `new Error("HTTP 404: Not Found")` for any non-ok status, and `retry` retries **all** errors with exponential backoff. A 400/404 from Universalis (bad datacenter name, delisted item) is deterministic — retrying adds `1000 + 2000 = 3000 ms` of dead latency per failed item before failing identically, and triples pointless upstream load.

## Bottleneck Analysis
```ts
// APIService.ts:706-708 — status collapsed into a string, undiscriminable downstream
if (!response.ok) {
  throw new Error(`HTTP ${response.status}: ${response.statusText}`);
}
```
```ts
// utils/index.ts:857-872 — every error is retried
for (let i = 0; i < attempts; i++) {
  try {
    return await fn();
  } catch (error) {
    lastError = error instanceof Error ? error : new Error(String(error));
    ...
    if (i < attempts - 1) {
      await sleep(delayMs * Math.pow(2, i)); // Exponential backoff — even for 404
    }
  }
}
```
Only 429 / 5xx / network / timeout errors are plausibly transient; 4xx (except 429) are not.

## Proposed Optimization
1. Throw a typed error carrying the status from `fetchWithTimeout` (e.g., `class HttpError extends Error { constructor(public status: number, ...) }`).
2. Add an optional predicate to `retry` — default preserves current behavior for all existing callers:
```ts
export async function retry<T>(
  fn: () => Promise<T>,
  maxAttempts = 3,
  delayMs = 1000,
  logger?: Logger,
  shouldRetry: (error: unknown) => boolean = () => true
): Promise<T> { ... if (!shouldRetry(error)) throw lastError; ... }
```
3. APIService passes `(e) => !(e instanceof HttpError) || e.status === 429 || e.status >= 500`.

## Expected Improvement
~3 s of dead latency removed per deterministic failure (visible in Discord command response time when an item 404s); 3× → 1× upstream calls for permanently failing requests.

## Trade-offs
- Slightly wider `retry` signature (backward-compatible via default).
- Requires care that genuinely transient 4xx-ish cases (429) stay retryable — covered by the predicate.
- If Universalis ever misuses 4xx for transient conditions, those would no longer self-heal via retry.

## Benchmark Approach
Mock `FetchClient` returning a fixed 404: measure `getPriceData` wall time and fetch-call count before (≈3 s, 3 calls) vs after (≈0 ms, 1 call). Verify 429 and 503 still retry 3× with backoff, and network `TypeError`/`AbortError` behavior is unchanged.
