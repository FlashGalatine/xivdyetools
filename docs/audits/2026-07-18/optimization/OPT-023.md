# [OPT-023]: og-worker — O(n) dye lookups via `getAllDyes().find()` and redundant per-candidate LAB/ΔE work in the harmony scan

## Impact

LOW

## Category

Per-request CPU micro-waste (repeated linear scans + recomputed conversions)

## Location

- `apps/og-worker/src/services/svg/dye-helpers.ts:184-187` (`getDyeByItemId` — `getAllDyes().find(...)`)
- `apps/og-worker/src/og-data-generator.ts:88-100` (`getDyeInfo` — same pattern; called up to 4× per crawler request, comparison route up to 16×)
- `apps/og-worker/src/services/svg/harmony.ts:116-148` (per-candidate `ColorService.hexToLab` + `getDeltaE_Oklab` inside the per-target-hue loop)

## Current Performance

- `getAllDyes()` returns a **fresh array copy** on each call; both lookup helpers then linear-scan it. The comparison image route calls this up to 16 times per request → 16 array copies + 16 × O(136) scans.
- The harmony hue scan recomputes each candidate's LAB via `hexToLab` even though `DyeDatabase.initialize()` already precomputes `dye.lab` on every entry, and computes an OKLAB ΔE for **every candidate on every target hue** although only the winning candidate's ΔE is displayed: ~136 × 3 hue targets = ~400 ΔE computations where ≤4 are needed.

## Bottleneck Analysis

```ts
// dye-helpers.ts:184-187
export function getDyeByItemId(itemId: number): Dye | undefined {
  const allDyes = dyeService.getAllDyes();      // fresh copy per call
  return allDyes.find((d) => d.itemID === itemId);
}

// harmony.ts:128-137 (inside per-hue, per-candidate loop)
const candidateLab = ColorService.hexToLab(candidateDye.hex);   // dye.lab already exists
...
const delta = ColorConverter.getDeltaE_Oklab(dye.hex, candidateDye.hex); // computed even for losers
```

Absolute numbers are small (136 dyes), but this executes on a CPU-billed worker whose requests already pay wasm rasterization, and it repeats on every image request.

## Proposed Optimization

1. Build a module-scope `Map<number /* itemID */, Dye>` once in `dye-helpers.ts`; both `getDyeByItemId` and og-data-generator's `getDyeInfo` use it (O(1), zero array copies).
2. In the harmony scan: use `candidateDye.lab` directly; track only the best-`hueDiff` candidate per target hue and compute its ΔE **after** the loop selects the winner.

## Expected Improvement

Removes ~16 array copies + linear scans per comparison request and ~99% of the ΔE/LAB computations per harmony request. Estimated sub-millisecond savings per request — worthwhile only because it is free to take alongside REFACTOR-024's dye-service consolidation.

## Trade-offs

None. Pure equivalent-result restructuring; `dye.lab` is guaranteed populated by `DyeDatabase.initialize()`.

## Benchmark Approach

`vitest bench` on `generateHarmonyOG` and `generateComparisonOG` with fixed inputs, before/after (expect measurable relative drop in the pre-render portion; assert identical SVG output strings to prove equivalence).

## Status

**RESOLVED 2026-07-19** — module-scope `Map<itemID, Dye>` backs `getDyeByItemId` and og-data-generator's `getDyeInfo` (no more fresh-array copies + linear scans, up to 16× per comparison request); the harmony scan computes the displayed ΔE only for each target hue's winner (~99% fewer ΔE computations) with LAB conversions served from ColorConverter's LRU cache.
