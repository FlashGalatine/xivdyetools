# [OPT-004]: Per-isolate in-memory rate limiting for oauth /auth/* endpoints understates protection

## Impact
MEDIUM (security-adjacent)

## Category
I/O / Architecture (rate-limit backend choice)

## Location
- File: `apps/oauth/src/services/rate-limit.ts`
- Function: module singleton `limiter` (lines 35-38) with the middleware in `apps/oauth/src/index.ts:132-145`

## Current Performance
With the Durable Object path dead (see REFACTOR-006 — no binding, class unexported, flag unset), all `/auth/*` rate limiting runs through a per-isolate `MemoryRateLimiter`. Configured limits like "10/min per IP for /auth/discord" are actually 10/min *per isolate*: Cloudflare spreads traffic across many isolates and colos, and isolates recycle, resetting counters. Real attacker throughput against credential-stuffing/token-grinding targets is an unbounded multiple of the configured limit.

## Bottleneck Analysis
This is not a latency bottleneck but an effectiveness one: the limiter's state has narrower scope (isolate) than its threat model (global per-IP). The in-memory backend is only meaningful for single-isolate abuse patterns; distributed or patient attackers never hit it.

## Proposed Optimization
Two options via existing infrastructure:
1. **Enable the DO limiter** (preferred; see REFACTOR-006): per-IP Durable Object gives globally consistent counters. The wrapper `checkRateLimitDO` already exists and fails open on DO errors.
2. **KV/Upstash backend:** `@xivdyetools/rate-limiter` already ships KV and Upstash backends; auth endpoints are low-QPS, so eventual-consistency imprecision of KV is acceptable for brute-force limiting.

## Expected Improvement
Configured limits become globally meaningful: a distributed attacker is capped near 10-30 req/min per IP as designed, rather than 10-30 per isolate. No measurable latency cost for legitimate users (auth flows are seconds-long human interactions; +1-10 ms per request is invisible).

## Trade-offs
- DO: one extra sub-request per auth request, small DO duration cost; must fix the DO's eternal-alarm reschedule first (REFACTOR-006) or every IP ever seen incurs a perpetual 2-minute wake.
- KV: write amplification on hot IPs and 60s-ish consistency lag — acceptable at these limits, but less precise than DO.

## Benchmark Approach
In a preview deployment, replay N req/min from one IP through `wrangler dev --remote`-spread traffic (or synthetic multi-colo runners) and record how many pass with (a) memory backend, (b) DO/KV backend; also measure added p50 latency per auth request before/after.
