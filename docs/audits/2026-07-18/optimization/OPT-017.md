# [OPT-017]: getPresets window-function total reports 0 for out-of-range pages

## Impact
LOW

## Category
I/O / Correctness nuance of a prior optimization (COUNT(*) OVER())

## Location
- File: `apps/presets-api/src/services/preset-service.ts`
- Function: `getPresets` (lines 177-201; total extraction at 190-192)

## Current Performance
A prior optimization merged the count query into the page query via `COUNT(*) OVER() as _total`, reducing 2 round trips to 1. However, the window-function total is only readable when at least one row is returned. For a `page` beyond the last page, `rows.length === 0` → `total` falls back to `0` and `has_more = false` — so a client paginating past the end sees the collection "shrink" to zero instead of the stable total. Clients that persist/display `total` after over-paging show wrong counts; naive "while page*limit < total" pagination loops self-correct only by accident.

## Bottleneck Analysis
The single-query design is correct and fast for all in-range pages; the defect is confined to the empty-result path where the window function has no row to ride on. Restoring accuracy needs a second query only on that rare path.

## Proposed Optimization
```ts
const rows = result.results || [];
let total = rows.length > 0 ? rows[0]._total : 0;

// Out-of-range page: fall back to a real count so total stays stable
if (rows.length === 0 && page > 1) {
  const countRow = await db
    .prepare(`SELECT COUNT(*) as total FROM presets WHERE ${whereClause}`)
    .bind(...params)
    .first<{ total: number }>();
  total = countRow?.total ?? 0;
}
```
(Alternative: document `total: null` on empty pages — cheaper but pushes handling onto every client.)

## Expected Improvement
Stable, correct `total`/`has_more` on all pages; zero extra cost on the hot path (extra COUNT executes only when a client over-pages, which is rare and already an edge flow).

## Trade-offs
One additional D1 query on the empty-page path; duplicated WHERE clause between the two statements (kept in sync via the shared `whereClause`/`params` variables already in scope).

## Benchmark Approach
Functional: seed >20 presets, request `page=999` and assert `total` matches the page-1 total. Performance: confirm via D1 analytics that in-range requests still issue exactly one statement.
