# [OPT-010]: `StorageService.isAvailable()` performs a localStorage write probe on every access

## Impact
MEDIUM

## Category
Redundant I/O on hot path (synchronous localStorage writes)

## Location
- `apps/web-app/src/services/storage-service.ts:24-33` (the probe)
- Called from every public method: `:40`, `:71`, `:100`, `:117`, `:134`, `:159`, `:187`, `:204`, `:229`, `:246`, `:520`, …

## Current Performance
Every `getItem`/`setItem`/`hasItem`/`removeItem`/`getKeys`/`getItemsByPrefix`/… first executes:

```ts
// storage-service.ts:24-33
static isAvailable(): boolean {
  try {
    const test = `${STORAGE_PREFIX}_test`;
    localStorage.setItem(test, 'test');   // synchronous, disk-backed write
    localStorage.removeItem(test);        // + a second write
    return true;
  } catch { return false; }
}
```

Two synchronous localStorage **writes** per storage access. A single tool mount performs dozens of storage reads (ConfigController lazy loads, collections/favorites, persisted tool state), i.e. hundreds of probe writes per navigation.

## Bottleneck Analysis
localStorage writes are the slow path: synchronous, disk-backed, and they fire cross-tab `storage` events. Notably `auth-service.ts:207` registers a `window 'storage'` listener, so every probe write makes **other open tabs** wake up and run that filter. The probe cost dwarfs the actual read it guards (`localStorage.getItem` is comparatively cheap), and it runs even for pure read operations.

## Proposed Optimization
Memoize the probe result; invalidate only on a caught write failure:

```ts
private static available: boolean | null = null;

static isAvailable(): boolean {
  if (this.available !== null) return this.available;
  try {
    const test = `${STORAGE_PREFIX}_test`;
    localStorage.setItem(test, 'test');
    localStorage.removeItem(test);
    this.available = true;
  } catch {
    this.available = false;
  }
  return this.available;
}

// In setItem's catch: on QuotaExceeded/SecurityError, optionally set this.available = null
// so the next call re-probes.
```

## Expected Improvement
Eliminates ~2 synchronous writes per storage access (hundreds per navigation, thousands per session) and removes the cross-tab `storage`-event noise that currently pings the auth listener in every other tab. Micro-benchmarks put a localStorage write at ~0.05-0.5 ms depending on platform — this removes an easy 10-100 ms of cumulative jank per tool switch on slower devices.

## Trade-offs
A backend that *becomes* unavailable mid-session (private-mode quirks, quota exhaustion) is detected on the next failed real write instead of pre-flight — the existing try/catch in `setItem`/`getItem` already handles that failure gracefully, so behavior degrades identically, just one operation later.

## Benchmark Approach
Wrap a tool navigation (e.g. Harmony → Budget) in `performance.mark`/`measure` before and after the change; additionally count `localStorage.setItem` invocations via a dev-only monkey-patch. Verify cross-tab: open two tabs, count `storage` events received in tab B during tab A's navigation (should drop to near zero).

## Status

**RESOLVED 2026-07-19** — `isAvailable()` memoizes the probe (with `resetAvailabilityCache()` for tests/recovery); the two synchronous localStorage writes per storage access and the cross-tab storage-event noise are gone.
