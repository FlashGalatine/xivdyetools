# [OPT-007]: Preset-favorites autocomplete fires up to 50 service-binding subrequests per keystroke, with no autocomplete rate limit

## Impact
MEDIUM

## Category
Redundant API calls / fan-out / missing rate limiting

## Location
- `apps/discord-worker/src/index.ts:871-896` (`getFavoritedPresetsAutocompleteChoices`)
- `apps/discord-worker/src/services/preset-favorites.ts:26` (`MAX_PRESET_FAVORITES = 50`)
- Contrast: autocomplete rate limiting exists only in moderation-worker (`apps/moderation-worker/src/index.ts:272-295`); discord-worker's `handleAutocomplete` (`index.ts:597-725`) has none

## Current Performance
Every keystroke in `/preset favorite remove` triggers 1 KV read plus up to **50 parallel `PRESETS_API.fetch` calls** â€” each a full HMAC sign (`crypto.subtle.importKey` + `sign`) on the caller side and a D1 query on the presets-api side. Discord fires autocomplete on nearly every keypress: a user with a full favorites list typing a 10-character filter â‰ˆ **500 subrequests** for one field.

## Bottleneck Analysis
- N+1 resolution: favorites are stored as bare preset IDs, so displaying names requires one preset fetch per ID, per keystroke, with no caching between keystrokes.
- No throttle: discord-worker applies `checkRateLimit` only to commands (`index.ts:462-476`), never to the autocomplete interaction type â€” unlike moderation-worker, which caps autocomplete at 60/min.
- The 50-way fan-out also burns against per-request subrequest budgets; combined with anything else that fans out, this is the closest path to the cap.

## Proposed Optimization
1. **Denormalize:** store `{ id, name }` pairs in the KV favorites value (captured at favorite-add time, where the preset object is already in hand). Autocomplete then needs **zero** service-binding calls; resolve the authoritative preset only on selection.
2. Or add a bulk endpoint to presets-api (`GET /api/v1/presets?ids=a,b,c`) â†’ 1 subrequest per keystroke.
3. Add a lightweight autocomplete rate limit to discord-worker's `handleAutocomplete`, mirroring moderation-worker's (fail-soft: return empty choices when limited).

## Expected Improvement
~50 subrequests/keystroke â†’ 0-1 (â‰¥98% reduction); autocomplete latency drops from a parallel-fetch tail (slowest of 50) to a single KV read; presets-api D1 load from this path effectively disappears.

## Trade-offs
- Denormalized names go stale if a preset is renamed â€” acceptable for autocomplete display; the selected value is still the ID and is re-resolved on use. A rename hook could refresh entries lazily.
- Bulk endpoint requires a presets-api change and coordinated deploy.
- New rate limit adds one KV read per keystroke (or reuse the Upstash backend already configured for commands).

## Benchmark Approach
Count subrequests per autocomplete invocation (wrangler tail / Workers analytics) for a 50-favorite user typing a 10-char query, before/after; measure autocomplete p95 response time (Discord requires < 3 s) and presets-api request volume attributable to `/presets/{id}` GETs.
