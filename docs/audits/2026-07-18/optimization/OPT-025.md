# [OPT-025]: api-worker — static endpoint responses recomputed on every request

## Impact

LOW

## Category

Per-request CPU / memoization of pure-static computations

## Location

- `apps/api-worker/src/routes/dyes.ts:75-84` (`GET /categories` — runs `dyeService.searchByCategory(category)` per category per request)
- `apps/api-worker/src/routes/dyes.ts:128-155` (`GET /consolidation-groups` — refilters all 136 dyes into A/B/C groups + unconsolidated per request)
- `apps/api-worker/src/lib/validation.ts:369-373` (`buildFilterExcludeIds` — rescans all dyes per `/v1/match/closest` request with active filters)

## Current Performance

All three computations are pure functions of the static, immutable dye database (plus `isConsolidationActive()`, which is a date comparison). They are re-executed from scratch on every request even though their outputs never change within an isolate's lifetime (and for consolidation, only change at the activation-date boundary). With 136 dyes the absolute cost is small (tens of microseconds), and `s-maxage=86400` means the edge absorbs most production traffic — this is an origin-CPU tidiness win, not a latency fix.

## Bottleneck Analysis

```ts
// routes/dyes.ts:75-80 — N categories × O(n) scans per request
const data = categories.map((category) => ({
  name: category,
  count: dyeService.searchByCategory(category).length,
}));

// lib/validation.ts:369-373 — full DB scan per filtered match request
export function buildFilterExcludeIds(filters: DyeQueryFilters): number[] {
  if (!hasActiveDyeFilters(filters)) return [];
  const allDyes = dyeService.getAllDyes();          // fresh copy
  return allDyes.filter((d) => !dyeMatchesFilters(d, filters)).map((d) => d.id);
}
```

## Proposed Optimization

- `/categories`: compute the payload once at module scope (data is static).
- `/consolidation-groups`: memoize keyed by the `isConsolidationActive()` boolean so the response flips correctly at the activation date.
- `buildFilterExcludeIds`: memoize by the filter tuple — 8 tristate flags ⇒ at most 3^8 keys, realistically a handful; a tiny `Map<string, number[]>` keyed by `JSON.stringify(filters)` suffices.

```ts
const filterExcludeCache = new Map<string, number[]>();
export function buildFilterExcludeIds(filters: DyeQueryFilters): number[] {
  if (!hasActiveDyeFilters(filters)) return [];
  const key = JSON.stringify(filters);
  let ids = filterExcludeCache.get(key);
  if (!ids) {
    ids = dyeService.getAllDyes().filter((d) => !dyeMatchesFilters(d, filters)).map((d) => d.id);
    filterExcludeCache.set(key, ids);
  }
  return ids;
}
```

## Expected Improvement

Eliminates repeated O(n·categories) and O(n) scans on warm isolates — micro-level CPU savings per request; primarily reduces allocation churn.

## Trade-offs

- Memoized arrays must be treated as immutable by callers (they currently are — passed into `Set` construction / spread).
- Consolidation memo must key on `isConsolidationActive()` to honor the date flip; a plain unconditional cache would freeze pre-activation output. (Consolidation activated 2026-04-28, so today the flag is stably `true` — still key on it for correctness.)

## Benchmark Approach

`vitest bench` invoking the three functions 10k times before/after; assert deep-equal outputs. Production-wise, compare Workers CPU-time percentiles for `/v1/dyes/categories` before/after (expect small but visible drop on cache-miss traffic).
