# [OPT-020]: `DelegatingLogger.time()` delegates to the parent, losing child context on timing logs

## Impact
LOW (observability quality; borderline bug)

## Category
Logging/tracing fidelity (context propagation in the delegation chain)

## Location
`packages/logger/src/core/base-logger.ts:317-323` (`DelegatingLogger.time`/`timeAsync`); the emitting call at `:255-264` (`BaseLogger.time` → `this.debug(...)`)

## Current Performance
`child(context)` returns a `DelegatingLogger` whose `debug/info/warn/error` correctly merge the child's context into every entry (L292-306, 325-327). But the timing methods bypass that merge entirely:

```ts
// packages/logger/src/core/base-logger.ts:317-323
time(label: string): () => number {
  return this.parent.time(label);          // parent emits — child context never merged
}

async timeAsync<T>(label: string, fn: () => Promise<T>): Promise<T> {
  return this.parent.timeAsync(label, fn);
}
```
`BaseLogger.time`'s closure logs via the **parent's** `debug` (L261: `this.debug(...)` where `this` is the parent), so the emitted entry carries only global context — not the child's `requestId`/`operation`/`userId`.

## Bottleneck Analysis
The whole point of the worker middleware stack is request correlation: `loggerMiddleware` creates per-request loggers keyed by `requestId`, and `child({ requestId })` is the documented pattern (`base-logger.ts:129-133`). Any code that does:

```ts
const reqLogger = logger.child({ requestId });
const end = reqLogger.time('d1-query');
await db.prepare(...).all();
end();   // → "d1-query: 12.34ms" WITHOUT requestId
```
produces timing lines that cannot be joined to their request in aggregated Cloudflare logs — precisely the entries (slow queries, slow renders) one most wants correlated. The data is silently degraded; nothing fails.

## Proposed Optimization
Implement the timing methods locally on `DelegatingLogger` so emission goes through the child's own (context-merging) `debug`:

```ts
time(label: string): () => number {
  const start = typeof performance !== 'undefined' ? performance.now() : Date.now();
  return () => {
    const end = typeof performance !== 'undefined' ? performance.now() : Date.now();
    const duration = end - start;
    this.debug(`${label}: ${duration.toFixed(2)}ms`, { duration, label }); // merges childContext
    return duration;
  };
}

async timeAsync<T>(label: string, fn: () => Promise<T>): Promise<T> {
  const end = this.time(label);
  try { return await fn(); } finally { end(); }
}
```
(Alternatively, refactor `BaseLogger.time` to accept an emit callback and share one implementation — avoids duplicating the timer body.)

## Expected Improvement
Timing entries carry full child context (`requestId` etc.), making duration logs filterable/joinable per request in log aggregation. No runtime cost change — same number of `performance.now()` calls and log writes.

## Trade-offs
~15 lines duplicated from `BaseLogger.time` unless the shared-emit refactor is done. Log *content* changes (entries gain context fields) — only a concern if any dashboard parses timing lines positionally, which structured JSON logging makes unlikely.

## Benchmark Approach
Not performance-sensitive; verify with a unit test: create `child({ requestId: 'abc' })`, call `time('x')()` with a spy adapter, assert the captured entry's `context.requestId === 'abc'` (currently fails, passes after fix).
