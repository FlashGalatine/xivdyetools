# [OPT-013]: Preset mutations re-fetch the row they just wrote ŌĆö use RETURNING and batch

## Impact
MEDIUM

## Category
I/O (redundant round trips; write-then-read races)

## Location
- File: `apps/presets-api/src/services/preset-service.ts`
- Functions: `updatePresetStatus` (311-324, UPDATE then `getPresetById`), `updatePreset` (436-437, UPDATE then `getPresetById`), `revertPreset` (465-478, UPDATE then `getPresetById`)
- Also: `apps/presets-api/src/handlers/moderation.ts:68-86` ŌĆö the status route chains `getPresetById` + log INSERT + UPDATE + re-fetch = 4 sequential D1 round trips

## Current Performance
Every preset mutation performs a write followed by a full re-read of the same row, doubling D1 round trips per mutation. The moderation status route stacks four sequential statements. Beyond latency, the re-read races with concurrent writers: the entity returned to the caller may reflect *someone else's* subsequent write rather than this request's.

## Bottleneck Analysis
D1 round-trip latency dominates these tiny statements; halving statement count roughly halves the D1-bound portion of mutation latency. The pattern exists only because the service returns the updated `CommunityPreset` ŌĆö which `RETURNING` can supply directly. D1 supports `RETURNING`, and this codebase already uses it (`votes.ts:61`).

## Proposed Optimization
```ts
// updatePresetStatus
const row = await db.prepare(
  'UPDATE presets SET status = ?, updated_at = ? WHERE id = ? RETURNING *'
).bind(status, now, id).first<PresetRow>();
return row ? rowToPreset(row) : null;
```
Apply the same to `updatePreset` (append `RETURNING *` to the dynamic UPDATE) and `revertPreset`. Combine with BUG-020's fix so the moderation route becomes a single `db.batch([conditionalStatusUpdateReturning, logInsert])` ŌĆö 4 round trips ŌåÆ 2 (initial read for validation + atomic batch).

## Expected Improvement
~50% fewer D1 statements on all preset mutations; lower tail latency for moderation actions; returned entities are guaranteed to reflect exactly this request's write (eliminates the read-back race).

## Trade-offs
`RETURNING *` rows still pass through `rowToPreset` JSON parsing (no change). Rows matched-but-unchanged vs not-found must be distinguished via the returned row being null ŌĆö semantics match the current `getPresetById`-null behavior. Slightly less obvious code for the dynamic-SET builder.

## Benchmark Approach
Time `PATCH /moderation/:id/status` and `PATCH /presets/:id` end-to-end (p50/p95) in a preview deployment before/after; alternatively count statements via D1 analytics for the same request volume.
