# [OPT-011]: Palette extraction runs K-means synchronously on the main thread (loading state never paints)

## Impact
MEDIUM

## Category
Main-thread blocking / perceived responsiveness

## Location
- `apps/web-app/src/components/extractor-tool.ts:3085-3165` (`extractPalette`)
- `apps/web-app/src/components/extractor-tool.ts:3171-3200+` (`findColorPositions` full-image scan)
- Auto-run on init: `:437-441` (`restoreSavedImage` → `void this.extractPalette()`)

## Current Performance
`extractPalette` is declared `async` but contains **no await**. It sets the button to "Extracting…" and disables it (`:3101-3105`), then immediately and synchronously runs:
1. `ctx.getImageData(0, 0, canvas.width, canvas.height)` over the full canvas (`:3109`);
2. `PaletteService.pixelDataToRGBFiltered` (per-pixel array build);
3. `extractAndMatchPalette` — K-means++ clustering + dye matching from `@xivdyetools/core` (`:3118-3120`);
4. `findColorPositions` — another full-image scan (`:3128`).

Because there is no yield between the DOM write and the heavy work, the browser never gets a frame: the "Extracting…" state and disabled styling **never paint**. On large images the tab freezes for the whole duration. The same blocking work also runs unprompted during tool init whenever a saved image is restored from storage (`:441`), delaying the tool's first paint.

## Bottleneck Analysis
The dominant costs are the per-pixel array construction and K-means iterations, both O(pixels × iterations) on the main thread. A 4K screenshot is ~8.3 M pixels → tens of millions of operations before the next frame. `findColorPositions` already demonstrates the mitigation (grid sampling `step` at `:3183`) but the K-means input is not downsampled.

## Proposed Optimization
Tiered:
1. **Stopgap (2 lines):** yield before the heavy section so the loading state actually renders:
   ```ts
   this.extractPaletteBtn.textContent = LanguageService.t('matcher.extractingPalette');
   ...
   await new Promise<void>((r) => requestAnimationFrame(() => r()));
   const imageData = ctx.getImageData(...);
   ```
2. **Downsample:** sample every Nth pixel when building the K-means input (mirror the `step` heuristic from `findColorPositions:3183`) — order-of-magnitude less work with negligible palette-quality loss.
3. **Full fix:** move pixel filtering + K-means into a Web Worker, transferring the `ImageData` buffer (zero-copy) and posting back matches; keep dye matching on the main thread or ship the dye table to the worker.

## Expected Improvement
- Stopgap: honest UI feedback (spinner/disabled state visible), no throughput change.
- Downsampling: ~10× reduction in blocking time (e.g. a 2-3 s freeze on a 4K image → ~200-300 ms).
- Worker: main thread stays interactive throughout; extraction time itself unchanged but perceived hang eliminated, including during tool init with a restored image.

## Trade-offs
Worker adds a build artifact, a message protocol, and duplicate access to the dye database (bundle cost or message payload). Downsampling introduces a small chance of missing tiny accent colors — mitigable by flooring the sample step for small images. The rAF-yield stopgap keeps the freeze but restores feedback honesty; ship it first.

## Benchmark Approach
Instrument with `performance.now()` around steps 1-4 for representative images (500 px, 1080p, 4K). Measure Long Task entries (`PerformanceObserver({ type: 'longtask' })`) before/after; success criterion for the worker variant: zero long tasks > 50 ms attributable to extraction, and the "Extracting…" state visible in a frame capture.
