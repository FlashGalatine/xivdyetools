# [OPT-019]: `base64UrlEncodeBytes` builds its intermediate string with per-byte concatenation

## Impact
LOW

## Category
Algorithmic / memory (string building in a public utility)

## Location
`packages/crypto/src/base64.ts:39-45` (`base64UrlEncodeBytes`); symmetric decode loop at `:58-78` is array-indexed and fine

## Current Performance
```ts
// packages/crypto/src/base64.ts:39-45
export function base64UrlEncodeBytes(bytes: Uint8Array): string {
  let binary = '';
  for (let i = 0; i < bytes.length; i++) {
    binary += String.fromCharCode(bytes[i]);   // one JS string append per byte
  }
  return btoa(binary).replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}
```
One `String.fromCharCode` call plus one string append per byte. Engines mitigate with ropes, but per-byte call overhead dominates for larger inputs, and pathological engine behavior on very large accumulations is possible.

For the dominant in-repo workloads this is irrelevant: HMAC-SHA256 signatures (32 bytes, `packages/auth/src/hmac.ts:128`) and JWT header/payload segments (hundreds of bytes). But the function is the published `@xivdyetools/crypto` package's general-purpose encoder, so arbitrary-size payloads (file bodies, image buffers) are legitimate future inputs.

## Bottleneck Analysis
- ≤1 KB inputs: negligible either way (~microseconds).
- 100 KB-1 MB inputs: chunked `String.fromCharCode.apply` conversion is typically ~10-50× faster than per-byte appends, and avoids one transient string per byte.
- A naive single `String.fromCharCode(...bytes)` would be *wrong* to suggest — it overflows the argument/stack limit around ~64-128k elements — hence chunking.

## Proposed Optimization
Chunked conversion, keeping the identical output:

```ts
export function base64UrlEncodeBytes(bytes: Uint8Array): string {
  const CHUNK = 0x8000; // 32k — safely under engine argument limits
  const parts: string[] = [];
  for (let i = 0; i < bytes.length; i += CHUNK) {
    parts.push(String.fromCharCode.apply(null,
      bytes.subarray(i, i + CHUNK) as unknown as number[]));
  }
  return btoa(parts.join(''))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}
```
(When the runtime baseline allows, `bytes.toBase64({ alphabet: 'base64url', omitPadding: true })` — the Uint8Array.prototype.toBase64 proposal already shipping in V8 — replaces the whole body; guard with a feature check.)

## Expected Improvement
~10-50× faster encoding for KB+ payloads; unmeasurable for the current 32-byte signature path. Reduced GC pressure from intermediate strings.

## Trade-offs
Slightly less obvious code (chunk constant, `apply` cast). Zero output change — encode/decode round-trip tests in `base64.test.ts` cover it. If callers are contractually guaranteed small inputs forever, keeping the current shape is defensible — this is an opportunistic cleanup, not a hotspot fix.

## Benchmark Approach
`vitest bench` encoding 32 B / 10 KB / 1 MB `Uint8Array`s, old vs new implementation; assert identical output strings across random buffers (property test with round-trip through `base64UrlDecodeBytes`).
