# [OPT-029]: maintainer: XIVAPI locale fetches run sequentially (worst case 40 s)

## Impact
LOW

## Category
Serialized network I/O (maintainer dev tool)

## Location
- `apps/maintainer/src/services/xivapiService.ts:34-68` (`fetchItemNames` sequential loop)
- Timeout wrapper: `apps/maintainer/src/utils/fetchWithTimeout.ts:36-68` (10 s per request as invoked at `:37-41`)

## Current Performance
`fetchItemNames` awaits each of the four locale requests (`en`, `ja`, `de`, `fr`) one at a time in a `for` loop, each with a 10 s timeout:

```ts
// xivapiService.ts:34-41
for (const lang of SUPPORTED_LANGUAGES) {
  try {
    const response = await fetchWithTimeout(
      `${XIVAPI_BASE}/sheet/Item?rows=${itemId}&language=${lang}`,
      {},
      10000
    )
    ...
```
Happy path pays 4× round-trip latency (~1-2 s total on a typical connection); a slow or unreachable XIVAPI stalls the "Fetch Names" form action for up to **40 s** (4 × 10 s timeouts) before the user sees the collected errors.

## Bottleneck Analysis
The four requests are fully independent — no request depends on another's result; the shared `result` object is only aggregated after each response. Serialization is pure accident of the loop structure. The per-language error strings are already independent (`result.errors.push(...)` per lang).

## Proposed Optimization
Fan out with `Promise.allSettled`, preserving the per-language error collection:

```ts
const outcomes = await Promise.allSettled(
  SUPPORTED_LANGUAGES.map(async (lang) => {
    const response = await fetchWithTimeout(
      `${XIVAPI_BASE}/sheet/Item?rows=${itemId}&language=${lang}`, {}, 10000)
    if (!response.ok) throw new Error(`HTTP ${response.status}`)
    const data: XivapiItemResponse = await response.json()
    if (!data.rows?.length) throw new Error('No data found')
    return { lang, name: data.rows[0].fields.Name }
  })
)
for (let i = 0; i < outcomes.length; i++) {
  const lang = SUPPORTED_LANGUAGES[i]
  const o = outcomes[i]
  if (o.status === 'fulfilled') {
    let name = o.value.name
    const prefix = dyePrefixes[lang]
    if (prefix && name) name = stripDyePrefix(name, prefix)
    result.names[lang] = name
    result.autoFilled.push(lang)
  } else {
    result.errors.push(`Failed to fetch ${lang}: ${o.reason?.message ?? o.reason}`)
  }
}
```

## Expected Improvement
Total latency drops from the **sum** of the four requests to the **max** of them: ~4× faster on the happy path, and worst case falls from 40 s to 10 s when XIVAPI is down.

## Trade-offs
Four concurrent hits to XIVAPI instead of one at a time — negligible for a local dev tool fetching a single item on demand; XIVAPI v2 tolerates this comfortably. Error message ordering stays deterministic via the index loop.

## Benchmark Approach
Time `fetchItemNames` (console.time) for a real item on a normal connection (expect ~4× reduction), and with XIVAPI blocked via devtools request blocking (expect 10 s instead of 40 s).
