# [OPT-008]: Analytics performs ~10-12 KV operations per command; verification-read retry logic is a no-op

## Impact
MEDIUM

## Category
KV quota / redundant I/O

## Location
`apps/discord-worker/src/services/analytics.ts:127-171` (`incrementCounter`), `:197-209` (`trackUniqueUser`), `:214-230` (`trackCommandWithKV`)

## Current Performance
Per command, `trackCommandWithKV` runs three `incrementCounter`s (`total`, `cmd:{name}`, `success|failure`) plus `trackUniqueUser`. Each `incrementCounter` performs `getWithMetadata` → `put` → verification `get` (3 ops); `trackUniqueUser` performs `get` (+ `put` once per user per day). Total: **~9-11 KV ops, of which 3 are writes, per command** — all inside `ctx.waitUntil`, so latency is hidden but quota is not. On the free tier's 1,000 writes/day this saturates at ~333 commands/day, after which **all** KV writes in the namespace (favorites! preferences! collections!) fail for the rest of the day.

## Bottleneck Analysis
The verification read never does what it intends:

```ts
// analytics.ts:147-166
await kv.put(fullKey, String(newValue), { expirationTtl: STATS_TTL, metadata: {...} });

// Read back to verify (simple optimistic check)
const verification = await kv.get(fullKey);
const verifiedValue = parseInt(verification || '0', 10);
if (verifiedValue >= newValue) {
  return;                       // <-- reading your own just-written value: essentially always true
}
```
A `kv.get` immediately after your own `kv.put` from the same isolate observes your own write, so `verifiedValue >= newValue` virtually always holds — the retry loop and backoff (`:158-166`) are dead logic that only add one extra read per counter. Cross-colo lost increments remain lost (acknowledged in comments; Analytics Engine is the accurate store anyway).

## Proposed Optimization
1. **Delete the verification read** — saves 3 reads/command with zero behavior change.
2. **Collapse the three counters into one daily-bucketed JSON key** (`stats:daily:{date}` holding `{total, success, failure, cmd: {...}}`) — 1 RMW instead of 3, cutting writes per command from 3 to 1. Races are already accepted for these approximate counters.
3. Longer term: rely wholly on Analytics Engine (`trackCommand` already writes it) and serve `/stats` from AE's SQL API, removing KV counters entirely.

## Expected Improvement
Steps 1+2: ~10 KV ops → ~3 per command (1 write). Free-tier write headroom triples (~1,000 commands/day before saturation), and the failure blast radius on user-data writes shrinks accordingly.

## Trade-offs
- Daily-bucket key changes `getStats` aggregation (sum across day keys for 30-day totals) — moderate code churn, needs a migration/fallback read of old keys (a fallback pattern already exists at `:287-291`).
- AE-only requires an API token + external SQL call from `/stats`, and AE data lags slightly.

## Benchmark Approach
Instrument KV operation counts per command (wrap the namespace or count in tests); track daily KV write totals in the Cloudflare dashboard before/after; verify `/stats` output parity against Analytics Engine numbers for a comparison week.
