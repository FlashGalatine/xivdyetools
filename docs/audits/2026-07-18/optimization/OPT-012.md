# [OPT-012]: Extractor persists up to 2 MB image data-URL into localStorage, risking global quota exhaustion

## Impact
MEDIUM

## Category
Storage architecture / quota risk

## Location
- `apps/web-app/src/components/extractor-tool.ts:80-81` (`MAX_IMAGE_STORAGE_SIZE = 2 * 1024 * 1024`)
- Writes: `:236-243` (image-loaded event) and `:1328-1334` (drop zone)
- Read/restore: `:365-368`
- Failure mode: `apps/web-app/src/services/storage-service.ts:82-92` (quota errors only `logger.warn`/`error` + `return false`)

## Current Performance
Uploaded images up to 2 M **string characters** are stored as a base64 data-URL under `v3_matcher_image`. localStorage quota is ~5 MB and commonly accounted in UTF-16 code units — a 2 M-char string can consume ~4 MB of the budget, i.e. up to ~80 % of the entire app's storage, which is shared with tool configs, collections/favorites, the auth token, saved palettes, and the SecureStorage cache (whose own ceiling is 5 MB, `storage-service.ts:378`). The write itself is also a multi-MB synchronous string serialization on the image-load path.

## Bottleneck Analysis
When quota trips, **every subsequent** `StorageService.setItem` fails: `QuotaExceededError` is caught and logged but callers get `false` and mostly ignore it — config changes, new favorites, and collection edits silently stop persisting. The image key is the single largest tenant by an order of magnitude, so it is the practical cause of any quota exhaustion. Meanwhile the app already ships an IndexedDB wrapper (`services/indexeddb-service.ts`) used for price caching, which has no such practical quota constraint.

## Proposed Optimization
Move image persistence to IndexedDB:
1. Add an `IMAGE_CACHE` store to `indexeddb-service.ts` (same pattern as `PRICE_CACHE`).
2. Store the image as a `Blob` (from the original `File`/canvas) rather than a base64 string — ~25 % smaller and no encode/decode cost.
3. Restore path becomes async: `const blob = await indexedDBService.get(...); img.src = URL.createObjectURL(blob);` — the existing `restoreSavedImage` already tolerates async decode via `img.onload`/`onerror` (`:428-450`); revoke the object URL in `onload`.
4. One-time migration: on init, if `v3_matcher_image` exists in localStorage, move it to IndexedDB and delete the key.

## Expected Improvement
- Frees up to ~4 MB (~80 %) of the localStorage budget; eliminates the silent-persistence-failure mode for configs/collections.
- Removes a multi-MB synchronous localStorage write from the image-load path (image saves become async and off the critical path).
- Allows raising the persisted-image size cap if desired (IndexedDB quotas are orders of magnitude larger).

## Trade-offs
- Restore is asynchronous — a one-frame delay before the saved image reappears on tool load (imperceptible; extraction already dominates).
- IndexedDB can be unavailable (private modes); keep the existing "skip persistence" fallback rather than falling back to localStorage.
- Small migration code carried for a release or two.

## Benchmark Approach
Before/after: fill the extractor with a ~1.9 MB data-URL image, then measure `StorageService.getSize()` and attempt config/collection writes — verify no `QuotaExceededError` after the change. Measure image-load handler duration (`performance.mark`) to confirm removal of the synchronous write cost.

## Status

**RESOLVED 2026-07-19** — images persist to a new `image_cache` IndexedDB store (DB v2), with a one-time migration of the legacy `v3_matcher_image` localStorage key on first restore (freeing up to ~4 MB / ~80% of the shared quota and ending the silent QuotaExceededError cascade for configs/collections). Writes are async and off the image-load critical path; the size cap rose 2 MB → 8 MB since it now only bounds memory/extraction cost. Stored as data-URL string rather than Blob for a minimal-risk change (Blob conversion noted as optional follow-up).
