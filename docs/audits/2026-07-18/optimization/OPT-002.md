# [OPT-002]: KVRateLimiter — drop the post-put verification read (one fewer billed KV read per allowed request)

## Impact
MEDIUM

## Category
I/O reduction / billed-operation cost / latency (Cloudflare KV)

## Location
`packages/rate-limiter/src/backends/kv.ts:227-235` (verification read inside `increment()`); hot-path consumer: `apps/api-worker/src/middleware/rate-limit.ts:24-26` (KVRateLimiter on every `/v1/*` request)

## Current Performance
Every allowed request through `KVRateLimiter.check()` performs **three sequential KV operations**:

1. `kv.get` (checkOnly read, kv.ts:139)
2. `kv.getWithMetadata` + `kv.put` (increment read-modify-write, kv.ts:202, 222)
3. `kv.get` (post-put "verification", kv.ts:228)

That is 3 reads + 1 write per admitted request, executed serially — each a network round-trip within the colo.

```ts
// packages/rate-limiter/src/backends/kv.ts:227-235
// Verify write succeeded (simple optimistic check)
const verification = await this.kv.get(kvKey);
if (verification) {
  const verified: KVEntry = JSON.parse(verification) as KVEntry;
  if (verified.count >= entry.count) {
    return;
  }
}
// else: sleep 10-30ms and retry the whole read-modify-write
```

## Bottleneck Analysis
The verification read provides **no correctness benefit** (established in BUG-022, `bugs/BUG-022.md`):
- Lost updates pass the `verified.count >= entry.count` check anyway (two writers both writing 6 both "verify" successfully) — so it does not detect the races it exists for.
- When it *does* fail (concurrent window reset or stale read), the retry re-runs the entire read-modify-write and **double-counts** the request — actively harmful.
- On a verification miss it also adds a 10-30ms `setTimeout` backoff (kv.ts:238-241) to request latency.

Meanwhile it accounts for one third of the limiter's read volume on every single rate-limited request across api-worker traffic, directly consuming the KV free-tier read budget and adding one serial round-trip to p50 latency.

## Proposed Optimization
Delete lines 227-241 (verification read + backoff) and the now-unused version metadata plumbing (KVMetadata write at 222-225 — also dead per BUG-022). `increment()` becomes: read `getWithMetadata` (or plain `get`), compute, `put`, return; the retry loop remains solely for *thrown* KV errors, which is what it is actually good at handling.

## Expected Improvement
- **−33% KV reads** per admitted request (3 → 2), −25% total KV ops (4 → 3).
- One fewer serial round-trip: roughly 5-20ms off p50 for rate-limited endpoints (KV in-colo read latency), plus elimination of the occasional 10-30ms retry backoff.
- Removes the double-increment code path entirely (correctness win bundled with the cost win).

## Trade-offs
None in correctness — the check was cosmetic (see analysis in BUG-022). The class docblock's "optimistic concurrency" claims (kv.ts:8, 15, 190-192) must be rewritten to honestly describe best-effort fixed-window semantics, and the `maxRetries` option's meaning narrows to "retries on thrown KV errors." Anyone who believed the OCC comments loses nothing real.

## Benchmark Approach
1. **Unit:** count mock-KV operations per `check()` call before/after (test-utils MockKV `_store` access counts or a spy) — assert 3 → 2 reads.
2. **Integration:** wrangler dev + `hey`/`autocannon` against a KV-limited route; compare p50/p95 and Cloudflare dashboard KV read counts over an identical request volume.
3. **Regression:** existing kv.test.ts suite must pass unchanged except tests that specifically asserted the verification read.

## Status

**RESOLVED 2026-07-19 (with BUG-022)** — verification read + backoff + version metadata deleted; `increment()` is now read→compute→put (3 → 2 reads per admitted request, −25% total KV ops, one fewer serial round-trip), and the double-increment path is gone. Class docs rewritten to describe best-effort semantics; `maxRetries` now applies only to thrown KV errors.
