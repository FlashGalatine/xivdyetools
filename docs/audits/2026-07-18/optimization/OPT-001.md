# [OPT-001]: Batch price paths have no request deduplication (cold-cache stampede)

## Impact
MEDIUM

## Category
I/O — duplicate upstream requests / missing in-flight coalescing

## Location
- `packages/core/src/services/APIService.ts:959-1001` (`getPricesForItems`)
- `packages/core/src/services/APIService.ts:1013-1060` (`getPricesForDataCenter`)
- Contrast: single-item dedupe via `pendingRequests` at `packages/core/src/services/APIService.ts:529-550`
- Unused ready-made building block: `AsyncLRUCache.getOrCompute` at `packages/core/src/utils/index.ts:155-281`

## Current Performance
`getPriceData` carefully deduplicates concurrent identical requests through the `pendingRequests` map (the CORE-BUG-001/002 fix). The batch methods have no equivalent: two users triggering `/budget` in the same Worker isolate within the same cold-cache window each run the full cache-check pass plus `fetchBatchPriceData` for ~125 items — N identical upstream Universalis batch calls (2 requests per call after chunking) plus N sets of duplicate cache writes.

## Bottleneck Analysis
```ts
// APIService.ts:986-998 — no in-flight tracking anywhere in this path
this.logger.debug(`Fetching ${uncachedItemIDs.length} uncached items (${results.size} cached)`);
const batchResults = await this.fetchBatchPriceData(uncachedItemIDs);
const cacheWrites: Promise<void>[] = [];
for (const [itemID, priceData] of batchResults) { ... }
await Promise.all(cacheWrites);
```
The `pendingRequests` map is also invisible to the batch path, so a batch fetch and a concurrent single-item `getPriceData` for the same item both hit the API. The window is widest exactly when it hurts most: cold cache after deploy/TTL expiry, when commands cluster.

## Proposed Optimization
Coalesce identical in-flight batches:
- Key an in-flight map by `sorted(uncachedItemIDs).join(',') + ':' + (dataCenterID ?? 'universal')` and share the promise across concurrent callers, cleaning up on settle (mirror the CORE-BUG-002 delete-before-resolve pattern); **or**
- Route each 100-item chunk through `AsyncLRUCache.getOrCompute` (`utils/index.ts:155-281`) — a class built in this repo for exactly this purpose and currently unused by APIService.

## Expected Improvement
On cold cache with N concurrent commands: N identical Universalis batch calls → 1 (a strict N× reduction in upstream requests and cache writes for the overlap window). Also reduces 429 exposure in combination with the rate-limiter burst issue (BUG-046).

## Trade-offs
- Slightly more state to manage; the in-flight map must be cleaned on the failure path or a failed batch poisons subsequent calls (same hazard class the single-item path already solved).
- Exact-key coalescing only helps identical item sets; overlapping-but-different sets still fetch separately (acceptable — the dominant case is the identical full-dye-set query).

## Benchmark Approach
Test with a mock `FetchClient` counting requests: fire 5 concurrent `getPricesForItems(all136TradeableIds)` on a cold `MemoryCacheBackend`; assert upstream call count drops from 10 (5 callers × 2 chunks) to 2, and that all 5 callers receive identical complete Maps. Repeat with an injected failure to verify the in-flight entry is cleared.

> Source: evidence/core-analysis.md (2026-07-18 deep-dive, core area)

## Status

**RESOLVED 2026-07-19** — both batch paths coalesce identical in-flight batches via a `pendingBatchRequests` map keyed by sorted IDs + datacenter, cleaned up on settle; cache writes inside the shared promise are best-effort (BUG-011).
