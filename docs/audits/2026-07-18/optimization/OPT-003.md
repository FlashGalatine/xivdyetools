# [OPT-003]: storeCharacters — N+1 sequential inserts plus non-atomic delete/insert on the login critical path

## Impact
MEDIUM

## Category
I/O (N+1 D1 queries, missing batch)

## Location
- File: `apps/oauth/src/services/user-service.ts`
- Function: `storeCharacters` (lines 195-213); caller `apps/oauth/src/handlers/xivauth.ts:475-484`

## Current Performance
On every XIVAuth login with characters: one `DELETE` round trip, then one `INSERT` round trip **per character** in a sequential `for` loop. A 10-character account = 11 sequential D1 calls appended to the login critical path (each a network round trip from the worker to D1). Additionally, a failure mid-loop leaves the user with a partially-written character list until their next successful login (delete-then-insert is not atomic).

## Bottleneck Analysis
D1 statement latency is dominated by per-call round-trip overhead, not row work; issuing N+1 tiny statements serially multiplies that overhead linearly with character count, directly inflating p95 login latency for multi-character accounts. The sequential `await` in the loop prevents even parallelism.

## Proposed Optimization
```ts
export async function storeCharacters(db: D1Database, userId: string, characters: XIVAuthCharacter[]): Promise<void> {
  await db.batch([
    db.prepare('DELETE FROM xivauth_characters WHERE user_id = ?').bind(userId),
    ...characters.map((char) =>
      db.prepare(
        `INSERT INTO xivauth_characters (user_id, lodestone_id, name, server, verified)
         VALUES (?, ?, ?, ?, ?)`
      ).bind(userId, char.id, char.name, char.home_world, char.verified ? 1 : 0)
    ),
  ]);
}
```
One round trip, executed atomically by D1's batch transaction.

## Expected Improvement
Removes ~N×RTT from login latency — tens to hundreds of milliseconds for multi-character accounts — and closes the partial-write window entirely (all-or-nothing character replacement).

## Trade-offs
None meaningful. Batch size is bounded by realistic account character counts (single digits to low tens), far under D1 batch limits.

## Benchmark Approach
Wrap the current and batched implementations with `Date.now()` timing in a preview deployment; log `characters.length` alongside duration for XIVAuth logins and compare p50/p95 before/after for accounts with ≥3 characters.

**RESOLVED 2026-07-19** — `storeCharacters` runs the DELETE + all INSERTs in one atomic `db.batch`, removing N+1 round trips from the XIVAuth login path and the partial-write window.
