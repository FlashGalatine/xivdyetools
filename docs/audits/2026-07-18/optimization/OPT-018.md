# [OPT-018]: Contrast matrix computes every symmetric pairwise contrast twice

## Impact
LOW

## Category
Algorithmic (redundant computation over a symmetric matrix)

## Location
`packages/svg/src/contrast-matrix.ts:240-252` (cell loop calling `calculateContrast` per cell), `:103-116` (`calculateContrast` → `ColorService.getContrastRatio`)

## Current Performance
The row/column cell loop calls `calculateContrast(rowDye.hex, colDye.hex)` for every off-diagonal cell — both (i,j) **and** (j,i):

```ts
// packages/svg/src/contrast-matrix.ts:240-252
dyes.forEach((colDye, colIndex) => {
  ...
  if (rowIndex === colIndex) {
    elements.push(generateDiagonalCell(...));
  } else {
    const contrast = calculateContrast(rowDye.hex, colDye.hex);  // computed for (i,j) AND (j,i)
    elements.push(generateContrastCell(cellX, cellY, CELL_SIZE, contrast));
  }
});
```

WCAG contrast ratio is symmetric (`(L1+0.05)/(L2+0.05)` with L1 = lighter), so half the calls are pure duplicates. Each call performs two full hex→linearized-luminance conversions (per-channel `Math.pow` gamma expansion) inside `ColorService.getContrastRatio`. At the maximum supported 6 dyes: 30 calls where 15 suffice; 60 luminance conversions where 21 (one per dye, memoized) would do.

## Bottleneck Analysis
Absolute cost is tiny — tens of microseconds per matrix on Workers hardware — so this is not a latency fix. The relevance is (a) it runs inside per-request Discord command handling on a CPU-time-budgeted Worker alongside resvg rasterization, and (b) the duplicate computation obscures the symmetric structure, inviting future divergence (e.g. someone tweaking one triangle's formatting and not the other).

## Proposed Optimization
Precompute the upper triangle (and per-dye luminance implicitly) once, then look up:

```ts
const contrastCache = new Map<string, ContrastResult>();
const pairKey = (i: number, j: number) => (i < j ? `${i}:${j}` : `${j}:${i}`);

for (let i = 0; i < dyes.length; i++) {
  for (let j = i + 1; j < dyes.length; j++) {
    contrastCache.set(pairKey(i, j), calculateContrast(dyes[i].hex, dyes[j].hex));
  }
}
// in the cell loop:
const contrast = contrastCache.get(pairKey(rowIndex, colIndex))!;
```

## Expected Improvement
~2× fewer contrast computations (n² − n → (n² − n)/2 calls); ~65% fewer luminance conversions at n = 6. Microbenchmark-scale absolute savings — do it opportunistically when the file is next touched, not as standalone work.

## Trade-offs
Slight code complexity (a cache map and key helper) against a currently trivially-readable loop. No behavioral change — the same values render in the same cells; symmetry becomes guaranteed by construction rather than by recomputation.

## Benchmark Approach
Vitest bench (`vitest bench`) over `generateContrastMatrix` with 6 dyes, before/after; assert call counts by spying on `ColorService.getContrastRatio` (expect 15 calls, down from 30). Snapshot-compare the emitted SVG strings to prove output identity.
