# [OPT-006]: Price cache тАФ per-datacenter storage, unimplemented "stale fallback", and per-item lookup fan-out

## Impact
MEDIUM

## Category
Caching / resilience / latency

## Location
`apps/discord-worker/src/services/budget/price-cache.ts:27-30` (TTL constants), `:55-84` (`getCachedPrice`), `:135-152` (`getCachedPrices` fan-out), `:190-227` (`fetchWithCache`)

## Current Performance
`/budget` price caching uses `caches.default` with a 5-minute freshness window. On a warm colo, ~15-40 items resolve from cache; on any colo change, all items miss and require a full Universalis round-trip (typically 1-3 s). Each lookup issues one `cache.match` per item, and each store one `cache.put` per item (~30-80 cache ops per command).

## Bottleneck Analysis
1. **Per-colo cache.** The Cache API is datacenter-local. Discord's interaction traffic usually lands on a stable set of colos, but every colo switch is a *complete* cold miss. Effective hit rates are materially lower than a KV/DO-backed cache; the 5-minute window rarely amortizes across colos.
2. **Dead retention window.** The header comment promises a "5-minute freshness window with 15-minute stale fallback"; `CACHE_MAX_AGE_SECONDS = 900` retains entries 15 minutes, but `getCachedPrice` returns `null` for anything older than 300 s and **no code path ever reads stale data**. The 300-900 s band is dead weight and the docblock is misleading.
3. **Fan-out.** `getCachedPrices` issues N parallel `cache.match` calls; `setCachedPrices` N parallel `cache.put`s.

```ts
// price-cache.ts:72-75 тАФ entries older than 5 min are always discarded
const age = Date.now() - entry.cachedAt;
if (age > CACHE_TTL_SECONDS * 1000) {
  return null; // Expired
}
```

## Proposed Optimization
1. **Implement stale-if-error** (the advertised behavior): when `fetchFn` throws in `fetchWithCache`, re-read the cache accepting entries up to `CACHE_MAX_AGE_SECONDS` old and serve them with a "prices may be up to 15 min old" marker. This converts Universalis outages from hard failures into degraded responses.
2. Alternatively set `CACHE_MAX_AGE_SECONDS = CACHE_TTL_SECONDS` and fix the docblock, if stale serving is not wanted.
3. Optional: consolidate to one bucket entry per `(world, schema-version)` holding `Record<itemId, entry>` тАФ the common repeat-`/budget`-same-world case becomes 1 read + 1 write instead of ~30 each.

## Expected Improvement
Stale-if-error: `/budget` keeps working through Universalis blips (currently a hard error path via `UniversalisError`). Bucketing: ~60 cache ops тЖТ ~2 per warm command; modest CPU/latency win.

## Trade-offs
- Stale prices up to 15 min during outages тАФ label them in the embed.
- Bucket approach loses per-item TTL granularity: when any item expires, all are re-fetched (acceptable тАФ the fetch is already batched).
- A KV-backed cache would fix the per-colo miss problem but re-introduces the KV write-quota pressure that motivated the Cache API migration; only worth it on the paid tier.

## Benchmark Approach
Log `fromCache` / `fromApi` ratios (already returned by `fetchWithCache`) plus colo ID (`request.cf.colo`) for a week; compare hit rate and p50/p95 command latency before/after. For stale-if-error, fault-inject `fetchFn` and assert a degraded (not failed) response.
