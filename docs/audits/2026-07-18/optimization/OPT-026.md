# [OPT-026]: Repeated KV reads of the same preference keys within a single command

## Impact
LOW

## Category
Redundant KV reads / serial latency ahead of the 3-second interaction deadline

## Location
- `apps/discord-worker/src/handlers/commands/budget.ts:57` (`createUserTranslator` â†’ `resolveUserLocale`, which reads `prefs:v1:{userId}` and legacy `i18n:user:{userId}` â€” `apps/discord-worker/src/services/i18n.ts:126-162`)
- `apps/discord-worker/src/handlers/commands/budget.ts:133` and `:368` (`getUserPreferences` reads `prefs:v1:{userId}` **again**)

## Current Performance
A single `/budget find` performs 2-3 serial KV `get`s against the same two keys (locale resolution reads `prefs:v1` + possibly `i18n:user`; the world lookup re-reads `prefs:v1`) before any real work starts. Each KV get is typically single-digit ms warm but can be tens of ms cold â€” all inside the pre-defer window that must complete within Discord's 3-second budget.

```ts

const t = await createUserTranslator(env.KV, userId, interaction.locale);
...
// budget.ts:132-134 â€” read #3, same key
```

## Bottleneck Analysis
`resolveUserLocale` deliberately does a "direct KV read to avoid circular dependency with preferences.ts" (`i18n.ts:132-134`), so the preferences blob is parsed twice per command with no sharing. The reads are serial (translator first, prefs later), adding avoidable round-trips to the interactive path rather than the deferred background path.

## Proposed Optimization
Read `UserPreferences` once per interaction â€” in `handleCommand` (or a small request-context object) â€” and derive both the locale and the world/clan from it, passing the object into `createUserTranslator`-style factories:
```ts
const prefs = await getUserPreferences(env.KV, userId, logger);
const t = createTranslatorFromPrefs(prefs, interaction.locale);
```
The legacy `i18n:user:` fallback read can stay inside the factory, gated on the unified prefs lacking a language.

## Expected Improvement
1-2 fewer serial KV round-trips per command (~5-30 ms warm, more on cold reads), and one JSON parse instead of two. Small per-command, but applies to every preference-aware command.

## Trade-offs
Plumbing change through handler signatures (or a context object) touches many handlers; keep the old functions as wrappers to migrate incrementally. No behavioral risk if the fallback order (unified â†’ legacy â†’ Discord locale â†’ 'en') is preserved.
