# [OPT-027]: `priceData` getter clones the entire price Map on every access inside per-card loops

## Impact
LOW

## Category
Redundant allocation / GC churn in render loops

## Location
- `apps/web-app/src/services/market-board-service.ts:222-224` (`getAllPrices` returns `new Map(...)`)
- Property getters: `extractor-tool.ts:113-115`, `harmony-tool.ts:136-141`, `gradient-tool.ts:117-122`, `mixer-tool.ts:125-130`
- Loop usages: `extractor-tool.ts:2817-2818` (`.has` **and** `.get` — two getter hits per dye), `harmony-tool.ts:1663-1667` (per result card), `mixer-tool.ts:1637`, `:1847`, `gradient-tool.ts:1601`, `:2376`

## Current Performance
`getAllPrices()` defensively copies the full price map (136+ entries after a full fetch). Four tools expose it as a property getter named `priceData`, which *looks* like a cached field but allocates a fresh Map on **every** property access. Render loops then read it per item — e.g. the extractor does `this.priceData.has(dye.itemID)` followed by `this.priceData.get(dye.itemID)!`, i.e. two full map copies per dye per render pass. With 8-16 result cards that is ~30 map allocations (plus ~4,000 entry copies) per price refresh, repeated on every `prices-updated`/re-render cycle.

## Bottleneck Analysis
Each clone is O(n) over the cache; individually cheap (~µs), but it runs inside per-card loops during render bursts (price fetch completion re-renders all cards), producing avoidable allocations and GC pressure precisely when the UI is busiest. The defensive copy exists to protect the service's internal state, but callers only need read access.

## Proposed Optimization
Two compatible options:
1. **Snapshot per render (one line per site):**
   ```ts
   const prices = this.priceData;         // one clone
   for (const card of this.v4ResultCards) {
     const info = prices.get(card.data.dye.itemID);
     ...
   }
   ```
2. **Use the existing point lookup:** `this.marketBoardService.getPriceForDye(itemID)` (`market-board-service.ts:259-261`) reads the internal map directly with no copy; or add a `getPricesView(): ReadonlyMap<number, PriceData>` returning the internal map typed read-only, eliminating clones without sacrificing encapsulation at the type level.

## Expected Improvement
Removes ~n×m entry copies per render pass (n = cache size, m = cards × getter hits). Micro-level win — sub-millisecond per pass on desktop — but free, and it eliminates GC churn during price-refresh render bursts on low-end devices.

## Trade-offs
Option 2's `ReadonlyMap` is only compile-time protection (a caller could cast and mutate); option 1 retains the defensive copy but hoists it. Either is strictly better than the status quo; no behavioral change.

## Benchmark Approach
`performance.mark` around `updateV4ResultCardPrices()` (harmony) and the extractor card loop with 16 cards and a warm 136-entry cache; compare allocation counts via DevTools allocation sampling before/after.
