# [OPT-015]: CharacterColorService.findClosestDyes — per-call defensive copy + full sort for top-N

## Impact
LOW

## Category
Algorithmic / memory — avoidable copies and O(n log n) selection for tiny k

## Location
- `packages/core/src/services/CharacterColorService.ts:325-361` (`findClosestDyes`: `getAllDyes()` copy at `:336`, full sort at `:359`)
- Same pattern: `packages/core/src/services/CharacterColorService.ts:374-399` (`findDyesWithinDistance`)
- Per-dye re-conversion: `packages/core/src/services/CharacterColorService.ts:282-283` (`dye.rgb → hex` every call)

## Current Performance
Each call copies the entire dye array (`dyeService.getAllDyes()` returns a defensive copy), allocates a `CharacterColorMatch` object for all ~130 dyes, and full-sorts them to slice the top `count` (default 3). The Swatch Matcher use case matches an entire 192-color sheet → 192 calls → 192 array copies, ~25 000 match-object allocations, and 192 full sorts. `calculateDistanceWithMethod` also re-converts each dye's RGB to hex on every call (`:283`); the ColorConverter LRU absorbs most of that after warm-up, but it is still a per-call Map churn for 125 keys × 192 colors.

## Bottleneck Analysis
```ts
// CharacterColorService.ts:336, 351-360
const allDyes = dyeService.getAllDyes();          // defensive copy per call
const results: CharacterColorMatch[] = [];
for (const dye of allDyes) {
  ...
  results.push({ characterColor: color, dye, distance });   // ~130 objects per call
}
results.sort((a, b) => a.distance - b.distance);  // O(n log n) to extract top 3
return results.slice(0, count);
```
None of this is expensive in isolation (n≈130); it multiplies across sheet-level batch operations.

## Proposed Optimization
- Single-pass top-k selection (k ≤ 3 typically): keep a small bounded array of best matches instead of scoring-all + sorting-all; allocate result objects only for winners.
- Fetch the dye list once per batch: add a `findClosestDyesBatch(colors: CharacterColor[], dyeService, options)` entry point that reuses one dye array (and optionally pre-computes each dye's hex once) across all colors.
- Alternatively, delegate to `DyeSearch.findClosestDye` (k-d tree + candidate re-rank) via a hex conversion of the character color, unifying the two matching implementations.

## Expected Improvement
Per sheet (192 colors): ~192 array copies and ~25 000 transient allocations eliminated; selection goes O(n log n) → O(n·k). Modest wall-clock gain (likely single-digit ms total) but meaningful GC-pressure reduction in the browser during interactive swatch matching, and it removes a duplicate matching code path.

## Trade-offs
- Slightly more code; the current version is simpler and perfectly fine for one-off lookups.
- A batch API is additive (non-breaking) but grows the public surface.
- Delegating to `DyeSearch` couples this service to the k-d tree's candidate-threshold semantics (see REFACTOR-003) — do that only after the thresholds are hardened.

## Benchmark Approach
Micro-benchmark: match all 192 eye colors with `count: 3, matchingMethod: 'oklab'` before/after (vitest bench or manual `performance.now()` loop, 100 iterations); record wall time and, in the browser, allocation counts via the performance profiler. Assert result parity (identical top-3 per color) between old and new implementations.
