# [OPT-024]: og-worker bundle / cold-start inventory — dual DyeService init, eager 6-locale parse, character-color JSONs; wasm dominates

## Impact

LOW (informational sizing; actionable items are small wins)

## Category

Bundle size / cold-start cost

## Location

- `apps/og-worker/src/services/translator.ts:14-21` (eager preload of all 6 locale JSONs at module scope)
- `apps/og-worker/src/services/fonts.ts:52-58` (5 font buffers: SC 290 KiB + KR 176 KiB + Onest 120 KiB + Space Grotesk 131 KiB + Habibi 33 KiB ≈ 750 KiB)
- `@resvg/resvg-wasm` static import (~2.4 MiB) via `apps/og-worker/src/services/renderer.ts:15`
- `CharacterColorService` bundled skin/hair JSON data (MiB-class per workspace sizing notes) via `apps/og-worker/src/services/svg/dye-helpers.ts:18`
- Dual `new DyeService(dyeDatabase)` (`og-data-generator.ts:79`, `dye-helpers.ts:17` — see REFACTOR-024)

## Current Performance

Cold start pays, in order of cost: wasm module compile/instantiate (lazy on first image request), font `Uint8Array` materialization (lazy, cached), full dye-DB initialization **twice** (validation + 3 indexes + hue buckets + k-d tree, ×2), 6 locale JSON parses (eager at module load), character-color JSON parses (lazy per sheet). Post-OPT-001 the CJK fonts are near-optimal (~466 KiB combined vs ~20 MiB full fonts).

## Bottleneck Analysis

- The **eager 6-locale preload is intentional and correct** — it is the concurrency-safe stateless-translator pattern (REFACTOR-001) and should stay.
- The **duplicate DyeService** is pure waste (free win — covered by REFACTOR-024).
- The **wasm blob is the dominant, irreducible term**; nothing else materially moves the bundle while it is present.
- **Character-color data** is the next-largest lever: dye-helpers imports the whole `CharacterColorService`, pulling all race-specific tables even though swatch OG only ever *reads* them for row/col annotation; `packages/core` ships per-race split files (`data/character_colors/`) that could be imported selectively.
- **Habibi** renders only hex digits and ASCII labels; a digits+punctuation subset would be ~2 KiB vs 33 KiB.

## Proposed Optimization

Priority-ordered:

1. (Free) Single `DyeService` instance — REFACTOR-024.
2. (If bundle pressure appears) Import per-race character-color data selectively or drop the race-specific scan behind OPT-005's index build; savings potentially several hundred KiB.
3. (Cosmetic) Subset Habibi to its used glyph set via the existing `subset-cjk-fonts.py` machinery (~31 KiB saved).

## Expected Improvement

Modest cold-start reduction (one dye-DB init instead of two; fewer JSON parses) and up to a few hundred KiB of bundle if item 2 is taken. No user-visible latency change on warm isolates.

## Trade-offs

Per-race dynamic composition (item 2) adds code complexity and a behavior-preservation burden for the swatch annotation feature — only worth doing if the bundle approaches plan limits (paid plan ceiling 10 MiB; sibling discord-worker already sits ~8 MiB per workspace notes). Item 3 requires wiring Habibi into the subset script's regeneration checklist.

## Benchmark Approach

- `wrangler deploy --dry-run --outdir dist` before/after and compare reported gzip size.
- Cold-start: measure first-request latency on a fresh `wrangler dev --local` process (or production p99 after deploy) for `/og/default.png`, 10 trials each.
