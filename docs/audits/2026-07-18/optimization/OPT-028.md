# [OPT-028]: DyeSearchBox search input has no debounce

## Impact
LOW

## Category
Unnecessary re-renders (per-keystroke work)

## Location
- `apps/web-app/src/components/dye-search-box.ts:215-219` (input handler)
- Consumers rebuild the grid per event: `dye-selector.ts` / `dye-grid.ts` (search-changed listeners)
- Contrast (already debounced): `apps/web-app/src/components/v4/preset-tool.ts:635` (`_searchDebounce`)

## Current Performance
```ts
// dye-search-box.ts:216-219
this.on(searchInput, 'input', () => {
  this.searchQuery = searchInput.value;
  this.emit('search-changed', this.searchQuery);
});
```
Every keystroke emits `search-changed`, and the parent re-filters the full dye list (136 entries) and rebuilds the swatch grid DOM. Typing a 10-character query performs 10 full filter + render passes; fast typing interleaves renders between keystrokes, causing visible jank on low-end/mobile devices.

## Bottleneck Analysis
The filter itself is cheap; the cost is the DOM rebuild of the grid (dozens of elements with swatches, listeners, and tooltips) executed per event. The preset tool already solved the identical problem with a `window.setTimeout` debounce (`preset-tool.ts:635`), so the codebase has an established pattern — DyeSearchBox simply predates it.

## Proposed Optimization
Debounce the emit by 120-150 ms using the component's managed timer so destroy cleans it up:

```ts
private searchDebounce: ReturnType<typeof setTimeout> | null = null;

this.on(searchInput, 'input', () => {
  this.searchQuery = searchInput.value;
  if (this.searchDebounce) this.clearSafeTimeout(this.searchDebounce);
  this.searchDebounce = this.safeTimeout(() => {
    this.emit('search-changed', this.searchQuery);
  }, 150);
});
```
Keep the clear-button path immediate (emit `clear-all` synchronously as today, `:223-229`).

## Expected Improvement
~5-10× fewer grid rebuilds while typing (one per pause instead of one per keystroke); smoother input latency on mobile. Imperceptible added delay (150 ms trailing).

## Trade-offs
Results lag one debounce interval behind the last keystroke — standard search-box behavior. Tests listening for `search-changed` synchronously after dispatching an input event need fake timers.

## Benchmark Approach
Playwright/devtools: type a 12-character query at 80 ms/keystroke into the dye selector; count `search-changed` dispatches and measure total scripting time in the performance trace before/after. Target: ≤3 emits for the burst and reduced long-task time.

## Status

**RESOLVED 2026-07-19** — `search-changed` is debounced 150 ms via the component's managed `safeTimeout` (cleaned up on destroy); the clear-button path stays immediate.
