# [OPT-022]: universalis-proxy — item-ID normalization not deduplicated and not applied to the upstream URL

## Impact

LOW

## Category

Cache hit-rate / upstream request canonicalization

## Location

`apps/universalis-proxy/src/index.ts:114-121` (`normalizeItemIds`), `:195-196` (cache key), `:204` (upstream URL uses **raw** `itemIds`)

## Current Performance

`normalizeItemIds` sorts the IDs so `[1,2,3]` and `[3,1,2]` share a cache entry — good. But:

1. It does **not** deduplicate: `"5729,5729"` normalizes to `"5729,5729"`, a different cache key than `"5729"`, so identical logical queries fragment into separate cache entries and separate upstream fetches.
2. The **upstream URL is built from the raw, unnormalized parameter** (`.../aggregated/${datacenter}/${itemIds}?...`), so permutations that share one cache key still send differently-ordered URLs upstream on their first miss (also defeating any upstream-side URL caching), and duplicated IDs are forwarded verbatim.

## Bottleneck Analysis

```ts
function normalizeItemIds(itemIds: string): string {
  return itemIds.split(',').map(Number)
    .filter((n) => !isNaN(n) && n > 0)
    .sort((a, b) => a - b)
    .join(',');                                   // ← no dedupe
}
...
const normalizedIds = normalizeItemIds(itemIds);
const cacheKey = `aggregated:${datacenter.toLowerCase()}:${normalizedIds}`;
...
upstreamUrl: `${c.env.UNIVERSALIS_API_BASE}/aggregated/${datacenter}/${itemIds}?listings=5&entries=5`,
//                                                                    ^^^^^^^ raw, not normalizedIds
```

The web-app's batched fetcher chunks in stable order so fragmentation is modest in practice — this mainly hurts third-party/manual callers.

## Proposed Optimization

```ts
function normalizeItemIds(itemIds: string): string {
  return [...new Set(
    itemIds.split(',').map(Number).filter((n) => !isNaN(n) && n > 0)
  )].sort((a, b) => a - b).join(',');
}
...
upstreamUrl: `${c.env.UNIVERSALIS_API_BASE}/aggregated/${datacenter.toLowerCase()}/${normalizedIds}?listings=5&entries=5`,
```

## Expected Improvement

- Duplicate-bearing queries collapse onto the canonical cache entry (fewer upstream fetches, fewer cache entries).
- One canonical upstream URL per logical query improves upstream cacheability and log legibility.

## Trade-offs

Response shape: Universalis keys results by itemID, so removing duplicate IDs does not change the payload a client can consume. Behavior change is strictly canonicalization; the coalescer key already equals the cache key, so no coalescing semantics change.

## Benchmark Approach

Unit tests: `normalizeItemIds('5,5,3') === '3,5'`; integration test asserting two requests (`"5729,5729"` and `"5729"`) yield one upstream fetch and one cache entry (mock upstream, count calls).

## Status

**RESOLVED 2026-07-19** — `normalizeItemIds` dedupes as well as sorts, and the upstream URL is built from the normalized IDs + lowercased datacenter (one canonical upstream URL per logical query). Route test updated to assert the canonical URL.
