# [OPT-021]: universalis-proxy — every coalesced waiter re-writes the same cache entry (N puts per burst)

## Impact

LOW

## Category

Redundant work under concurrency (Cache API writes / `waitUntil` slots)

## Location

- `apps/universalis-proxy/src/services/cached-fetch.ts:92-104` (cache-miss path)
- `apps/universalis-proxy/src/services/cached-fetch.ts:155-166` (`revalidateInBackground` path)

## Current Performance

`coalescer.coalesce()` correctly deduplicates the upstream fetch (1 fetch per burst), but `cacheService.storeAsync(cacheKey, data, config)` executes **after** the coalesce in every caller. N concurrent requests for the same key therefore produce 1 upstream fetch but **N identical `cache.put()` operations**, each re-serializing the payload with `JSON.stringify` and consuming a `waitUntil` slot (Workers cap concurrent `waitUntil` tasks per request). The background revalidation path has the same shape.

## Bottleneck Analysis

```ts
// cached-fetch.ts:92-104
const data = await coalescer.coalesce<T>(cacheKey, async () => {
  const response = await fetchFromUpstream(upstreamUrl);
  ...
  return response.json() as Promise<T>;
});

// Store to cache (async, non-blocking)
cacheService.storeAsync(cacheKey, data, config);   // ← runs once per WAITER, not once per fetch
```

The burst scenario is precisely the one coalescing exists for (cold cache + simultaneous clients), so the duplicated writes fire exactly when the isolate is busiest. The cost per put is small (JSON stringify of a bounded payload + Cache API write), but it is pure waste ×(N−1).

## Proposed Optimization

Move the store inside the coalesced fetch function so only the winner writes:

```ts
const data = await coalescer.coalesce<T>(cacheKey, async () => {
  const response = await fetchFromUpstream(upstreamUrl);
  if (!response.ok) throw new UpstreamError(response.status, response.statusText);
  const parsed = (await response.json()) as T;
  cacheService.storeAsync(cacheKey, parsed, config);   // exactly once per upstream fetch
  return parsed;
});
```

Apply the same change in `revalidateInBackground`.

## Expected Improvement

Under a k-way burst: Cache API writes and JSON serializations drop from k to 1; frees k−1 `waitUntil` slots. No change in single-request behavior.

## Trade-offs

None functional. Marginal consideration: if the winner's `storeAsync` fails, waiters no longer provide redundant write attempts — irrelevant, since `store()` already swallows failures and the entry repopulates on the next miss.

## Benchmark Approach

Unit test with a mocked Cache API: fire 10 concurrent `cachedFetch` calls for one key against a slow mocked upstream; assert `cache.put` call count === 1 (currently 10). Optionally measure isolate CPU time via `wrangler dev --local` profiling before/after under `autocannon` burst load.

## Status

**RESOLVED 2026-07-19** — `storeAsync` moved INSIDE the coalesced fetch function on both the miss path and the background-revalidation path, so a k-way burst performs 1 cache write instead of k (and frees k−1 waitUntil slots).
